## 一、选题背景与研究意义
随着大语言模型在知识问答中的应用普及，面向学术论文的问答系统成为科研辅助的重要方向。当前通用问答系统在论文场景仍存在三个核心问题：
1. 回答可读但证据不可追溯。
2. 多论文场景下证据混杂、引用不稳定。
3. 证据不足时仍可能输出高风险结论。

本课题拟从零搭建论文问答系统，采用“规范驱动（Spec-driven）+ 证据优先”路线，构建一个可运行、可评估、可复现的论文问答系统。课题意义如下：
1. 工程意义：形成“规范-实现-验收”闭环开发范式，提升研发可控性。
2. 应用意义：为论文阅读与研究综述提供本地化、高可信辅助工具。
3. 学术意义：探索 Hybrid RAG、GraphRAG、Sufficiency Gate 协同机制在论文问答中的效果。

## 二、国内外研究现状（简述）
目前相关研究主要集中在：
1. 混合检索（BM25 + Dense Retrieval）以提升召回覆盖和语义匹配。
2. 图增强检索（GraphRAG）以补足多跳证据链。
3. 引用一致性与可解释性约束，降低幻觉风险。
4. 流式、多轮、来源卡片化等产品化交互体验。

不足在于：部分工作偏重模型效果，缺少工程可复现与可回滚设计；部分系统交互较好，但证据约束不强。本课题重点解决“可用性与可信性并重”的落地问题。

## 三、研究目标与拟解决的关键问题
### 1. 研究目标
1. 构建完整论文问答链路：入库、清洗、索引、检索、重排、生成、门控、引用。
2. 建立“答案必须可追溯到证据”的输出机制。
3. 在多论文场景提升多跳问题回答质量与稳定性。
4. 完成从调试型界面向产品化交互架构的迁移验证。

### 2. 拟解决的关键问题
1. 如何在召回率与噪声抑制间取得平衡。
2. 如何保证回答中的引用编号与来源一一对应且稳定。
3. 如何降低硬规则对语义检索能力的压制。
4. 如何在证据不足时实现“可追溯部分回答优先”的安全降级。

## 四、研究内容与技术路线

### 1. 主要研究内容
1. 论文入库与清洗：PDF 解析、Chunk 切分、`clean_text` 规范化与内容类型标注。
2. Hybrid RAG：BM25/向量/融合检索与多论文聚合。
3. Query Rewriting 与意图校准：提升检索查询质量，减少无效改写。
4. 图扩展与重排：对初检证据做 1-hop 扩展并 rerank。
5. 证据策略门控：Evidence Policy + Sufficiency Gate。
6. 多轮会话与可观测性：trace、诊断字段、运行日志。
7. 产品化迁移：Next.js 前端 + Node 网关 + Python 内核。

### 2. 技术路线（开题要求的“详细思路与路线”）
`用户问题`  
-> `Query Rewriting`  
-> `Hybrid Retrieval (BM25 + Dense)`  
-> `Graph Expansion (1-hop)`  
-> `Rerank`  
-> `Sufficiency Gate`  
-> `Answer with Citations`

关键约束：
1. 检索索引统一使用 `clean_text`。
2. 回答引用必须来自 chunk 证据集合。
3. 证据不足时触发澄清/弱回答，不输出无依据结论。

## 五、研究方法与实施方案
1. 规范驱动法：以 `openspec/specs` 与 `changes` 管理需求、设计、任务和验收。
2. 对比实验法：基于 `reports` 对比 M5/M6/M7.5 等阶段效果。
3. 工程迭代法：先稳定本地链路，再推进三层架构迁移。
4. 风险回滚法：保留旧 UI 与旧策略开关，保证可回退。

## 六、可行性分析
### 1. 技术可行性
1. 技术路线清晰：可按“入库-清洗-索引-检索-重排-门控-回答”的模块化顺序逐步实现。
2. 工具链成熟：Python 生态可支持 PDF 解析、检索索引、向量检索、服务化部署与日志追踪。
3. 方案可拆分：每个模块均可独立调试与验收，便于从零搭建过程中分阶段交付。

### 2. 资源与时间可行性
1. 数据来源明确：选取公开论文 PDF 作为语料，满足课程设计实验需求。
2. 时间节点明确：2026年3月1日完成开题，2026年4月1日完成论文初稿，可形成“开题-实现-初稿”闭环。
3. 风险可控：采用保守基线先行策略，先保证基础问答可用，再逐步增强图扩展与产品化能力。

### 3. 验证可行性
1. 按问题集进行阶段测试，关注检索相关性、引用一致性与证据可追溯性。
2. 通过运行日志记录关键指标（召回、重排、门控决策、引用映射），支撑后续论文实验章节撰写。

## 需求

### 需求:系统必须提供基于 LLM 的结构化实体抽取能力
当 chunk 缺失 `entities` 字段时，系统必须调用 OpenAI 兼容 LLM 接口进行实体抽取，并要求模型输出 JSON 结构化结果。该能力必须支持从统一运行时配置读取 `graph_entity` 位点参数，禁止仅依赖静态配置。

#### 场景:缺失 entities 触发 LLM 抽取
- **当** 输入 chunk 未提供 `entities` 且存在可用 `clean_text`
- **那么** 系统必须调用 LLM 抽取并返回实体列表结果

#### 场景:graph entity 配置由运行时覆盖
- **当** 管理员在模型设置中更新 `graph_entity` 的 provider/api_base/model
- **那么** 图构建任务必须按最新运行时配置执行实体抽取

### 需求:系统必须使用 Pydantic 校验实体输出结构
系统必须使用 Pydantic 模型校验每个实体项，至少包含 `entity_name` 与 `entity_type` 两个字段；不满足 schema 的输出必须视为失败。

#### 场景:模型返回结构合法
- **当** LLM 返回的 JSON 包含合法的 `entity_name` 与 `entity_type`
- **那么** 系统必须成功解析并产出结构化实体对象集合

#### 场景:模型返回结构不合法
- **当** LLM 返回缺字段、字段类型错误或非 JSON 内容
- **那么** 系统必须标记该 chunk 抽取失败并触发异常处理分支

### 需求:系统必须限制并发并聚合异步抽取任务
系统必须提供异步批处理函数，使用受控并发（例如信号量）与 `asyncio.gather` 聚合多个 chunk 的抽取结果，并禁止无上限并发调用外部 API。

#### 场景:批量处理多个 chunk
- **当** 系统需要为多条缺失实体的 chunk 执行抽取
- **那么** 系统必须在不超过配置并发度的条件下并行调用并聚合结果

### 需求:系统必须对 API 异常进行容错处理且不中断整体流程
对于单个 chunk 的请求超时、限流或解析失败，系统必须记录可观测错误信息并返回可处理的降级结果，禁止因单请求失败中断整个图构建任务。该容错策略必须在本地模型与外部 API 路径下保持一致。

#### 场景:出现 API 限流
- **当** 外部 API 返回限流错误（如 HTTP 429）
- **那么** 系统必须执行受控重试或降级处理，并继续处理其他 chunk

#### 场景:单请求失败不影响整体构图
- **当** 某个 chunk 的实体抽取最终失败
- **那么** 系统必须仅对该 chunk 返回空实体或失败标记，且整体构图流程继续执行

#### 场景:本地 graph entity 模型异常
- **当** 本地图实体抽取模型不可用或返回非法结构
- **那么** 系统必须记录失败并继续处理其余 chunk，不得中断整体构图

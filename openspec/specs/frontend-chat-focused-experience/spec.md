# frontend-chat-focused-experience 规范

## 目的
定义以对话为中心的聊天工作台体验，并确保聊天页在部署环境下仍能稳定读取运行态与建立会话连接。

## 需求
### 需求:系统必须提供以对话为中心的聊天主界面
系统必须移除聊天页内与模型连接相关的配置输入区域，聊天页必须仅承载会话内容与对话输入相关操作。

#### 场景:访问聊天页
- **当** 用户进入“对话问答”页面
- **那么** 页面必须不显示 Answer/Embedding/Rerank 配置表单

### 需求:系统必须支持 Markdown 聊天内容渲染
系统必须支持 Markdown 渲染，且必须支持代码高亮与公式渲染能力。

#### 场景:AI 返回带代码和公式的消息
- **当** AI 消息包含代码块与公式语法
- **那么** 系统必须以高亮代码与可读公式形式展示消息内容

### 需求:系统必须提供聊天空白状态
系统必须在无历史消息时展示居中快捷指令卡片组（至少 3 个），并必须支持用户点击卡片后直接将卡片文本作为输入发送，禁止出现无引导空白区域。

#### 场景:首次进入无对话
- **当** 当前会话没有任何消息
- **那么** 页面必须展示可点击的快捷指令卡片并支持一键发起提问

### 需求:系统必须在模型未配置时阻止误用并提供跳转入口
聊天页在读取运行态摘要（`/api/admin/runtime-overview`）时，必须具备与壳层一致的路由稳定性和响应容错。接口异常时，系统必须提供可读错误提示并保持聊天输入区与导航的可操作性；当未显式配置 `NEXT_PUBLIC_KERNEL_BASE_URL` 时，浏览器侧请求必须优先走同域相对路径或受控 rewrite，禁止默认请求 `127.0.0.1:8000`。

#### 场景:运行态接口地址缺失
- **当** 前端未显式配置 `NEXT_PUBLIC_KERNEL_BASE_URL`
- **那么** 聊天页运行态请求必须仍能通过当前站点的受控路径命中 Kernel 服务，而不是命中 Next 前端 404 页面或浏览器本机回环地址

#### 场景:运行态接口返回非 JSON
- **当** 运行态接口返回 HTML 或其他非 JSON 内容
- **那么** 系统必须显示“运行态概览不可用，请检查内核服务”等提示，且不得出现 JSON 解析异常堆栈

### 需求:系统必须以部署友好策略建立聊天 WebSocket 连接
聊天页必须以部署友好策略建立 WebSocket 连接。未显式配置 `NEXT_PUBLIC_GATEWAY_WS_URL` 时，系统必须根据当前页面来源自动推导聊天连接地址，并必须在 HTTPS 页面使用 `wss`；禁止默认连接 `ws://localhost:8080/ws`。

#### 场景:远程部署下打开聊天页
- **当** 用户通过服务器域名、公网 IP 或平台单入口域名打开“对话问答”页面且未显式配置 `NEXT_PUBLIC_GATEWAY_WS_URL`
- **那么** 聊天页必须尝试连接当前站点来源对应的 `/ws`，而不是连接用户浏览器所在设备的 `localhost`

### 需求:系统必须提供可区分的聊天气泡视觉
系统必须对用户消息与 AI 消息使用可区分的毛玻璃卡片样式，保证信息分层清晰。

#### 场景:同屏展示用户与 AI 消息
- **当** 会话中同时存在用户与 AI 消息
- **那么** 两类消息气泡必须在视觉上可被快速区分

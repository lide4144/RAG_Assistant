# backend-paper-lifecycle-integration 规范

## 目的

定义后端模块与 SQLite 论文存储的集成契约，确保导入、目录查询、状态聚合与删除/重建等操作通过统一的数据库访问层执行，维护数据一致性与论文生命周期完整性。

## 需求

### 需求:后端模块必须通过数据库访问层读写论文状态
后端各模块（导入、目录、任务、删除/重建）必须通过统一的论文存储访问层与 SQLite 交互，禁止直接操作 `papers.json` 或其他文件产物。系统必须确保所有论文状态查询与更新都经过数据库访问层，以保持数据一致性。

#### 场景:导入流程查询论文是否存在
- **当** 导入流程需要确认某篇论文是否已存在
- **那么** 系统必须通过数据库访问层查询 SQLite 论文主记录，而不是读取 `papers.json`

#### 场景:目录查询获取论文列表
- **当** 目录模块需要获取论文列表
- **那么** 系统必须通过数据库访问层从 SQLite 查询，禁止直接读取 `papers.json` 或拼接文件产物

### 需求:后端必须支持论文生命周期状态转换
后端必须支持单篇论文生命周期状态的完整转换链路，包括从 `dedup` 到 `ready` 的正常流程，以及 `failed`、`deleted`、`rebuild_pending` 等异常或维护状态的设置与查询。

#### 场景:导入流程更新论文阶段状态
- **当** 导入流程完成某阶段的处理
- **那么** 系统必须更新该论文在 SQLite 中的阶段状态与更新时间

#### 场景:删除流程标记论文状态
- **当** 用户删除一篇论文
- **那么** 系统必须将该论文标记为 `deleted` 状态，并编排清理相关派生产物

#### 场景:重建请求设置重建状态
- **当** 用户请求重建某篇论文的索引
- **那么** 系统必须将该论文标记为 `rebuild_pending` 状态，供后台任务调度

### 需求:后端必须维护派生产物归属一致性
后端在创建、更新或删除论文的派生产物（chunk、summary、structure、向量索引）时，必须同步更新 SQLite 中的产物归属记录。系统禁止在文件系统操作与数据库归属记录之间出现不一致。

#### 场景:索引构建后记录产物归属
- **当** 某篇论文的向量索引构建完成
- **那么** 系统必须在 SQLite 中记录索引产物状态、版本与构建时间

#### 场景:删除论文时清理产物归属
- **当** 论文被删除或重建
- **那么** 系统必须更新相关产物归属记录，并编排实际的文件清理或标记过期

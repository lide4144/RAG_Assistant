# frontend-llm-connection-settings 规范

## 目的
待定 - 由归档变更 add-llm-visual-config-and-model-detection 创建。归档后请更新目的。
## 需求
### 需求:系统必须提供 LLM 连接设置面板
系统必须在“模型设置”页面中同时提供“配置编辑区”和“生效状态概览区”。概览区必须显示各 stage 的当前生效 provider/model 与状态，不得仅作为输入表单页面。

#### 场景:进入设置页查看生效状态
- **当** 管理员进入“模型设置”页面
- **那么** 页面必须展示各 stage 当前生效模型摘要与状态等级

### 需求:系统必须在探测成功后提供模型选择
系统必须在模型探测成功后展示模型下拉框，并禁止在未探测成功时展示无效模型选项。

#### 场景:探测成功
- **当** 后端返回至少一个可用模型
- **那么** 系统必须显示模型下拉并允许管理员选择目标模型

### 需求:系统必须支持保存最终配置
系统必须支持在模型设置页中编辑并保存 Marker/Surya 运行档位参数，包括 `recognition_batch_size`、`detector_batch_size`、`layout_batch_size`、`ocr_error_batch_size`、`table_rec_batch_size`、`model_dtype`，并在保存后可回显。

#### 场景:保存 marker tuning 配置
- **当** 管理员修改 Marker 运行档位并点击保存
- **那么** 系统必须完成参数校验、持久化并回显最新生效值

#### 场景:参数越界提示
- **当** 管理员提交非法批大小或不支持的 dtype
- **那么** 系统必须给出字段级错误提示并保留其他已编辑输入

### 需求:系统必须提供可理解的错误反馈
系统必须根据后端错误码展示可理解提示，禁止仅显示无上下文的通用失败文案；对于密钥错误、网络失败、模型不可用等常见错误必须给出可执行指引。

#### 场景:鉴权失败提示
- **当** 后端返回 `AUTH_FAILED`
- **那么** 系统必须提示管理员检查 API Key 或权限并保持已填写字段不丢失

### 需求:系统必须提供阶段级错误反馈
系统必须在保存失败时明确提示是哪个 stage 的配置失败，并给出可理解的错误原因，禁止仅显示无上下文的通用失败文案。

#### 场景:单 stage 校验失败
- **当** 仅 `embedding` 配置缺失关键字段
- **那么** 系统必须提示 embedding 配置错误且保留 answer/rerank 输入状态

### 需求:系统必须提供模型连接设置的保存反馈
系统必须在“测试并保存”操作期间显示加载状态，并必须在保存成功后给出 Toast 成功提示。

#### 场景:保存成功反馈
- **当** 管理员点击“测试并保存”且后端返回成功
- **那么** 按钮必须显示加载结束状态并展示成功 Toast

### 需求:系统必须支持 Provider 预设联动
系统必须在管理员选择 Provider 后自动填充推荐 API Base，且必须允许管理员在自动填充后手动覆盖。

#### 场景:选择服务商
- **当** 管理员在 Provider 下拉框中选择某服务商
- **那么** 系统必须自动填充推荐 API Base 并允许后续修改

### 需求:系统必须提供安全的 API Key 输入体验
系统必须默认掩码显示 API Key 输入值，并必须提供显隐切换控件；禁止在非用户显式操作下明文展示密钥。

#### 场景:切换密钥可见性
- **当** 管理员点击 API Key 显隐图标
- **那么** 系统必须仅切换当前输入框的显示形态且不改变其实际值

## 目的

定义基于 Marker 的 PDF 结构化解析规范，确保标题质量、结构保真、LLM 增强降级与可观测性满足入库要求。

## 需求
### 需求:系统必须输出章节树中间表示
系统必须基于 Marker 结构化解析结果生成稳定的章节树中间表示，至少包含标题文本、层级、页码范围、父子关系与关联 chunk 列表。系统禁止在无状态标记的情况下静默丢弃标题层级信息。

#### 场景:显式标题层级被保留为章节树
- **当** PDF 含有可识别的标题层级
- **那么** 系统必须输出章节树中间表示并供后续 structure retrieval 使用

### 需求:结构解析质量必须可观测
系统必须为每篇论文记录结构解析质量状态，至少区分 `ready`、`degraded` 与 `unavailable`，并记录导致降级的原因摘要。当前端或 QA 请求需要结构检索时，系统必须能够读取该状态。

#### 场景:结构质量状态落盘
- **当** 一次 Marker 解析完成
- **那么** ingest 报告必须记录 `structure_parse_status` 与对应原因，且字段可序列化

### 需求:结构解析失败时必须提供兜底信号
当 Marker 无法稳定提取章节层级、标题顺序混乱或 PDF 版式导致结构树不可信时，系统必须输出可消费的兜底信号，供 QA 路由降级回 chunk 检索。系统禁止把不可信结构当作完整目录继续入索引。

#### 场景:双栏或非标准标题 PDF 触发降级
- **当** PDF 因双栏乱序、无显著标题样式或解析异常导致章节树不可信
- **那么** 系统必须将 `structure_parse_status` 置为 `degraded` 或 `unavailable`，并阻止不可信章节树进入结构索引

### 需求:本地 Marker 解析能力
系统必须支持通过运行时配置控制 Surya/Marker 的显存敏感参数，至少包括 `RECOGNITION_BATCH_SIZE`、`DETECTOR_BATCH_SIZE`、`LAYOUT_BATCH_SIZE`、`OCR_ERROR_BATCH_SIZE`、`TABLE_REC_BATCH_SIZE` 与 `MODEL_DTYPE`。系统必须支持通过运行时配置启用 Marker `--use_llm` 及其 `llm_service`，并允许按服务保存 provider 所需的连接参数。系统必须在参数缺失或非法时回退到默认安全值；当 LLM 增强解析失败但基础解析仍可继续时，系统必须执行可审计的降级而不是整批中断。

#### 场景:8GB 安全档位生效
- **当** 管理员选择 8GB 安全档位（低并行 + `float16`）
- **那么** 系统必须按该档位运行 Marker 解析并在运行报告中可追踪

#### 场景:非法参数自动回退
- **当** 运行时配置提供非法批大小或不支持的 dtype
- **那么** 系统必须记录告警并回退到默认安全值，不得中断整批导入

#### 场景:使用 OpenAI 类服务运行 Marker LLM 解析
- **当** 管理员为 Marker 配置 `OpenAIService` 并提供 `openai_api_key`、`openai_model` 与可选 `openai_base_url`
- **那么** 系统必须以该服务执行 LLM 增强解析，并在运行记录中保留服务类型与生效字段摘要

#### 场景:Marker LLM 失败后降级继续
- **当** Marker LLM 服务不可达、鉴权失败或 provider 特定参数缺失，但基础解析链路可继续
- **那么** 系统必须记录降级原因、使用的回退路径与结果可信度说明，并继续完成可用的导入结果

### 需求:标题候选与质量门禁
系统必须基于 Marker 输出生成带有结构语义的标题候选并执行质量门禁；当 `parser_engine=marker` 且当前文档未触发回退时，系统必须优先按 `H1/顶层标题 -> H2/次级标题 -> markdown 首行` 的顺序选择候选，再按需降级到非结构化来源。系统禁止将 Marker 候选、metadata 与首页首行作为同权来源直接混选，也禁止将占位文本、版权页眉或模板文案作为最终标题写入 `papers.json`。

#### 场景:Marker 顶层标题优先于低层候选
- **当** Marker 同时产出论文主标题与后续章节标题，且顶层标题通过质量门禁
- **那么** 系统必须优先采用顶层标题，不得让章节标题仅因规则分更高而覆盖主标题

#### 场景:高优先层被拒绝后向下一级降级
- **当** Marker 顶层标题候选命中黑名单、明显包含作者机构串或未达到质量门禁
- **那么** 系统必须拒绝该层候选并继续尝试次一级结构化候选，而不是直接落回 metadata 或首页首行

#### 场景:结构化候选全部不可用时再回退
- **当** Marker 成功解析但未能提供通过门禁的结构化标题候选
- **那么** 系统必须按既定回退顺序继续尝试非结构化来源，并记录结构化标题不可用的原因

### 需求:成功解析后的结构语义必须可保留
当 Marker 成功解析并输出结构化 block 时，系统必须在统一中间表示中保留足够的结构语义，至少包括 block 类型与 markdown 来源可用性诊断；系统禁止在成功路径中仅保留纯文本与标题层级而静默抹平其余结构语义。

#### 场景:表格与公式 block 类型可追溯
- **当** Marker 识别出表格块、公式块或其他非普通正文 block
- **那么** 系统必须在中间表示或等价诊断字段中保留这些 block 的类型信息，供后续链路判断是否可消费

#### 场景:markdown 来源可用性被记录
- **当** Marker 成功输出 markdown 但下游未直接采用其完整结构
- **那么** 系统必须记录 markdown 来源是否可用、是否被仅部分消费，而不是让该状态静默消失

### 需求:结构块语义必须可供下游消费
当 Marker 成功解析出表格块、公式块或其他显著结构块时，系统必须将这些结构语义以稳定字段暴露给 chunking、retrieval 与 citation 下游；系统禁止让这些结构信息只存在于调试工具或临时变量中。

#### 场景:结构块进入 chunking 输入
- **当** Marker 识别出表格块或公式块
- **那么** 统一中间表示必须向 chunking 暴露相应块类型，供 chunk 生成保留结构语义

#### 场景:结构块进入 retrieval 与 citation 输入
- **当** 某个表格块或公式块被切成可检索 chunk
- **那么** retrieval 与 citation 下游必须能够读取其结构块类型，而不是只能看到普通正文文本

### 需求:结构化正文保真输出
系统必须保留 Marker 提供的结构化正文信息，并在入库分块前输出稳定的章节层级、公式文本、表格文本序列与章节树中间表示，禁止在无告警情况下静默丢弃结构信息。

#### 场景:章节层级可追踪
- **当** PDF 含显式标题层级
- **那么** 系统必须在中间表示中保留层级信息并供后续切块策略与章节树索引使用

#### 场景:公式与表格文本不丢失
- **当** PDF 页面包含公式块或表格区域
- **那么** 系统必须将对应文本保留到中间表示，并可在 chunk 文本中追溯

#### 场景:结构信息不可信时显式标记
- **当** 章节层级抽取结果无法形成稳定章节树
- **那么** 系统必须输出结构降级状态与原因，而不是静默省略结构信息

### 需求:解析可观测性字段
系统必须在 ingest 报告中记录标题决策的结构化诊断信息，至少包含最终 `title_source`、`title_confidence`、被采用的候选层级以及结构化候选缺失或被拒绝的原因摘要。系统禁止只暴露最终标题而不暴露决策路径。

#### 场景:结构化标题被采用的观测
- **当** 某文档最终采用 Marker 结构化标题
- **那么** ingest 报告必须能区分该标题来自 `H1`、`H2` 或 markdown 首行层，而不能仅笼统标记为 `marker`

#### 场景:结构化候选被拒绝的观测
- **当** Marker 已提供结构化标题候选但最终未被采用
- **那么** ingest 报告必须记录候选被拒绝或降级的原因，便于区分“识别失败”与“决策未采用”

#### 场景:结构语义被保留但未消费的观测
- **当** Marker 已成功解析出 block 类型或 markdown 来源，但当前下游链路未将其作为优先输入消费
- **那么** ingest 报告必须能够区分“结构语义存在但未消费”和“结构语义不存在”这两种状态

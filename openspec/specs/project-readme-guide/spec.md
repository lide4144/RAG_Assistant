# project-readme-guide 规范

## 目的
定义项目根 README 的结构、命令示例、排障指南与生产部署说明，确保开发者和运维能通过文档完成开发、验证和部署。

## 需求
### 需求:项目 README 基础结构
系统必须在项目根目录维护 `README.md`，并且该文档必须包含“项目概览、已实现里程碑、快速开始、模块级使用手册、参数参考、关键命令示例、常见问题/排错”章节。

#### 场景:新用户首次访问仓库
- **当** 用户打开项目根目录的 `README.md`
- **那么** 用户必须能够看到完整的章节结构，并在不阅读源码的情况下找到按流程执行与按模块执行的入口

### 需求:按里程碑组织的使用说明
系统必须在 `README.md` 中按里程碑（至少 M1、M1.5、M2、M2.1、M3）说明输入、输出与最小可执行命令示例。

#### 场景:用户按里程碑执行流程
- **当** 用户参考 README 执行某里程碑命令
- **那么** 文档必须明确该命令的输入路径与输出产物位置

### 需求:示例命令可复用
系统必须在 README 中提供可直接复用的 CLI 示例，并禁止仅给出抽象描述而不提供命令。

#### 场景:用户复制示例命令
- **当** 用户从 README 复制命令执行
- **那么** 用户必须能够定位该命令对应的脚本入口与产物路径

### 需求:README 模块级使用手册
系统必须在 `README.md` 中提供模块级使用手册，且每个核心模块（至少包含 `app.ingest`、`app.clean_chunks`、`app.build_indexes`、`app.graph_build`、`app.qa`）必须说明用途、输入、输出、最小可运行命令与典型使用场景。

#### 场景:用户按模块单独运行
- **当** 用户只想运行某个模块而非完整流水线
- **那么** 用户必须能够在 README 中找到该模块的独立命令、输入前置条件与产物路径

### 需求:README 参数语义说明
系统必须在 `README.md` 中提供关键参数说明，且参数说明必须覆盖“参数名称、所属模块、默认行为、调整后影响、常见取值建议”。

#### 场景:用户调整检索相关参数
- **当** 用户需要调整检索质量相关参数（例如 `top_k_retrieval`、`top_n_evidence`、`fusion_weight`）
- **那么** README 必须明确这些参数如何影响召回范围、证据数量或融合行为

#### 场景:用户调整 embedding 相关配置
- **当** 用户配置 `embedding` 相关参数（例如 provider/model/batch_size/cache）
- **那么** README 必须说明这些参数对构建耗时、质量一致性与失败恢复行为的影响

### 需求:README 排障与验证指引
系统必须在 `README.md` 中提供常见失败场景的排障指引，并且每个场景必须包含“症状、可能原因、检查步骤、修复建议”。对于生产部署，README 还必须提供单域 Nginx 反向代理的最小部署结构、关键路径转发规则和验证步骤；对于 Cloud Studio 等端口封装环境，README 还必须说明单端口代理入口的启动方式。

#### 场景:QA 命令返回空结果或低相关证据
- **当** 用户运行 QA 后证据为空或相关性明显偏低
- **那么** README 必须提供可执行的检查路径（索引文件、graph 文件、配置项、输入数据完整性）与对应修复建议

#### 场景:用户进行生产反代部署
- **当** 用户按照 README 在生产环境部署该项目
- **那么** README 必须说明前端、Kernel、Gateway 的内部监听关系、Nginx 对 `/`、`/api/*`、`/ws` 的代理方式，以及验证部署是否成功的步骤

#### 场景:用户在 Cloud Studio 中部署
- **当** 用户通过 Cloud Studio 的应用封装能力暴露该项目
- **那么** README 必须说明为什么只应暴露单个代理端口，以及对应的启动脚本或操作步骤

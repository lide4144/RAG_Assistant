# paper-assistant-mode 规范

## 需求
### 需求:系统必须提供论文助理默认模式
系统必须将默认回答组织为“主题方向 + 可追溯证据 + 下一步可问问题”的用户友好结构，禁止仅输出面向开发调试的结构化字段。

#### 场景:默认回答包含三段式结构
- **当** 用户提出开放式灵感问题
- **那么** 回答必须至少包含主题方向总结、对应证据引用和后续追问建议

### 需求:系统必须在证据不足时执行最小澄清
系统必须保持最小澄清策略（一次仅一条澄清问题），并且澄清问题文案必须面向科研任务语义。

#### 场景:澄清问题任务语义化
- **当** 系统判定需要澄清
- **那么** `clarify_questions` 必须为 1 条，且内容必须引导用户补充主题、方法或实验约束

### 需求:系统必须限制连续澄清并在上限后给出可用回答
系统必须跟踪当前会话主题下的连续澄清次数；当达到上限（默认 2）时，下一轮不得继续输出 `clarify`，必须输出低置信但可追溯的主题化回答，并明确不确定性。发布评测必须以会话链路方式验证该行为。

#### 场景:连续澄清达到上限后强制回答
- **当** 同一主题会话连续两轮 `decision=clarify`
- **那么** 第三轮系统必须输出可追溯主题化回答（含引用），并在结果中标记低置信/待验证，而不是再次澄清

#### 场景:多轮链路增长门槛
- **当** 系统执行 `B_multi_turn` 桶评测
- **那么** 每条会话链必须在限定轮次内至少出现一次可追溯回答，且不得出现超上限连续澄清

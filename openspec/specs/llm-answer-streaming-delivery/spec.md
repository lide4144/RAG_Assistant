## 需求
### 需求:回答阶段必须支持可配置流式传输
当 `answer_stream_enabled=true` 且 `answer_use_llm=true` 时，系统必须使用流式方式获取回答内容，并禁止阻塞到整包完成后才返回首段内容。

#### 场景:开启流式后返回首段内容
- **当** `answer_stream_enabled=true` 且证据门控允许进入 `llm_answer` 路径
- **那么** 系统必须以流式方式接收回答，并在最终完成前提供可观测的增量输出事件

### 需求:CLI 在流式模式下必须显示增量内容
当命令行模式执行 QA 且启用回答流式时，系统必须在终端逐步显示回答增量，而不是仅在流结束后一次性打印完整答案。

#### 场景:CLI 实时显示流式增量
- **当** `answer_stream_enabled=true` 且通过 CLI 执行 `run_qa`
- **那么** 终端输出必须在流式进行中持续更新，并在流式结束后输出最终结构化结果或降级结果

### 需求:流式路径必须保持证据约束与结构化输出
流式传输仅改变传输方式，系统必须继续满足 `answer` 与 `answer_citations` 的结构化输出要求，并禁止引用 `evidence_grouped` 之外的 chunk。

#### 场景:流式完成后结构化校验
- **当** 流式回答接收完成
- **那么** 系统必须执行与非流式一致的结构化解析与 citation 子集校验，不通过时必须降级

### 需求:流式失败必须统一降级且可追踪
流式过程出现首字超时、连接中断、空流或解析失败时，系统必须回退模板回答并记录统一 warning 与诊断信息，禁止中断主流程。

#### 场景:流式中断触发降级
- **当** 流式连接在完成前中断或超时
- **那么** 系统必须输出模板回答，追加对应 fallback warning，并写入可审计诊断字段

### 需求:前端必须消费并展示回答增量流
当回答流式模式开启时，系统必须在 UI 侧持续渲染增量内容，禁止等待完整回答后一次性展示。

#### 场景:UI 实时展示流式内容
- **当** 回答进入流式生成路径
- **那么** 用户必须在最终完成前看到持续更新的文本输出

### 需求:流式链路中断必须降级并保持上下文连续
系统必须在流式连接中断、超时或空流时返回可恢复结果并保留当前会话上下文，禁止清空或打断对话链。

#### 场景:流式中断后恢复交互
- **当** 回答流式连接在完成前异常中断
- **那么** 系统必须输出降级提示与可继续提问入口，且历史消息保持可用

### 需求:系统必须通过网关统一流式分发
流式回答必须由网关统一分发，保证前端消费语义一致；禁止由不同后端直接暴露不兼容流式协议给前端。

#### 场景:不同后端统一流式格式
- **当** 请求分别命中 Python 本地回答与 Web 回答链路
- **那么** 前端接收的流式事件结构必须一致且可复用同一渲染逻辑

### 需求:系统必须支持流式来源分段推送
系统必须支持在回答流式进行中推送来源集合或增量来源，并在 `messageEnd` 前完成引用映射闭合。

#### 场景:流式中先到来源后到正文
- **当** 后端先完成来源聚合
- **那么** 系统必须允许先推送 `sources` 事件并在后续 `message` 中引用对应来源

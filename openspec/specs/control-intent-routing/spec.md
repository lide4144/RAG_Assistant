# control-intent-routing 规范

## 需求
### 需求:系统必须由 LLM planner 统一解释控制意图与多轮关系
系统必须由顶层 `LLM planner` 统一解释控制意图、追问/换题关系与结构化参数，并将这些结果写入正式 `planner decision`；禁止在正式执行链中再使用独立规则路由器为控制意图或多轮关系生成另一份顶层解释结果。

#### 场景:控制意图解释不再依赖独立规则链
- **当** 用户输入“换成表格展示，并继续上一轮比较”或等价表达
- **那么** 系统必须由 `LLM planner` 同时产出控制意图参数和多轮关系判断，而不得在正式执行链中再调用独立规则路由器重解释该请求

### 需求:系统必须输出能力规划结果而非单一意图标签
系统必须在路由阶段输出能力规划结果，至少包含 `primary_capability`、`strictness`、`action_plan` 与 `planner_confidence`；禁止仅以单一 `intent_type` 作为复合查询的唯一决策产物。

#### 场景:复合查询输出能力规划
- **当** 用户输入同时包含目录过滤与跨文档对比的复合查询
- **那么** 系统必须输出包含多个顺序动作的能力规划结果，而不是只返回单一 `retrieval_query`

### 需求:系统必须在检索前执行控制意图路由
系统必须将 `control intent（控制意图）` 识别作为 `planner / policy（规划器 / 策略层）` 的正式职责，并在检索前输出结构化结果（至少包含 `intent_type（意图类型）` 与必要参数）；禁止仅依赖关键词/正则列表作为唯一识别手段。对于统一 planner 模式，控制意图不得再误入证据型 QA 主链，也不得由 `sufficiency gate（充分性门控）` 或 `evidence gate（证据门）` 直接裁决其最终交互姿态。

#### 场景:表达变体仍可识别控制意图
- **当** 用户使用非预设措辞表达格式控制意图（如“换成表格展示”）
- **那么** 系统必须识别为控制意图并输出结构化参数

#### 场景:控制意图不进入证据型 QA 主链
- **当** 用户输入“用中文回答我”或等价的样式控制指令
- **那么** 系统必须由 `planner / policy（规划器 / 策略层）` 直接处理该控制意图，而不得将其送入证据型 QA 主链并触发基于证据的拒答

#### 场景:控制意图与主能力共存时保持主能力不丢失
- **当** 用户输入“先列出昨天上传的论文，再用表格展示方法差异”
- **那么** 系统必须保留主能力为跨文档对比，并将“表格”作为结构化参数而不是把整轮请求降格为纯格式控制

### 需求:系统必须提供可配置的路由开关与回退
系统必须支持语义路由开关，并在语义路由失败时回退到最小可用路径，不得中断主流程。对于统一 planner 模式，系统必须在规划节点失败、超时或输出非法 JSON 时回退到最小可用单步路径，并记录回退原因。

#### 场景:语义路由失败回退
- **当** 路由模型调用失败或输出不合法
- **那么** 系统必须回退到默认 QA 路径并记录路由失败观测信息

#### 场景:planner 失败回退到单步路径
- **当** 统一 planner 超时或返回无法解析的结构化结果
- **那么** 系统必须回退到最小可用单步路径，且不得继续执行多步计划

### 需求:系统必须提供意图路由可观测字段
系统必须输出意图路由观测字段，至少包含路由来源、命中意图、置信度、`primary_capability`、`planner_confidence`、`interaction_decision_source` 与是否进入受控结束路径；当控制意图被识别时，trace 中必须能够证明其解释来源于正式 `LLM planner` 决策，而不是来自旧规则链或尾部补写。

#### 场景:控制意图来源可审计
- **当** 一次请求完成控制意图与多轮关系判定
- **那么** trace 中必须能够证明这些结果来自正式 `LLM planner decision`，并记录是否进入受控结束路径

# frontend-saas-shell-navigation 规范

## 目的
定义前端全局 SaaS 导航壳层在多页面工作台中的导航、连接状态与运行态摘要行为，确保本地开发、远程部署和反向代理环境下都能稳定工作。

## 需求
### 需求:系统必须提供全局 SaaS 导航壳层
系统在壳层加载运行态摘要时，必须将管理接口请求稳定路由到 Kernel 服务；当地址未配置或上游返回非 JSON 错误页时，系统必须降级为可读状态提示，禁止抛出前端 JSON 解析异常。未显式配置 `NEXT_PUBLIC_KERNEL_BASE_URL` 时，壳层必须优先使用当前站点下的受控相对路径策略；在生产部署中，系统必须优先通过同域 `/api/admin/*` 入口访问运行态接口，并由反向代理将请求转发到 Kernel，禁止默认请求 `http://127.0.0.1:8000` 或要求浏览器直接访问 `8000` 端口。

#### 场景:未显式配置 Kernel 地址时仍可读取运行态
- **当** 前端未设置 `NEXT_PUBLIC_KERNEL_BASE_URL`
- **那么** 系统必须通过当前站点下的受控路径（如相对 `/api/admin/runtime-overview` 或等效 rewrite）请求到 Kernel 的 `/api/admin/runtime-overview`

#### 场景:生产环境通过同域入口访问运行态
- **当** 系统部署在带有 Nginx 反向代理的生产域名下且未显式配置浏览器侧 Kernel 绝对地址
- **那么** 浏览器必须通过当前站点下的 `/api/admin/runtime-overview` 访问运行态接口，而不是跨域请求单独的 Kernel 域名或端口

#### 场景:运行态接口返回 HTML 错误页
- **当** 运行态接口响应 `content-type` 非 `application/json`
- **那么** 系统必须显示“运行态概览加载失败”类可读提示且不出现 `Unexpected token '<'` 解析错误

### 需求:系统必须提供三项主导航入口
系统必须在侧边导航中提供“对话问答”“知识处理”“模型设置”三项入口，禁止使用英文术语作为主标签。

#### 场景:导航可见性
- **当** 任意页面加载完成
- **那么** 用户必须始终可见上述三项导航入口

### 需求:系统必须通过路由实现无刷新页面切换
系统必须通过前端路由进行页面切换，禁止通过整页刷新进行导航。

#### 场景:从对话问答切换到模型设置
- **当** 用户点击“模型设置”导航项
- **那么** 系统必须在不刷新浏览器页面的前提下切换到设置页面并保留应用壳层

### 需求:系统必须在导航中标记当前页面状态
系统必须将连接状态显示为不可点击的轻量级状态指示器，禁止以按钮样式渲染。连接成功时必须展示绿色发光圆点与文本 `🟢 已连接`。

#### 场景:连接成功指示
- **当** 系统与后端健康检查通过
- **那么** 导航区域必须展示无边框状态指示器而非按钮控件

待定 - 由活动变更 research-assistant-user-friendly-workflow 同步创建。归档后请更新目的。
## 需求
### 需求:系统必须异步受理浏览器侧批量论文上传
系统必须在浏览器侧批量上传论文时，先接收并暂存文件，再以后台任务方式执行导入流程；接口必须立即返回可追踪的任务标识，禁止要求单个 HTTP 请求同步完成整套入库、清洗与索引流程。

#### 场景:浏览器一次性上传多个 PDF
- **当** 用户在前端一次性选择多个 PDF 并点击开始导入
- **那么** 系统必须立即返回导入任务已受理的响应与 `task_id`，并在后台继续执行后续导入流程

#### 场景:已有导入任务运行时重复提交
- **当** 同一时刻已有论文导入任务处于 `queued` 或 `running`
- **那么** 系统必须复用现有活动任务并显式返回未新建任务的受理结果，而不是并发启动第二个同类导入

### 需求:系统必须提供低门槛论文导入入口
系统必须在 Library 中提供统一导入入口，支持本地文件导入，并以任务语义反馈处理结果；当导入触发后续流水线任务时，系统必须展示该任务的状态与进度摘要。对于浏览器文件上传，系统必须允许用户在请求返回后继续查看后台状态，而不是要求等待同步完成。

#### 场景:导入后给出可理解结果
- **当** 用户完成论文导入提交
- **那么** 系统必须先反馈任务已受理或复用中的状态，并在任务终态后反馈成功/失败数量与可继续动作

#### 场景:一次导入多篇 PDF
- **当** 用户在知识处理页一次性提交多篇 PDF 且后台任务持续运行
- **那么** 页面必须显示该批任务的总论文数、已完成数量、处理中数量与失败数量，并随着任务推进持续刷新

### 需求:系统必须向批量论文导入用户展示逐项状态
系统必须在批量导入过程中提供单篇论文级的状态可见性，至少让用户识别当前处理项、已完成项与失败项；禁止让用户只能从整批任务的笼统状态猜测个体论文是否卡住。

#### 场景:批量导入中存在失败样本
- **当** 一次导入中的部分论文处理失败
- **那么** 页面必须在进度区域或其关联明细中显示失败论文及失败原因，而不是仅在任务结束后汇总一个失败总数

#### 场景:导入结果分类可见
- **当** 一次导入包含新增、重复与失败样本
- **那么** 系统必须分别展示新增数量、已存在跳过数量、失败数量及原因

#### 场景:导入后任务摘要可见
- **当** 导入触发切块清洗或图构建
- **那么** 系统必须展示后续任务当前状态（queued/running/succeeded/failed）与最近更新时间

### 需求:系统必须支持专题化组织论文
系统必须允许用户按专题对论文进行分组，供 Chat 侧限定上下文。

#### 场景:按专题提问
- **当** 用户在某专题下提问
- **那么** 系统必须在回答中使用该专题的论文范围作为默认语料

### 需求:系统必须隐藏底层管线术语
导入流程中禁止向普通用户暴露 chunks/index/build 等实现术语作为必需理解内容。

#### 场景:导入流程无工程术语阻断
- **当** 用户执行导入流程
- **那么** 页面必须使用用户可理解的操作语义，不要求用户理解底层管线步骤

### 需求:系统必须在导入完成后显示后续处理链路状态
系统必须在导入结果中展示后续处理链路（切块清洗、图构建）的当前状态与可执行动作，禁止将用户停留在“导入完成”但无下一步引导的状态。对于批量导入，该链路状态必须与批次进度并列呈现，而不是替代批次进度。

#### 场景:导入成功后显示下一步动作
- **当** 用户完成论文导入
- **那么** 系统必须展示“继续处理/开始图构建/进入聊天验证”等可执行下一步

#### 场景:导入处理中查看后续状态
- **当** 用户查看一个仍在后台运行的批量导入任务
- **那么** 页面必须同时显示整批任务的完成度与当前所处流程阶段，用户无需二选一地理解任务状态

### 需求:批量导入界面必须展示与论文实体一致的逐篇状态
批量导入界面必须展示与 SQLite 论文生命周期一致的逐篇状态，而不是仅展示上传文件名对应的临时进度项。系统禁止在页面刷新、任务结束或会话切换后丢失单篇论文状态可见性。

#### 场景:页面刷新后恢复逐篇状态
- **当** 用户在导入任务运行期间刷新页面或稍后重新进入导入页
- **那么** 系统必须能够基于 SQLite 论文记录与任务状态恢复该批论文的逐篇状态展示

#### 场景:同一批导入包含新增与重复论文
- **当** 一次批量导入中同时存在新增论文与已存在论文
- **那么** 页面必须分别展示新增、重复跳过与失败的逐篇状态，并与对应 `paper_id` 或稳定记录一致

### 需求:导入结果必须能够指向论文级后续动作
导入结果必须允许用户基于单篇论文状态继续执行查看详情、删除、重建或进入聊天验证等后续动作。系统禁止仅提供整批任务级结果而无法定位到具体论文。

#### 场景:单篇论文索引失败后重建
- **当** 某篇论文导入成功但索引阶段失败
- **那么** 页面必须能够定位该论文并提供重建索引或等价后续动作

#### 场景:单篇论文已 ready
- **当** 某篇论文已完成处理并进入 `ready`
- **那么** 页面必须能够从导入结果直接跳转到该论文相关的目录详情或聊天验证入口

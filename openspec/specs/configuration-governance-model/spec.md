## 需求
### 需求:系统必须定义统一的配置 owner 模型
系统必须为可持久化或可运行时覆盖的配置字段定义统一 owner 分类，至少包括 `static`、`runtime` 与 `env_only` 三类。系统必须为每个新增配置字段声明所属 owner，禁止继续以“按模块各自决定”的方式新增配置来源。

#### 场景:新增配置字段进入系统
- **当** 系统引入新的配置字段
- **那么** 该字段必须被明确归类为 `static`、`runtime` 或 `env_only` 之一

### 需求:系统必须定义统一的来源优先级语义
系统必须为不同 owner 字段提供统一的来源优先级解释。`static` 字段必须以静态配置为基线并回退到安全默认值；`runtime` 字段必须声明是否允许环境变量覆盖，并在允许时明确 `env`、runtime 持久化与默认值之间的优先级；`env_only` 字段必须由环境变量控制，禁止伪装为前端或静态可编辑字段。

#### 场景:查询字段最终值来源
- **当** 管理员或开发者查看某个配置字段的最终生效值
- **那么** 系统必须能够用一致语义解释该值来自 `default`、`runtime`、`env` 或安全默认回退

### 需求:系统必须限制前端设置页的治理边界
系统必须将前端设置页定义为 runtime owner 字段的管理入口，禁止将其扩张为全量系统配置编辑器。涉及检索策略、会话策略、图谱扩展等系统基线行为的字段必须继续保留在静态配置治理范围内；`planner` 若迁移为 runtime owner，必须以独立高风险配置面管理，而不是混入普通模型连接卡片。

#### 场景:评估某个基线策略字段是否进入前端
- **当** 团队评估新的设置项是否应出现在前端设置页
- **那么** 只有被归类为 `runtime` owner 的字段才允许进入前端设置页，且顶层规划器必须采用独立风险面板

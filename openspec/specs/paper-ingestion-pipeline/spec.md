## 目的

定义论文入库流水线的最小可用规范，包括 PDF 解析、chunk 化、结构化输出与验收要求。
## 需求
### 需求:系统必须在运行时依赖清单中声明 PDF 解析基础依赖
系统必须在可部署的运行时依赖清单中显式声明 PDF 解析所需基础依赖，至少包括 `PyMuPDF`；禁止依赖镜像历史状态、手工安装或环境偶然残留来满足 PDF 解析能力。

#### 场景:从依赖清单构建全新镜像
- **当** 部署系统仅依据项目依赖清单构建一个全新运行镜像
- **那么** 该镜像必须具备 PDF 解析基础依赖，并能够执行 PDF 入库而不因缺少 `PyMuPDF` 直接失败

### 需求:批量论文摄取命令
系统必须提供统一摄取命令用于批量处理 PDF 与 URL 两类输入。命令必须支持目录内 PDF 发现，同时支持通过参数或清单文件传入 URL 列表；两类输入必须进入同一入库编排并输出统一产物。

#### 场景:混合输入统一入库
- **当** 用户同时提供 PDF 输入目录与 URL 列表
- **那么** 系统必须在一次运行中处理两类输入并生成统一的文档映射与分块产物

#### 场景:仅 URL 输入可运行
- **当** 用户未提供 PDF 目录但提供有效 URL 列表
- **那么** 系统必须完成 URL 入库流程并输出可用于后续检索的产物

### 需求:输入文件发现与过滤
系统必须处理输入中的 `.pdf` 文件与有效 URL，且必须忽略不支持的文件类型与非法 URL 条目，不得中断整批处理。

#### 场景:混合输入包含无效条目
- **当** 输入集合包含 `a.pdf`、`b.txt` 与非法 URL
- **那么** 系统必须处理 `a.pdf`、跳过无效条目并记录跳过原因

### 需求:论文元数据映射输出
系统必须为每篇已处理文档写入统一映射记录，并在原有 `paper_id`、`title`、`path` 基础上扩展 `source_type` 与 `source_uri`（或等价字段）以区分 PDF 与 URL 来源。对于 PDF 来源，系统必须额外记录解析引擎与标题质量信息（至少包括 `parser_engine`、`title_source`、`title_confidence` 或等价字段），并在 Marker 成功解析时按结构化信任路径决策标题。系统禁止将命中标题黑名单的候选直接写入 `title`，也禁止在 Marker 已提供合规顶层标题时退回采用更低优先级来源。

#### 场景:URL 文档映射完整落盘
- **当** 一条 URL 文档处理成功
- **那么** 文档映射记录必须包含来源类型与来源 URI 字段并可查询

#### 场景:增量写入保持映射连续性
- **当** 多次导入在不同时间发生
- **那么** `papers.json` 必须保留历史记录并保持可追溯连续性

#### 场景:PDF 映射包含结构化标题来源
- **当** 一篇 PDF 文档通过 Marker 成功解析并导入成功
- **那么** 文档映射记录必须包含可定位到结构化层级的标题来源语义，并可用于排查标题误识别

#### 场景:结构化信任阻止误用章节标题
- **当** Marker 同时识别出论文主标题与正文章节标题
- **那么** 系统必须将主标题写入映射记录，不得因章节标题文本更短或规则分更高而覆盖主标题

#### 场景:标题候选命中黑名单
- **当** 当前优先层的标题候选命中占位或版权黑名单模式
- **那么** 系统必须拒绝该候选并继续尝试下一层合规候选或安全降级标题，不得直接写入黑名单标题

### 需求:成功解析后的结构化输入状态必须落盘
当 PDF 通过 Marker 成功解析时，入库流程必须落盘该文档有哪些结构化输入已成功保留、哪些仅部分消费、哪些因兼容性原因尚未进入优先链路。系统禁止把“结构语义存在但未消费”和“结构语义本身不存在”混为同一状态。

#### 场景:block 语义保留状态可审计
- **当** Marker 成功解析并输出 block 类型
- **那么** 入库元数据或报告必须能表明这些 block 语义是否已保留到统一中间表示

#### 场景:markdown 仅部分消费时显式披露
- **当** markdown 输出存在但当前仅被用于局部用途，例如标题候选
- **那么** 入库元数据或报告必须显式标记该 markdown 为“部分消费”，而不是默认为“已完整采用”

### 需求:结构块语义必须贯通到 chunking
当 PDF 通过 Marker 成功解析并识别出表格块、公式块或其他显著结构块时，入库流程必须在 chunk 生成阶段保留这些块语义，使对应 chunk 能携带结构类型。系统禁止在 chunk 层把结构块完全扁平为不可区分的普通正文。

#### 场景:表格块生成可区分 chunk
- **当** Marker 识别出表格块并进入 chunking
- **那么** 生成的 chunk 必须保留其表格语义，供后续检索与 citation 使用

#### 场景:公式块生成可区分 chunk
- **当** Marker 识别出公式块并进入 chunking
- **那么** 生成的 chunk 必须保留其公式语义，供后续检索与 citation 使用

### 需求:历史标题修复路径
系统必须提供能够复用结构化标题信号的历史标题修复路径。对于历史 PDF 记录，系统必须优先使用已存在的结构化候选或可重跑得到的 Marker 结果重新执行同一标题决策树；系统禁止仅以当前落盘标题作为 metadata 重新评分并将其称为结构化修复。

#### 场景:历史修复复用结构化候选
- **当** 某篇历史论文存在可访问的 Marker 结构化候选或可重跑得到该候选
- **那么** 修复流程必须沿结构化信任路径重算标题，并更新对应标题来源与置信度

#### 场景:历史修复缺少结构化输入
- **当** 某篇历史论文既无可用结构化候选也无法安全重跑 Marker
- **那么** 系统必须显式标记该论文无法进行结构化修复，而不是静默沿用旧标题完成“修复”

### 需求:chunk 数据结构
系统必须将解析结果写入 `chunks.jsonl`，且每个 chunk 记录必须包含 `chunk_id`、`paper_id`、`page_start`、`text` 字段。系统还必须支持通过清洗流程输出 `chunks_clean.jsonl` 增强记录，且禁止清洗过程覆盖已生成的 `chunks.jsonl` 与 `papers.json`。

#### 场景:任意 chunk 记录结构校验
- **当** 读取 `chunks.jsonl` 的任意一行并解析为对象
- **那么** 该对象必须包含 `chunk_id`、`paper_id`、`page_start`、`text`，且 `text` 不得为空字符串

#### 场景:联动清洗失败不影响主产物
- **当** ingest 主流程成功且 `--clean` 已开启，但清洗步骤执行失败
- **那么** 系统必须保留 `chunks.jsonl` 与 `papers.json`，并将清洗失败作为附加错误记录而非回滚主产物

### 需求:分块策略
系统必须优先依据可识别章节标题进行分段，并在分段过长时使用滑窗策略进行切块，目标窗口大小必须落在 300-500 token 近似区间，且相邻窗口必须包含约 50 token 重叠。系统必须在应用配置前完成参数合法性校验，禁止使用越界参数直接执行切块。对于 PDF 来源，系统必须优先使用结构化解析器提供的标题层级与块边界进行分段，并在结构化信息缺失时回退到文本规则分段。

#### 场景:超长段落触发滑窗
- **当** 某段文本超过单个 chunk 目标长度
- **那么** 系统必须将其拆分为多个重叠 chunk，且相邻 chunk 保持约 50 token 重叠

#### 场景:参数非法时回退后切块
- **当** 读取到非法 `chunk_size` 或 `overlap` 配置
- **那么** 系统必须先完成回退与告警，再按回退后的参数执行切块

#### 场景:结构化边界优先切块
- **当** 结构化解析结果包含章节层级与块边界
- **那么** 系统必须优先依据结构化边界分段，再在超长分段上执行滑窗切块

#### 场景:结构化信息缺失时回退
- **当** 当前文档缺少可用结构化边界
- **那么** 系统必须回退到文本规则分段并保持现有窗口大小与重叠约束

### 需求:页码可追溯性
系统必须保证每个 chunk 的 `page_start` 可追溯到原始 PDF 的实际起始页。

#### 场景:人工抽样校验页码
- **当** 从已处理论文中抽样 chunk 并回查 PDF 源页
- **那么** `page_start` 必须与文本起始页一致，允许少量噪声但不得系统性错位

### 需求:异常容错
系统必须在遇到单页解析异常时记录日志并跳过异常页，且不得因单页异常中止整篇或整批任务。

#### 场景:单页解析失败
- **当** 某 PDF 的某一页发生解析异常
- **那么** 系统必须记录该页异常信息并继续处理后续页面与后续文件

### 需求:最小稳定性验收
系统必须支持对至少 20 篇论文完成处理且流程无崩溃，以满足 M1 的最小可用验收要求。该验收前提还包括部署环境必须具备 PDF 解析所需的基础依赖，禁止因镜像缺包导致批处理在首个 PDF 上全部失败。系统还必须通过 M1.5 指定样例断言，以证明清洗与标注规则有效。

#### 场景:20 篇论文批处理
- **当** 输入目录中存在至少 20 篇可处理论文且部署环境依赖完整
- **那么** 系统必须完成运行并输出结果文件，且进程不得因缺少 PDF 解析依赖而异常退出

#### 场景:M1.5 样例断言通过
- **当** 执行 M1.5 验收检查
- **那么** 系统必须满足以下结果：`eff6f9d4b754:00109` 不含水印字符串；`eff6f9d4b754:00094` URL 归一化且含 `has_url`；`eff6f9d4b754:00093` 为 `reference`；`eff6f9d4b754:00095` 为 `dialogue_script`；`67978670bdb6:00004` 为 `front_matter`

### 需求:chunk 参数运行时校验
系统必须在运行时校验 `chunk_size` 与 `overlap` 配置值，并在非法值出现时应用安全回退，以避免切块行为异常。

#### 场景:chunk_size 越界
- **当** `chunk_size` 小于 300 或大于 500
- **那么** 系统必须记录告警并回退到默认安全值后继续执行

#### 场景:overlap 非法
- **当** `overlap` 小于 0 或大于等于生效的 `chunk_size`
- **那么** 系统必须记录告警并回退到默认安全值后继续执行

### 需求:Chunk 清洗与质量标注输出
系统必须基于 `data/processed/chunks.jsonl` 生成 `data/processed/chunks_clean.jsonl`，并为每条记录新增 `clean_text`、`content_type`、`quality_flags`、`section`（可空）字段，且必须保留原始 `text` 字段不变。

#### 场景:生成清洗后文件
- **当** 用户执行 chunk 清洗流程并输入 `data/processed/chunks.jsonl`
- **那么** 系统必须产出 `data/processed/chunks_clean.jsonl`，并为每条记录写入新增字段

### 需求:水印行过滤
系统必须在生成 `clean_text` 时删除包含以下子串的整行，匹配必须不区分大小写：`Authorized licensed use limited to`、`Downloaded on`、`IEEE Xplore`、`Restrictions apply`。

#### 场景:命中水印关键词的行被删除
- **当** 某 chunk 文本中某一行包含任一水印关键词
- **那么** 系统必须从 `clean_text` 中删除该整行

### 需求:URL 归一化与标记
系统必须将 `http://` 或 `https://` 链接归一化为 `<URL>`，并在 `quality_flags` 中追加 `has_url`。

#### 场景:chunk 含 URL
- **当** chunk 文本包含至少一个 HTTP/HTTPS 链接
- **那么** `clean_text` 中该链接必须替换为 `<URL>`，且 `quality_flags` 必须包含 `has_url`

### 需求:控制符清除与空白折叠
系统必须将不可见控制字符替换为空格，并折叠连续空白为单个空格或单个换行边界，以保证 `clean_text` 可读性。

#### 场景:文本含控制字符与多重空白
- **当** chunk 文本包含控制字符或连续空白
- **那么** 系统必须在 `clean_text` 中完成控制符清除与空白折叠

### 需求:乱码检测标记
系统必须计算 `weird_char_ratio`，当其大于 `0.35` 时，必须在 `quality_flags` 中追加 `garbled`，且禁止删除原始文本字段。

#### 场景:乱码比例超阈值
- **当** 某 chunk 的 `weird_char_ratio > 0.35`
- **那么** `quality_flags` 必须包含 `garbled`，并保留该 chunk 原始 `text`

### 需求:内容类型分类
系统必须基于规则对每个 chunk 赋值 `content_type`，并至少支持：`appendix`、`dialogue_script`、`reference`、`front_matter`、`watermark`、`formula_block`、`body`。

#### 场景:附录识别
- **当** 文本命中 `APPENDIX`
- **那么** `content_type` 必须为 `appendix`

#### 场景:对话脚本识别
- **当** 文本命中 `Player Choices`、`If the player chooses` 或 `Character:`
- **那么** `content_type` 必须为 `dialogue_script`

#### 场景:参考文献识别
- **当** 文本命中期刊与年份卷期页码模式
- **那么** `content_type` 必须为 `reference`

#### 场景:前置信息识别
- **当** 文本命中邮箱 `@` 或学校/单位信息模式
- **那么** `content_type` 必须为 `front_matter`

#### 场景:水印类型识别
- **当** 文本命中水印关键词
- **那么** `content_type` 必须为 `watermark`

#### 场景:公式块识别
- **当** 文本命中 `Eq.`、`(1)(2)(3)` 或高密度符号模式
- **那么** `content_type` 必须为 `formula_block`

#### 场景:默认正文类型
- **当** 文本不命中任何专门类型规则
- **那么** `content_type` 必须为 `body`

### 需求:同页短碎片合并
系统必须按 `paper_id + page_start` 分组并按 `chunk_id` 顺序扫描；当出现连续条目满足 `len(text)<=40` 且连续数量 `>=6` 时，必须合并为一个 `table_list` block chunk。合并结果必须将条目以换行拼接到 `clean_text`，并在 `quality_flags` 追加 `short_fragment_merged`；系统可选输出 `merged_from` 保存来源 `chunk_id` 列表。

#### 场景:连续短碎片达到阈值
- **当** 同页连续短碎片数量达到或超过 6
- **那么** 系统必须输出一个 `content_type=table_list` 的合并 block，并追加 `short_fragment_merged`

### 需求:入库主产物写入必须原子落盘
系统必须对 `chunks.jsonl`、`papers.json` 等主产物执行“临时文件 + 原子替换”写入协议，禁止在目标文件上直接追加或覆盖。

#### 场景:主产物写入成功
- **当** 入库流程完成主产物生成
- **那么** 目标文件必须为完整一致版本且不存在半成品可见窗口

#### 场景:主产物写入失败
- **当** 写入中途发生异常
- **那么** 系统必须保留旧版本目标文件并清理当前轮次临时文件

### 需求:入库写入与索引更新必须具备本地互斥
系统必须在本地写入与索引更新入口使用文件锁保护同一目标资源，禁止并发实例破坏数据一致性。

#### 场景:重复点击触发并发导入
- **当** 用户在短时间内重复触发导入并命中同一输出目录
- **那么** 系统必须阻止并发写入并向后触发方返回明确冲突提示

### 需求:Chunk 校验必须包含类型与约束
系统必须在 `chunk_id`、`paper_id`、`page_start`、`text` 字段存在校验之外执行类型与语义约束校验。

#### 场景:字段类型不合法
- **当** `page_start` 为非整数或 `chunk_id` 为 `null`
- **那么** 系统必须将该记录判定为非法并输出可定位的行级错误

#### 场景:文本为空或仅空白
- **当** `text` 为空字符串或仅包含空白字符
- **那么** 系统必须将该记录判定为非法并拒绝通过校验

### 需求:校验流程必须支持 fail-fast 阈值
系统必须支持 `max_errors` 阈值，在错误密集输入下提前终止并输出错误摘要。

#### 场景:超阈值停止校验
- **当** 校验错误数达到配置阈值
- **那么** 系统必须停止继续扫描并返回已收集错误及“已提前停止”标记

### 需求:入库流程必须支持增量合并
系统必须在导入时读取现有 `papers/chunks` 产物并进行去重合并，禁止默认覆盖历史已收录语料。

#### 场景:重复导入同一论文
- **当** 用户再次导入已存在论文
- **那么** 系统必须保留历史语料并将该论文标记为“已存在跳过”

#### 场景:新论文导入
- **当** 用户导入未收录论文
- **那么** 系统必须将该论文及其 chunks 追加到现有语料集合

### 需求:系统必须使用稳定论文标识
系统必须为同一来源内容生成稳定 `paper_id`，不得依赖临时目录路径导致标识漂移。

#### 场景:相同文件不同临时路径导入
- **当** 同一 PDF 经由不同临时路径重复导入
- **那么** 系统必须识别为同一论文并复用相同稳定 `paper_id`

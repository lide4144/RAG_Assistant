# evidence-policy-gate 规范

## 目的
待定 - 由变更 m7-evidence-policy 同步创建。后续可补充业务背景与边界。
## 需求
### 需求:系统必须优先使用结构化 claim 绑定执行门控
系统必须在生成最终回答前先构建结构化 `claim` 集合，并为每条关键结论显式绑定至少一个 `chunk_id`；当可用时，系统还必须记录对应 `section_id`。门控优先校验 claim 绑定完整性，禁止仅依赖对最终自然语言文本的后验字符串匹配作为唯一判据。

#### 场景:结构化 claim 绑定通过
- **当** 系统完成回答草稿并为每条关键结论生成结构化 claim 绑定
- **那么** 系统必须仅使用已绑定 claim 渲染最终回答，且门控以 claim 绑定结果作为主判据

#### 场景:claim 缺少来源时降级
- **当** 某条关键结论无法绑定到任何有效 `chunk_id`
- **那么** 系统必须剔除该 claim 或降级回答，不得将其作为强断言输出

### 需求:格式性编号不得触发数值型误判
系统必须将列表编号、标题编号、页码编号等格式性数字与真正的数值性关键结论区分开来。系统禁止将 `1.`、`2)`、`第 3 章`、`p.8` 等纯格式标记单独判定为需要证据支持的数值型 claim。

#### 场景:列表编号不触发误杀
- **当** 回答中的数字仅用于列表编号或章节编号
- **那么** 系统必须禁止将该数字单独识别为数值型关键结论

#### 场景:页码标记不触发误杀
- **当** 回答或引用中出现 `p.8`、`第 3 节` 等定位信息
- **那么** 系统必须将其识别为格式性定位标记，而不是独立事实结论

### 需求:关键结论必须绑定可追溯 citation
系统必须对回答中的关键结论执行引用覆盖校验。关键结论至少包括数字陈述、实验结果陈述、定义性陈述与结论性判断句。系统必须优先基于结构化 claim 绑定结果判断每条关键结论是否已被 citation 覆盖；每条关键结论必须关联至少一个 citation，且 citation 必须包含 `chunk_id`、`paper_id` 与 `section_page`。

#### 场景:关键结论覆盖通过
- **当** 系统完成回答草稿并识别出关键结论集合
- **那么** 每条关键结论必须至少绑定 1 条包含 `chunk_id`、`paper_id` 与 `section_page` 的 citation，未覆盖则不得进入最终回答

#### 场景:结构化 claim 优先于文本后验匹配
- **当** 系统已生成结构化 claim 绑定
- **那么** 门控必须优先以 claim 绑定结果为准，不得因列表编号或排版数字导致整段回答被误判拒答

### 需求:citation 必须可定位到已展示证据
系统必须验证每条关键结论绑定的 citation 能在 `evidence_grouped` 中定位到同一 `chunk_id` 的证据项。若 citation 无法定位，系统必须将该结论判定为未被证据支撑。

#### 场景:citation 无法定位
- **当** 关键结论绑定的 citation 在 `evidence_grouped` 中不存在对应 `chunk_id`
- **那么** 系统必须判定校验失败并触发证据不足门控流程

### 需求:证据不足必须进入 M8 Sufficiency Gate
当任一关键结论缺失 citation、citation 不可定位、语义支撑不足或 Gate 明确判定不通过时，系统必须进入 M8 Sufficiency Gate，禁止输出强断言回答，并追加 `insufficient_evidence_for_answer` 告警。

#### 场景:触发 M8 门控
- **当** 关键结论校验存在未通过项或 Gate 判定不通过
- **那么** 系统必须输出门控后的弱回答模板，并在 `output_warnings` 中包含 `insufficient_evidence_for_answer`

### 需求:LLM 回答输出必须经过 Gate 二次校验
系统必须对 `llm_answer_with_evidence` 的输出执行与模板回答一致的 evidence policy gate 校验，禁止绕过门控直接输出。

#### 场景:LLM 回答进入统一门控
- **当** 上游生成来源为 `llm_answer_with_evidence`
- **那么** 系统必须执行关键结论覆盖、citation 可定位与语义一致校验后才可输出最终回答

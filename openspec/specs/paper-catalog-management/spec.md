## 需求
### 需求:系统必须提供可检索的论文目录管理页
系统必须在 Library 中提供论文目录管理视图，支持按标题、来源、专题与状态进行检索和筛选。

#### 场景:用户按关键词定位论文
- **当** 用户在目录页输入关键词
- **那么** 系统必须返回标题或来源匹配的论文列表

#### 场景:用户按专题筛选论文
- **当** 用户选择某专题筛选
- **那么** 系统必须仅展示属于该专题范围的论文

### 需求:系统必须支持聊天侧复用目录筛选结果
系统必须允许聊天侧通过目录查询能力获取论文集合，并将结果作为后续总结或问答步骤的输入范围；禁止要求聊天侧仅通过正文 chunk 检索反推论文列表。

#### 场景:聊天侧先筛选再对比
- **当** 用户输入“列出我昨天上传的 3 篇大模型论文，并用表格对比一下它们的方法差异”
- **那么** 系统必须能够先返回满足条件的论文集合，并将该集合提供给后续对比步骤

### 需求:系统必须对目录结果集施加硬上限并披露截断
当目录查询命中数量超过后续执行上限时，系统必须裁剪为受控的 `selected_paper_set`，并输出 `matched_count`、`selected_count` 与 `truncated`；禁止将未经裁剪的大结果集直接交给后续总结或问答流程。

#### 场景:目录命中过大时披露截断
- **当** 目录查询命中 500 篇论文而下游执行上限为 20
- **那么** 系统必须仅返回受控的 20 篇论文集合并披露已发生截断

### 需求:系统必须展示论文可维护元数据
目录页必须展示每篇论文的稳定标识、标题、来源类型、导入时间与当前状态，支持用户判断是否重复导入或待处理。对于聊天侧复用场景，目录查询返回结果也必须至少包含上述核心元数据，以支持后续计划执行与结果披露。

#### 场景:用户查看论文详情
- **当** 用户打开某论文详情
- **那么** 系统必须展示核心元数据与专题归属信息

#### 场景:聊天侧目录筛选返回可执行元数据
- **当** 聊天侧通过目录筛选得到论文集合
- **那么** 返回结果必须包含 `paper_id`、标题、导入时间、状态等可用于计划执行的核心元数据

### 需求:系统必须支持目录级专题批量操作
系统必须允许用户在目录页对多篇论文进行专题归类或移除操作，并保持与 Chat 侧 topic scope 一致。

#### 场景:批量加入专题
- **当** 用户选择多篇论文并指定专题
- **那么** 系统必须一次性更新专题映射并反馈变更数量

### 需求:目录查询必须基于 SQLite 权威论文记录
目录查询、筛选与详情展示必须以 SQLite 中的权威论文记录与生命周期状态为准。系统禁止继续依赖 `papers.json`、导入报告或临时聚合结果作为目录页主数据源。

#### 场景:目录页按状态筛选论文
- **当** 用户在目录页按 `ready`、`failed`、`rebuild_pending` 或等价生命周期状态筛选论文
- **那么** 系统必须直接基于 SQLite 论文记录返回结果

#### 场景:目录页显示稳定来源信息
- **当** 用户查看某篇论文的目录详情
- **那么** 系统必须展示该论文的稳定来源信息、生命周期状态与更新时间，而不是展示临时导入路径

### 需求:目录结果必须披露论文级处理状态
目录结果必须披露每篇论文是否已完成可用、是否存在失败阶段、是否待重建，以及与当前查询相关的核心维护元数据。系统禁止仅显示笼统的 `active` 或不可区分的单状态结果。

#### 场景:用户识别待重建论文
- **当** 某篇论文的索引或图谱已过期
- **那么** 目录结果必须将其显示为待重建或等价状态，供用户采取后续操作

#### 场景:聊天侧复用目录筛选结果
- **当** 聊天侧通过目录查询能力获取论文集合
- **那么** 返回结果必须包含可用于后续执行的稳定 `paper_id` 与生命周期状态

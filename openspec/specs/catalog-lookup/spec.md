## 新增需求

### 需求:目录查询结果必须通过LLM生成自然语言
目录查询的结果禁止直接格式化输出。必须通过LLM生成自然语言的查询结果描述。

#### 场景:查询最近导入论文
- **当** 用户请求"列出最近导入的论文"
- **那么** 系统必须调用LLM生成自然语言回答
- **并且** 禁止直接输出格式化的论文列表

#### 场景:按条件筛选论文
- **当** 用户请求"查找关于机器学习的论文"
- **那么** 系统检索匹配论文后调用LLM生成回答
- **并且** LLM的回答必须包含论文简介和相关性说明

## 修改需求

### 需求:移除compose_catalog_answer的硬编码路径
catalog-lookup功能必须移除直接格式化输出的逻辑，所有目录查询结果必须经过LLM处理。

#### 场景:无匹配论文
- **当** 目录查询未找到匹配论文
- **那么** 禁止返回"未找到符合条件的论文"
- **并且** 必须调用LLM生成友好的未找到提示

#### 场景:返回多本论文
- **当** 目录查询找到多本匹配论文
- **那么** 禁止直接输出"基于目录元数据，匹配到这些论文："
- **并且** 必须调用LLM生成论文列表的自然语言描述

### 需求:目录元数据上下文格式化
目录查询必须将论文元数据（标题、作者、年份、状态等）格式化为LLM可理解的上下文。

#### 场景:元数据整合
- **当** 检索到匹配论文的元数据
- **那么** 将元数据格式化为结构化上下文
- **并且** 在LLM提示词中明确说明每个字段的含义

#### 场景:状态信息处理
- **当** 论文存在不同的处理状态（就绪、处理中、错误等）
- **那么** 将状态信息作为上下文传递给LLM
- **并且** 由LLM决定如何在回答中呈现状态信息

## 移除需求

### 需求:硬编码目录查询输出格式
**Reason**: 原逻辑直接输出"基于目录元数据，匹配到这些论文："并列出论文信息
**Migration**: 使用LLM基于论文元数据上下文生成自然语言的查询结果描述

### 需求:硬编码统计信息输出
**Reason**: 原逻辑硬编码了论文库统计信息的输出格式
**Migration**: 由LLM根据查询结果动态生成统计说明

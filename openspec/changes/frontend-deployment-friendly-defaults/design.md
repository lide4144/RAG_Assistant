## 上下文

当前前端把浏览器侧 HTTP 管理接口和 WebSocket 连接的默认目标写死为本机回环地址。这个策略仅适合开发者在同一台机器上同时运行 `frontend`、`gateway`、`kernel` 的联调场景；一旦页面从远程服务器提供给浏览器，`localhost` / `127.0.0.1` 就会被解析为终端用户自己的设备，导致运行态概览读取失败、聊天连接失败和知识处理页任务事件不可用。

这个变更横跨 Next.js 前端多个页面、环境变量注入策略、浏览器端地址推导逻辑和部署文档，属于跨模块行为修正。设计重点不是“换一个默认字符串”，而是统一前端在浏览器侧如何解析后端目标地址，并确保本地开发、同域反代和 HTTPS 部署三种场景都能成立。

## 目标 / 非目标

**目标：**
- 为浏览器侧 HTTP 管理接口定义部署友好的默认策略：未显式配置时默认走同域相对路径。
- 为浏览器侧 WebSocket 定义部署友好的默认策略：未显式配置时根据当前页面来源自动推导 `ws://` 或 `wss://`。
- 保留显式环境变量最高优先级，避免破坏本地分端口联调和特殊部署。
- 将地址解析逻辑集中到共享辅助层，避免 `chat`、`pipeline`、`app shell`、`settings shell` 各自维护不同默认值。
- 补充文档和测试，明确远程部署与本地联调的差异。

**非目标：**
- 不在本变更中引入新的反向代理实现或服务器编排脚本。
- 不在本变更中把所有前后端流量都改为必须经由 Next.js 代理。
- 不修改网关 WebSocket 协议、Kernel API 结构或业务回答逻辑。
- 不处理控制台中与本仓库无关的第三方脚本弃用提示。

## 决策

### 决策 1：新增共享“前端端点解析”能力，而不是在页面内继续散落默认值
前端需要引入统一的浏览器侧端点解析函数，用于生成：
- 管理接口基础地址
- WebSocket 地址

选择理由：
- 当前 `chat-shell`、`pipeline-shell`、`app-shell` 都各自保留默认值，容易再次出现一处修复、另一处遗漏。
- 共享解析函数更适合覆盖优先级、协议映射和测试用例。

替代方案：
- 分别在各页面组件中修改默认字符串。缺点是行为分散，后续维护容易回归。

### 决策 2：HTTP 管理接口默认走同域相对路径，而不是继续注入 `127.0.0.1`
当 `NEXT_PUBLIC_KERNEL_BASE_URL` 未配置时，浏览器侧对 `/api/admin/*` 的请求默认使用相对路径，由 Next.js rewrite 或上游反向代理决定真实去向。

选择理由：
- 相对路径天然适配“前端和 API 同域”部署模式，避免浏览器跨到用户本机。
- 与现有 `next.config.mjs` 中 `/api/admin/:path*` rewrite 机制一致，能复用已有稳定路径。

替代方案：
- 默认改成服务器公网 IP。缺点是构建时无法知道正确外部地址，且会把环境耦合写死在代码里。

### 决策 3：WebSocket 默认基于 `window.location` 推导 `/ws`
当 `NEXT_PUBLIC_GATEWAY_WS_URL` 未配置时，前端在浏览器中使用当前页面来源生成 WebSocket 地址：
- `http:` 对应 `ws:`
- `https:` 对应 `wss:`
- host 和端口复用当前页面来源
- path 使用 `/ws`

选择理由：
- 这是最符合浏览器部署语义的默认策略，特别适合 HTTPS + 反向代理场景。
- 自动切换到 `wss` 可以避免 mixed content 问题。

替代方案：
- 默认继续指向 `ws://localhost:8080/ws`。缺点是远程部署必坏。
- 默认指向相对 `'/ws'`。缺点是 `WebSocket` 构造需要绝对 URL 或手动补全协议，仍要依赖浏览器端推导。

### 决策 4：显式环境变量优先于推导默认值
如果设置了 `NEXT_PUBLIC_KERNEL_BASE_URL` 或 `NEXT_PUBLIC_GATEWAY_WS_URL`，前端必须直接使用这些值，不得再覆盖或二次拼接。

选择理由：
- 兼容本地多端口联调和复杂网络拓扑。
- 避免破坏已有部署或测试环境。

替代方案：
- 始终强制同域。缺点是会损伤开发灵活性，也不适合分离式部署。

### 决策 5：开发启动脚本必须尊重外部传入的前端端点变量
`scripts/dev-up.sh` 可以为本地联调提供默认注入，但当调用者已经显式传入 `NEXT_PUBLIC_KERNEL_BASE_URL` 或 `NEXT_PUBLIC_GATEWAY_WS_URL` 时，脚本必须保留这些值，不得覆盖为 `KERNEL_HOST/KERNEL_PORT` 派生地址。同时，当 `KERNEL_HOST=0.0.0.0` 时，Gateway 内部访问 Kernel 必须改用可连接的回环地址而不是 `0.0.0.0`。

选择理由：
- 远程开发环境（如 Cloud Studio、codespace、反向代理沙箱）依赖显式的公网或代理域名，脚本覆盖会直接破坏部署友好策略。
- `0.0.0.0` 适合监听，不适合作为客户端请求目标。

替代方案：
- 继续让脚本强制覆盖。缺点是会让“显式环境变量优先”在实际启动链路中失效。

### 决策 6：文档需要显式区分“开发注入”和“部署默认”
`scripts/dev-up.sh` 仍可为本地联调注入本机地址，但文档必须说明这只是开发模式。部署说明必须把推荐路径切换为同域反代或显式公网地址/WSS。

选择理由：
- 问题本质之一就是开发时的默认值被误用到了部署场景。
- 仅改代码不改文档，用户仍然会沿着错误心智模型部署。

替代方案：
- 保持文档不变。缺点是会继续放大“本地可用、服务器不可用”的误解。

## 风险 / 权衡

- [部署环境未配置 `/ws` 反向代理] → 前端默认同域后，WebSocket 仍会失败；在文档中明确要求代理转发 `/ws` 到 gateway，并保留环境变量直连覆写。
- [部分本地开发依赖直连 `127.0.0.1:8000`] → 保留 `NEXT_PUBLIC_KERNEL_BASE_URL` 优先级，并仅在未显式传值时由 `scripts/dev-up.sh` 注入本地默认值。
- [测试环境依赖旧默认值] → 增补针对默认推导逻辑的测试，避免只在人工联调时发现回归。
- [服务端渲染阶段无法访问 `window`] → 解析函数需要区分浏览器端与构建阶段，默认仅在客户端安全求值。

## Migration Plan

1. 新增共享端点解析辅助函数，并在聊天页、知识处理页、设置页和壳层统一接入。
2. 将浏览器侧管理接口默认基址改为相对路径策略，保留环境变量覆盖。
3. 将浏览器侧 WebSocket 默认地址改为基于 `window.location` 的协议感知推导。
4. 调整 `scripts/dev-up.sh`，仅在未显式传值时注入前端端点变量，并确保 Gateway 内部访问 Kernel 使用可连接地址。
5. 更新 `startup-guide` 与相关部署说明，明确本地联调和远程部署的推荐配置。
6. 增补前端测试，覆盖未配置地址、HTTPS/WSS 和显式环境变量覆盖三类场景。

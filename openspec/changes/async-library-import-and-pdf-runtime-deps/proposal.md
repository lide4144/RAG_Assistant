## 为什么

当前前端论文导入将多个 PDF 直接拼成单个浏览器请求，并在同一 HTTP 请求内同步完成入库、清洗与索引更新。批量上传时，这种模式容易在代理层或浏览器侧表现为 `Failed to fetch`，用户看不到真实导入状态。与此同时，生产环境部署清单未显式声明 `PyMuPDF`，导致 PDF 解析依赖在部分镜像中缺失，导入任务只能在运行时失败。

此外，前端 `llm-settings-panel.spec.ts` 中的测试状态对象缺少显式类型，导致 TypeScript 将 mock 状态推断为过窄结构，阻塞全量 `tsc --noEmit` 校验。

## 变更内容

- 将浏览器侧论文上传改为“先接收文件并落盘，再启动后台导入任务”，避免单个长请求承载完整导入链路。
- 将 `library_import` 纳入现有统一任务状态机，使导入任务与图构建任务共享 `queued/running/succeeded/failed/cancelled` 契约。
- 在前端导入工作台中接入导入任务轮询与结果刷新，提交成功后立即反馈 `task_id` 与后台运行状态。
- 将 `pymupdf` 明确加入运行时依赖清单，确保部署镜像具备 PDF 解析基础依赖。
- 修复设置页测试中的过窄类型推断问题，使 TypeScript 全量检查恢复可用。

## 功能 (Capabilities)

### 修改功能
- `library-quick-ingestion`: 批量浏览器上传必须异步受理并返回后台任务标识，避免长请求直接向用户暴露网络层失败。
- `pipeline-task-observability`: 统一任务状态机扩展到论文导入任务，前端可复用相同状态查询与终态处理机制。
- `paper-ingestion-pipeline`: PDF 解析运行环境必须显式包含 `PyMuPDF` 依赖，避免部署镜像缺包导致整批导入失败。

## 影响

- Python Kernel：新增文件暂存与后台导入线程，导入接口从同步完成改为异步受理。
- 前端：导入面板新增导入任务状态轮询与终态刷新逻辑。
- 部署：生产镜像需要随 `requirements.txt` 安装 `pymupdf`。
- 测试：新增导入任务后台执行回归测试，并修复设置页测试类型定义。

## 为什么

当前项目默认通过 `3000`、`8000`、`8080` 多端口分别暴露前端、Kernel 和 Gateway，这在本地联调可行，但在生产和托管环境中会带来跨域、`wss`、端口暴露和部署复杂度问题。项目需要一个明确、可落地的 Nginx 反向代理方案，把浏览器访问收敛到单域入口，降低生产部署门槛。

## 变更内容

- 新增面向生产环境的 Nginx 反向代理部署能力，统一前端页面、HTTP API 与 WebSocket 入口。
- 为前端部署相关能力补充单域同源访问要求，避免浏览器依赖 `8000/8080` 端口直连。
- 提供项目内可直接采用的 Nginx 配置示例、部署步骤和环境变量约定。
- 明确本地 `dev-up.sh` 联调模式与生产反代模式的边界，避免混用。

## 功能 (Capabilities)

### 新增功能

- `nginx-production-reverse-proxy`: 定义单域 Nginx 入口如何代理前端、`/api/*` 和 `/ws`，以及生产部署所需的配置与验证要求。

### 修改功能

- `frontend-saas-shell-navigation`: 前端壳层与各工作台页面需要补充“生产环境优先通过同域反代访问后端”的行为要求。
- `project-readme-guide`: 项目部署文档需要新增生产环境反代部署指南，并说明与本地开发脚本的区别。

## 影响

- 受影响代码和配置包括 `frontend/next.config.mjs`、前端端点解析逻辑、部署脚本与文档。
- 新增 Nginx 配置模板或示例部署文件。
- 生产部署将依赖 Nginx 或等价的反向代理层来暴露统一域名与 TLS。

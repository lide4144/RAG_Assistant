## 上下文

当前仓库已经具备两块相关基础能力：一是 `frontend/components/settings-shell.tsx` 已支持 stage 级 LLM 配置与 Marker tuning 保存，二是 runtime overview 已经聚合了 LLM stage 与 pipeline tuning 摘要。但 Marker `--use_llm` 的 provider 族配置尚未进入统一设置面板，导入链路也没有向前端暴露“LLM 增强失败但已降级继续”的专用状态。与此同时，知识处理页主要关注阶段任务与日志，缺少对 `data/indexes`、`data/processed` 等既有产物的管理视角，导致用户看到问题时只能依赖文件系统或日志定位。

这个变更横跨前端设置页、运行态持久化接口、Marker 导入状态契约与知识处理体验，属于典型的跨领域变更。设计需要保证三件事同时成立：前端可配置、后端可校验、导入结果可解释。

## 目标 / 非目标

**目标：**
- 将 Marker `--use_llm` 服务配置纳入现有设置页和 runtime overview，统一保存、读取和回显。
- 在导入失败降级时提供稳定的状态契约，使前端能够展示 `DEGRADED`、`failed_with_fallback` 等明确语义。
- 为 `Index` 与 `processed` 产物提供单独的管理视图或管理面板，并与流水线阶段和风险摘要联动。
- 在前端展示层引入 Magic MCP 组件生成的视觉骨架，用于风险卡片、状态摘要和产物面板，提升可读性而不改动核心交互模型。

**非目标：**
- 不在本变更中重写整个设置页或知识处理页路由结构。
- 不在本变更中引入新的 Marker provider；范围仅覆盖 Marker 文档已列出的服务族。
- 不在本变更中改变实际解析算法或新增复杂的后台任务调度系统。

## 决策

### 决策 1：复用现有 pipeline runtime 配置接口，扩展 Marker 配置子结构
将 Marker LLM 配置作为现有 pipeline runtime payload 的扩展字段，而不是新建独立配置接口。推荐结构包含 `marker_tuning` 与 `marker_llm` 两个子对象，其中 `marker_llm` 保存 `use_llm`、`llm_service`、provider 参数和脱敏摘要。

选择理由：
- 现有设置页已经一次性读取和保存 pipeline runtime 配置，扩展现有契约的改动最小。
- runtime overview 已聚合 pipeline 信息，复用原接口能减少前端请求拆分和状态不一致。

替代方案：
- 新增独立 `marker-llm-config` 接口。缺点是设置页需要双请求、双保存状态，且更容易产生“tuning 已保存但 LLM service 未保存”的局部不一致。

### 决策 2：将降级状态建模为显式运行态，而不是文本日志推断
后端需要在导入报告和运行态概览中显式返回降级字段，如 `degraded`, `fallback_reason`, `fallback_path`, `confidence_note`，前端禁止通过解析自由文本日志推断是否降级。

选择理由：
- 日志文本不稳定，无法支撑可靠 UI 映射和测试。
- 显式状态更适合纳入 `StatusMapper` 和可视化风险摘要。

替代方案：
- 仅在日志中拼接“fallback”关键字。缺点是脆弱且难以国际化、难做字段级测试。

### 决策 3：产物管理优先做“可见性 + 受控操作”，而不是完整文件浏览器
前端产物管理视图只暴露关键产物列表、元数据、健康状态和受控操作，不提供通用目录树浏览器。

选择理由：
- 用户关注的是 `processed` 和 `indexes` 的业务状态，而不是任意文件探索。
- 受控模型更利于限制破坏性操作并提供针对性的确认文案。

替代方案：
- 直接提供文件系统浏览器。缺点是交互复杂、权限边界模糊、视觉噪音高。

### 决策 4：Magic MCP 仅用于高价值展示组件，不作为页面结构生成器
Magic MCP 组件用于生成状态卡、降级警报区、产物抽屉或面板等高价值 UI 模块，再按现有页面结构进行整合，而不是整页替换。

选择理由：
- 现有页面已经有稳定布局和测试，模块化接入风险更低。
- 能满足你对视觉效果升级的要求，同时避免大面积重构。

替代方案：
- 直接重做整页。缺点是超出本次变更范围，并会放大回归风险。

## 风险 / 权衡

- [不同 LLM service 字段差异大，表单复杂度上升] → 通过 provider schema 映射统一驱动字段显示、校验和摘要生成，避免散落条件分支。
- [降级状态过多导致前端映射失控] → 将状态压缩为少量稳定枚举，并把详细原因放入辅助字段。
- [产物删除或重建操作具有破坏性] → 所有破坏性操作必须二次确认，并优先提供“重建入口”而非直接删除。
- [Magic MCP 组件可能与现有样式系统不完全一致] → 将其限定在局部容器内，并以现有 design token 包裹，避免全局样式污染。

## Migration Plan

1. 扩展 runtime config 数据模型与管理接口，支持 `marker_llm` 保存、读取、校验和脱敏回显。
2. 扩展导入报告与 runtime overview 契约，加入降级状态和产物健康摘要。
3. 在设置页加入 Marker LLM service 配置表单和摘要区，并保留现有 marker tuning 行为。
4. 在知识处理页加入降级状态卡与产物管理入口，再集成 Magic MCP 组件优化视觉层。
5. 补充端到端测试与接口校验测试，确保 provider 字段错误、降级提示和产物视图行为稳定。

## Open Questions

- 产物管理中的“删除”是否在首版开放，还是仅提供查看与重建入口更稳妥。
- `processed` 和 `indexes` 健康状态是实时扫描文件系统，还是基于最近一次导入报告缓存聚合。
- Marker LLM 配置中的敏感字段是否需要支持“已保存但不可回显”的独立占位语义。

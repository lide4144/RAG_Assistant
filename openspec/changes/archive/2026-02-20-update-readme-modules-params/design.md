## 上下文

当前 README 已覆盖里程碑级流程和主入口命令，但缺乏模块级视角：用户难以快速理解每个模块（ingest、clean_chunks、build_indexes、graph_build、qa、rewrite 等）的单独职责、输入输出和参数影响。对于需要精细调参或定位故障的场景，文档粒度不足会导致重复试错与源码追踪成本上升。

该变更仅针对文档信息架构，不改变应用代码、运行逻辑或配置默认值。

## 目标 / 非目标

**目标：**
- 将 README 从“流程导向”扩展为“流程 + 模块导向”双视角文档。
- 为关键模块提供统一结构说明：用途、输入、输出、核心命令、参数解释、示例组合。
- 为关键配置项提供参数字典，明确默认值、推荐范围、行为影响与常见误用。
- 增加故障排查与验证指引，降低首次接入和协作调试成本。

**非目标：**
- 不新增 CLI 参数。
- 不调整 `configs/default.yaml` 中现有配置语义。
- 不修改任何运行时代码、索引格式或日志结构。

## 决策

### 决策 1：README 采用“全局流程 + 模块手册 + 参数字典”三段式结构
- 方案 A（采用）：在保留现有全流程说明的基础上，新增模块化章节与参数参考章节。
- 方案 B（未采用）：仅在原有章节内追加参数注释。
- 理由：方案 A 信息检索效率更高，适配“按模块排障”和“按参数调优”两类高频任务。

### 决策 2：参数说明覆盖 CLI 与配置两层
- 方案 A（采用）：每个模块同时说明 CLI 参数和对应配置项（若存在）。
- 方案 B（未采用）：仅记录 CLI 参数。
- 理由：项目行为由 CLI 与配置共同决定，只覆盖 CLI 会遗漏关键行为开关。

### 决策 3：参数解释必须包含“影响/副作用”
- 方案 A（采用）：每个关键参数写明作用、默认行为、调整后影响和典型使用场景。
- 方案 B（未采用）：仅描述“参数含义”。
- 理由：仅有定义不够指导实践；影响说明能直接支持调参与回归验证。

## 风险 / 权衡

- [风险] README 过长影响初学者阅读路径
  → 缓解：保留快速开始路径，并将进阶内容集中到“模块手册/参数字典”章节。

- [风险] 文档与代码演进不同步
  → 缓解：在规范中要求 README 与实际命令、参数名保持一致，并在任务中增加一致性检查。

- [权衡] 文档维护成本上升
  → 缓解：采用统一模板，新增参数时按模板最小增量更新。

## 迁移计划

1. 更新 README 目录与章节结构。
2. 增补各模块用法与参数说明。
3. 通过一次命令级核对，确保文档中的入口名、参数名、路径名与当前实现一致。
4. 合并后在后续变更中按模板持续维护。

回滚策略：若结构调整导致可读性下降，可回滚到变更前 README 版本，不影响运行时行为。

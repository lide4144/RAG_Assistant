## 上下文

现有输出治理已定义 `answer_citations` 与证据分配规则，但缺少“结论粒度”的强制绑定与校验流程：系统可输出 citation，却未保证关键句（尤其是数字、实验结果、定义句）都被引用覆盖且可在证据中核验。M7 需要把这一约束前移到回答生成阶段，并在证据不足时统一进入 M8 Sufficiency Gate。

约束：
- 不改变现有 QA JSON 主体结构，优先在现有字段上强化约束。
- 兼容已有 evidence_grouped / answer_citations 管线，不引入破坏性 API 变更。
- 校验规则必须可自动测试，以支持 30 题回归与 10 条人工抽检。

## 目标 / 非目标

**目标：**
- 建立“关键结论 -> citation -> evidence chunk”三段可追溯链路。
- 对数字、实验结果、定义句实施强约束：无证据则降级，不允许强断言。
- 统一证据不足分流：在回答前触发 M8 Sufficiency Gate。
- 提供可执行验收路径：30 题全带 citations，10 条抽检可在 chunk 找到语义一致支撑。

**非目标：**
- 不在 M7 中重做检索/rerank 算法本身。
- 不在 M7 中新增复杂事实核查模型（如外部 verifier service）。
- 不改变上游 chunk 生产结构（仅消费既有 chunk/page/section 元数据）。

## 决策

1. 在 answer assembly 后增加“claim-citation coverage check”步骤
- 理由：最小侵入接入现有流水线，避免改动检索核心。
- 替代方案 A：在 LLM prompt 阶段强制输出逐句引用。未选原因：模型输出稳定性不足，仍需后校验。
- 替代方案 B：离线批处理修补 citations。未选原因：无法阻止在线错误回答。

2. 定义关键结论集合并执行强制覆盖
- 关键结论包括：数值句、实验结果句、定义句、结论性判断句（基于规则+轻量模式识别）。
- 理由：把“必须引用”的范围收敛为可测试对象。
- 替代方案：全句强制引用。未选原因：会显著增加冗余引用并影响可读性。

3. citation 最低字段强制为 `chunk_id + section_page`
- 理由：满足“可定位 + 可核验”最小要求，与现有结构兼容。
- 替代方案：只保留 `paper_id`。未选原因：粒度过粗，无法追溯到 chunk。

4. 证据不足统一进入 M8 Sufficiency Gate
- 触发条件：关键结论无法匹配 citation，或 citation 无法在 evidence_grouped 中定位，或定位后语义不一致。
- 理由：将失败路径统一，减少分散降级逻辑。
- 替代方案：仅 warning 不降级。未选原因：不能满足 M7 硬约束。

5. 增加离线验收工具链
- 批量回归脚本验证“30 题 citations 完整率=100%”。
- 抽检辅助输出（question、claim、citation、chunk snippet）支持人工核验 10 条。

## 风险 / 权衡

- [风险] 规则识别关键结论可能漏检或误检 -> 缓解：先覆盖数字/实验结果/定义句等高价值类型，并保留可配置规则。
- [风险] 语义一致性判断主观性较强 -> 缓解：采用“可定位 + 关键词/数值重叠 + 人工抽检”分层标准。
- [风险] 强约束可能提高“证据不足”触发率，导致回答更保守 -> 缓解：在 M8 模板中提供下一步检索建议，减少用户阻塞。
- [权衡] 不引入重型 verifier 降低实现成本，但自动判定精度受限。

## 迁移计划

1. 在现有 answer post-process 插入 claim-citation 校验模块（默认开启）。
2. 扩展 `answer_citations` 生成与校验，确保每条含 `chunk_id`、`section_page`。
3. 接入 M8 Gate：命中失败条件即输出弱回答并带 warning。
4. 新增自动化验收脚本与样例集；在 CI 中执行 30 题回归。
5. 发布后观察 `insufficient_evidence_for_answer` 告警比例，必要时调优判定阈值。

回滚策略：
- 保留开关 `evidence_policy_enforced`，紧急情况下可降级为“仅告警，不拦截”模式。

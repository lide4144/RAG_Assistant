## 新增需求

### 需求:本地文档导入必须采用默认轻量解析与可选增强解析模型
系统必须将本地文档导入建模为“基础解析器 + 可选增强解析器”两层结构。基础解析器必须作为本地默认可用路径存在；增强解析器仅在显式启用且可用时参与导入。该模型的降级逻辑必须参考 `example_projects/LightRAG` 的本地文档处理方法，即优先保证可用文本抽取，而不是将增强解析作为导入前提。

#### 场景:默认本地导入不依赖增强解析器
- **当** 用户在个人电脑环境执行本地文档导入且未显式启用增强解析
- **那么** 系统必须仍可通过基础解析器完成导入并输出可检索产物

#### 场景:增强解析失败不阻断导入
- **当** 用户显式启用增强解析器但当前环境无法满足其前置条件
- **那么** 系统必须自动回退到基础解析器并继续完成导入，同时记录增强降级事实

### 需求:系统必须支持多种本地文档类型的统一受理与路由
系统必须支持多种本地文档类型的统一受理，并按文件类型将输入路由到对应的基础解析器。首版至少必须为以下几类建立稳定策略：

- 纯文本与标记类：`txt/md/mdx/html/htm/tex/json/xml/yaml/yml/csv/log/conf/ini/properties/sql`
- PDF：`pdf`
- Office：`docx/pptx/xlsx`
- 文档兼容类：`rtf/odt/epub`
- 代码类：`c/h/cpp/hpp/py/java/js/ts/swift/go/rb/php/css/scss/less`

系统可以对部分类型先提供“轻量抽取”或“受控跳过并给出明确提示”，但禁止继续把本地导入语义局限为只处理 PDF。

#### 场景:文本类文件直接进入基础解析
- **当** 用户导入 `txt/md/html/csv/json/xml/yaml` 等文本类文件
- **那么** 系统必须直接使用轻量文本解析路径完成导入，而不是要求增强解析器参与

#### 场景:Office 文件走类型化基础解析
- **当** 用户导入 `docx/pptx/xlsx`
- **那么** 系统必须按文件类型路由到各自基础解析器，并输出统一导入结果语义

#### 场景:兼容类文件暂未完整支持时受控跳过
- **当** 用户导入 `rtf/odt/epub` 且当前版本尚未具备稳定基础解析器
- **那么** 系统必须返回受控跳过结果与明确提示，而不是无上下文失败

## 修改需求

### 需求:输入文件发现与过滤
系统必须处理输入中的 `.pdf` 文件与有效 URL，且必须忽略不支持的文件类型与非法 URL 条目，不得中断整批处理。对于本地文件导入，系统还必须允许按文件类型将输入路由到对应的基础解析器，并为后续扩展 `docx/pptx/xlsx` 等类型预留统一降级框架。

#### 场景:本地文件按类型路由到基础解析器
- **当** 导入流程识别到本地文件类型属于已支持的基础解析范围
- **那么** 系统必须按文件类型选择相应基础解析器，而不是要求统一依赖增强解析器

#### 场景:混合本地文件批量导入
- **当** 用户一次导入 `pdf/docx/pptx/xlsx/txt/md` 等混合文件集合
- **那么** 系统必须逐文件按类型执行基础解析或增强降级，并保证单文件失败不会中断整批处理

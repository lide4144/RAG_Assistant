## 上下文

当前 `sufficiency-gate` 的 `topic_match_score` 依赖统一分词函数进行问题与证据 token 交集计算。在中英无空格混写场景（如 `Transformer是什么`）下，问题 token 可能被识别为单个复合片段，导致与证据中的 `transformer` 无交集，最终触发 `topic_mismatch` 误拒答。与此同时，拒答模板文案默认指向 Sufficiency Gate，难以区分是否由 Evidence Policy Gate 二次触发拒答，增加排障成本。

## 目标 / 非目标

**目标：**
- 修复主题匹配中的中英边界分词问题，消除 `Transformer是什么` 类误拒。
- 让主题匹配判定对单一路径分词异常更稳健，降低误拒概率。
- 明确拒答来源，保证 trace 字段与拒答文案语义一致、可审计。
- 保持库外问题拒答能力与现有关键门控能力不退化。

**非目标：**
- 不重写整条检索链路或替换 rerank 模型。
- 不调整 UI 交互逻辑与页面结构。
- 不引入新的在线依赖或持久化组件。

## 决策

### 决策 1：在主题匹配分词中引入“中英边界切分”与问句后缀切分
- 方案 A（选中）：在现有 token 规则上增加边界切分（字母/数字/中文切换处切分），并保留 stopword 过滤。
- 方案 B：引入完整中文分词库。
- 理由：A 改动小、风险低、可直接覆盖本次故障；B 成本与副作用更高。

### 决策 2：主题匹配判定采用双查询来源稳健计算
- 方案 A（选中）：同时计算 `standalone_query` 与 `query_used(calibrated_query)` 的主题匹配分数并取稳健聚合（如 max）。
- 方案 B：仅维持 `standalone_query` 单路径计算。
- 理由：检索实际使用 `query_used`，双路径可减少文本表面形式带来的误判。

### 决策 3：拒答来源显式化
- 方案 A（选中）：新增拒答来源字段（例如 `final_refuse_source`），并使弱回答模板按来源输出准确文案。
- 方案 B：沿用统一“触发 Sufficiency Gate”模板。
- 理由：A 能避免诊断误导，提升运行日志可解释性。

## 风险 / 权衡

- [风险] 主题匹配放宽后可能让少量噪声问题通过门控。  
  [缓解] 保持 `sufficiency_topic_match_threshold` 默认值，并通过库外问题回归验证拒答能力。

- [风险] 双路径聚合可能掩盖重写漂移。  
  [缓解] trace 同时落盘两个原始分数与聚合分数，保留诊断可见性。

- [风险] 拒答文案分流可能影响现有测试断言。  
  [缓解] 更新回归测试断言为“来源-文案一致性”，避免硬编码单一文本。

## 迁移计划

1. 更新主题匹配分词与评分逻辑，补齐针对中英混写的单元测试。
2. 更新 Sufficiency Gate 判定与 trace 字段，增加双路径分数可观测性。
3. 更新拒答来源字段与弱回答模板分流逻辑。
4. 执行回归：
   - `Transformer是什么` 场景不再因 `topic_mismatch` 误拒；
   - 库外问题集仍为 `refuse/clarify`；
   - 拒答来源字段与文案一致。
5. 若出现回归，回滚到原有单路径判定与统一文案逻辑。

## 开放问题

- 双路径聚合采用 `max` 还是加权平均，需通过回归误拒率对比最终确定。
- 是否需要将“中英边界切分”复用到 claim-support 校验，避免后续出现相同类型误判。

## 新增需求

### 需求:系统必须输出章节树中间表示
系统必须基于 Marker 结构化解析结果生成稳定的章节树中间表示，至少包含标题文本、层级、页码范围、父子关系与关联 chunk 列表。系统禁止在无状态标记的情况下静默丢弃标题层级信息。

#### 场景:显式标题层级被保留为章节树
- **当** PDF 含有可识别的标题层级
- **那么** 系统必须输出章节树中间表示并供后续 structure retrieval 使用

### 需求:结构解析质量必须可观测
系统必须为每篇论文记录结构解析质量状态，至少区分 `ready`、`degraded` 与 `unavailable`，并记录导致降级的原因摘要。当前端或 QA 请求需要结构检索时，系统必须能够读取该状态。

#### 场景:结构质量状态落盘
- **当** 一次 Marker 解析完成
- **那么** ingest 报告必须记录 `structure_parse_status` 与对应原因，且字段可序列化

### 需求:结构解析失败时必须提供兜底信号
当 Marker 无法稳定提取章节层级、标题顺序混乱或 PDF 版式导致结构树不可信时，系统必须输出可消费的兜底信号，供 QA 路由降级回 chunk 检索。系统禁止把不可信结构当作完整目录继续入索引。

#### 场景:双栏或非标准标题 PDF 触发降级
- **当** PDF 因双栏乱序、无显著标题样式或解析异常导致章节树不可信
- **那么** 系统必须将 `structure_parse_status` 置为 `degraded` 或 `unavailable`，并阻止不可信章节树进入结构索引

## 修改需求

### 需求:结构化正文保真输出
系统必须保留 Marker 提供的结构化正文信息，并在入库分块前输出稳定的章节层级、公式文本、表格文本序列与章节树中间表示，禁止在无告警情况下静默丢弃结构信息。

#### 场景:章节层级可追踪
- **当** PDF 含显式标题层级
- **那么** 系统必须在中间表示中保留层级信息并供后续切块策略与章节树索引使用

#### 场景:公式与表格文本不丢失
- **当** PDF 页面包含公式块或表格区域
- **那么** 系统必须将对应文本保留到中间表示，并可在 chunk 文本中追溯

#### 场景:结构信息不可信时显式标记
- **当** 章节层级抽取结果无法形成稳定章节树
- **那么** 系统必须输出结构降级状态与原因，而不是静默省略结构信息

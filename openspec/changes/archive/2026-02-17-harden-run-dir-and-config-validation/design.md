## 上下文

当前 ingest 流程已可稳定产出数据，但在工程稳健性上存在三处已知短板：运行目录命名粒度为分钟导致潜在覆盖、chunk 参数缺少运行时约束校验、配置解析采用简化逻辑导致 YAML 边界场景脆弱。该变更聚焦在不破坏现有接口的前提下提升可追溯性和配置健壮性。

## 目标 / 非目标

**目标：**
- 保证每次运行都写入唯一目录，避免同分钟重复运行互相覆盖。
- 对 `chunk_size` 与 `overlap` 增加运行时校验和回退策略。
- 配置解析升级为标准 YAML 解析流程，提升兼容性与可维护性。
- 保持 `python -m app.ingest --input ... --out ...` 的现有调用方式不变。

**非目标：**
- 不改变 `chunks.jsonl`、`papers.json` 或 run trace 的字段结构。
- 不在本变更中实现检索/重排逻辑本体。
- 不调整现有 capability 的业务范围，仅提升实现质量。

## 决策

1. 运行目录唯一性：时间戳提升到秒级并在冲突时追加序号
- 方案：目录名改为 `YYYYMMDD_HHMMSS`，若目录已存在则追加 `_01`、`_02`。
- 原因：兼顾可读性与唯一性，且易于人工排查。
- 备选：直接使用毫秒/UUID（唯一性更强但可读性较弱）。

2. chunk 参数校验：在配置加载后统一验证
- 方案：新增校验逻辑，约束 `300 <= chunk_size <= 500`，并要求 `0 <= overlap < chunk_size`；若不满足则记录告警并回退到安全默认值（400/50）。
- 原因：避免非法参数把问题带到下游切块阶段。
- 备选：直接抛错终止（更严格但对批处理可用性不友好）。

3. 配置解析升级：使用标准 YAML 解析库
- 方案：引入 `PyYAML`，使用 `safe_load` 解析 `configs/default.yaml`，保留 dataclass 默认值作为缺省回退。
- 原因：标准库能力更完整，减少手写解析边界错误。
- 备选：保留当前自定义解析并持续打补丁（维护成本高）。

4. 向后兼容策略
- 方案：保持配置键名与 CLI 参数不变，仅增强校验与解析路径。
- 原因：最小化变更面，降低回归风险。

## 风险 / 权衡

- [引入新依赖导致环境安装步骤变化] -> 在依赖文档或任务中明确 `PyYAML` 安装，并在缺失时给出清晰错误信息。
- [参数回退掩盖用户配置错误] -> 记录显式警告并在 run report 中体现实际生效值。
- [目录命名策略变化影响外部脚本] -> 保持目录前缀格式稳定，仅增强精度与冲突后缀。

## 迁移计划

1. 更新 `app/runlog.py` 的目录生成逻辑并补充冲突处理。
2. 更新 `app/config.py` 为标准 YAML 解析，并新增参数校验函数。
3. 在 `app/ingest.py` 中接入校验后配置，记录告警与生效参数。
4. 增加最小验证：
   - 同秒多次调用目录唯一性验证
   - 非法 chunk 参数回退验证
   - YAML 解析兼容验证

回滚策略：
- 如发现兼容问题，可回滚到归档前版本；输出文件结构不变，数据层面无需迁移。

## 开放问题

- 回退策略是否还需要支持“严格模式”（遇到非法配置直接失败）。
- 生效配置回写是否需要单独落盘为 `effective_config.json`。

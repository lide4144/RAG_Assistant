## 上下文

当前代码库已经形成三套并行但未统一治理的配置面：

- 静态基线配置：`configs/default.yaml`
- LLM 运行时配置：`configs/llm_runtime_config.json`
- Pipeline 运行时配置：`configs/pipeline_runtime_config.json`
- 强覆盖入口：环境变量

其中，`default.yaml` 主要承载检索、改写、规划、会话、图谱等后端主行为参数，同时也提供部分 LLM 默认值；前端“模型设置”页则通过 `/api/admin/llm-config` 与 `/api/admin/pipeline-config` 维护两类运行时配置，并通过 `/api/admin/runtime-overview` 展示部分生效结果。当前系统已经在局部实现了“default / runtime / env”三层来源语义，但只在 Marker runtime 路径中较为清晰，尚未扩展到整套配置系统。

本变更涉及后端配置加载、运行时配置持久化、运行态概览与前端设置页边界定义，属于跨模块配置治理设计。利益相关者包括：

- 管理员：需要知道哪些设置可以在前端调整，哪些不能
- 开发者：需要知道新增字段应放在哪一层
- 运维人员：需要知道故障时先检查 YAML、runtime 还是 env

## 目标 / 非目标

**目标：**

- 建立统一的配置治理模型，为字段定义明确的 owner 与来源优先级。
- 明确区分“静态基线配置”“运行时可调整配置”“仅环境变量配置”三类责任边界。
- 统一运行态来源可观测语义，使系统能解释字段最终来源与回退路径。
- 收敛前端设置页的职责边界，避免其继续扩张为全量系统配置编辑器。
- 为后续设置页现代化迭代提供稳定约束：保持 `magic-mcp` 工作流兼容性，并保持中文语境与中文注解风格。

**非目标：**

- 不要求把全部配置物理合并为单一文件。
- 不在本设计中把 `default.yaml` 的全部字段都暴露到前端。
- 不在本设计中直接修改应用代码或前端实现。
- 不在本设计中重新设计所有已有 API 载荷，只定义后续应遵循的治理原则与演进方向。

## 决策

### 决策 1：统一的是“治理模型”，不是“单文件存储”

系统将采用统一配置治理模型，而不是追求把所有配置强行塞进 `default.yaml` 或单一 runtime 文件。

原因：

- 当前配置来源分散的根因不是文件数量，而是缺少统一语义。
- 不同配置天然具有不同生命周期：产品基线、部署后调整、宿主机强依赖、敏感信息并不适合共存于同一载体。
- 单文件方案会把运行时调整、敏感配置和算法策略耦合在一起，增加误改和回滚风险。

考虑过的替代方案：

- 方案 A：全部回收到 `default.yaml`
  - 未选用，因为不适合运行时管理，也不适合保存敏感密钥。
- 方案 B：全部前端化，统一存到 runtime store
  - 未选用，因为会把系统策略参数暴露给日常运维，破坏边界。

### 决策 2：为配置字段定义三类 owner

后续所有配置字段必须归入以下三类之一：

- `static`：静态基线配置，由 `default.yaml` 管理
- `runtime`：运行时可调整配置，由前端管理页和对应 runtime 持久化管理
- `env_only`：仅环境或宿主机配置，由环境变量控制

owner 的判断标准如下：

- 属于产品/算法/策略默认行为，且不应被日常运维随意调整的字段，归 `static`
- 属于部署后需要频繁调整且调整成本低、反馈直观的字段，归 `runtime`
- 属于敏感信息、宿主机依赖、路径/网络基础设施绑定项，归 `env_only`

考虑过的替代方案：

- 方案 A：不定义 owner，仅按历史习惯继续追加字段
  - 未选用，因为会继续制造新的配置漂移。
- 方案 B：只区分“前端可配”和“前端不可配”
  - 未选用，因为无法覆盖 env 强覆盖和敏感配置语义。

### 决策 3：统一来源优先级语义，但允许不同字段集有不同可用来源

系统需要统一解释“最终值由谁生效”，但不要求所有字段都支持全部来源。

统一优先级规则定义为：

- 若字段 owner 为 `static`：`default.yaml` -> 代码默认值
- 若字段 owner 为 `runtime`：`env`（仅当该字段允许 env 覆盖时）-> runtime 持久化 -> `default.yaml` 或代码默认值
- 若字段 owner 为 `env_only`：`env` -> 失败时回退到安全默认值或不可用状态

说明：

- 并非所有 runtime 字段都必须支持 env 覆盖，但若支持，必须在规范中明确。
- 运行态概览必须能按字段显示 `default`、`runtime`、`env` 等来源。
- LLM runtime 与 pipeline runtime 不再被视为两个“特殊例外”，而是统一纳入 runtime owner 模型。

考虑过的替代方案：

- 方案 A：所有 runtime 字段一律 `env > runtime > default`
  - 部分可接受，但会让本不需要 env 控制的字段过度复杂化，因此保留“可选 env 覆盖”。
- 方案 B：不同模块各自定义优先级
  - 未选用，因为会再次破坏可预测性。

### 决策 4：前端设置页只负责 runtime owner 字段，不扩张为全量系统配置面

前端“模型设置”页的职责将被明确限制为 runtime owner 字段的可视化管理页，而不是全量系统配置编辑器。

现阶段应继续覆盖：

- 五路 LLM 连接与模型配置
- `sufficiency_judge` 证据充分性判定小模型配置
- `planner` 顶层规划模型配置，但以独立高风险运行时面板暴露，而不是混入普通 stage
- Marker runtime tuning
- Marker LLM runtime 配置
- 与这些 runtime 字段直接相关的生效状态、字段来源与告警摘要

现阶段不应直接纳入前端设置页普通模型卡片的字段包括：

- 检索策略与阈值
- 改写规则与词表
- planner 的步数、论文数和摘要门槛等策略参数
- session、graph 扩展等系统级行为参数
- 宿主机路径、Redis 连接、模型缓存目录等 env-only 项

考虑过的替代方案：

- 方案 A：把 `default.yaml` 逐步全部前端化
  - 未选用，因为会显著提升误操作面和维护成本。
- 方案 B：前端只保留只读概览，完全不允许修改
  - 未选用，因为当前 LLM 和 Marker runtime 已证明存在真实调整需求。

### 决策 5：设置页后续现代化实现保持兼容性与表达风格约束

后续针对设置页的界面重构与现代化改造必须保持对 `magic-mcp` 工作流的兼容性，但不将“是否实际调用某个工具”固化为唯一合规标准；同时继续保持中文语境和面向运维/管理员的可读性。当实现进入代码阶段时，新增或调整的注解应保持中文风格，避免在复杂配置交互处留下全英文、语义弱的说明。

原因：

- 设置页已经是配置治理的主要操作面，后续迭代需要保持一致的现代化 UI 方法论。
- 中文语境与中文注解有助于降低本项目目标使用者的理解和维护成本。

考虑过的替代方案：

- 方案 A：不约束 UI 构建方式
  - 未选用，因为会让设置页后续演进风格继续漂移。
- 方案 B：只约束视觉效果，不约束实现方式
  - 未选用，因为本变更希望把实现约束提前写入治理边界。

## 风险 / 权衡

- [风险] 统一治理后，现有字段需要重新分类，短期内会增加规格梳理成本。
  → 缓解措施：先在 spec 层明确 owner 与边界，再分阶段落地实现。

- [风险] 部分字段可能处在 `static` 与 `runtime` 的模糊地带，团队对 owner 归属存在分歧。
  → 缓解措施：以“是否适合部署后频繁调整、是否影响系统语义稳定”为判断标准，并在 Open Questions 中保留待定项。

- [风险] 若运行态概览扩展为字段级来源展示，前后端契约会变复杂。
  → 缓解措施：优先为关键 runtime 字段提供字段来源，静态大字段分阶段推进。

- [风险] 前端不扩张为全量配置面，可能让部分用户觉得“设置入口不完整”。
  → 缓解措施：在设置页和规范中明确边界，区分“运行时设置”和“系统基线配置”。

- [风险] 若将 `magic-mcp` 写成唯一过程要求，会在外部服务不可用时阻塞实现。
  → 缓解措施：将其收敛为兼容性约束，要求页面结构与实现方式可继续接入 `magic-mcp` 工作流，但不把工具调用本身写成硬门槛。

## Migration Plan

1. 在增量 specs 中先为配置治理模型、LLM runtime 持久化、pipeline runtime 持久化和设置页边界补齐规范。
2. 确认每一类字段的 owner、默认来源与是否允许 env 覆盖。
3. 在后续实现阶段逐步把运行态概览扩展为更稳定的字段来源说明接口。
4. 在前端设置页中同步调整文案与信息结构，使用户能区分“系统基线”和“运行时设置”。
5. 保留现有 runtime 持久化文件结构的兼容读取能力，避免立即破坏历史配置。
6. 若实现落地后出现兼容性问题，允许回滚到当前配置加载行为，但保留已补齐的规格文档作为后续收敛依据。

- 是否需要新增独立 capability 来规范“配置字段注册表”或“字段 owner 清单”，避免 owner 规则只停留在文档层？

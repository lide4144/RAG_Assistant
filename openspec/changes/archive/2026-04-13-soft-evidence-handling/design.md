## 上下文

### 当前系统行为
当前 `sufficiency_gate` 在以下情况下会直接拒绝回答：
- 证据数量 < 2 条
- 证据只有噪声内容（front_matter/reference）
- 检索到的内容与问题主题不匹配

拒绝时返回：
- `decision: "refuse"` 或 `"clarify"`
- 弱回答模板："当前证据不足，请提供论文标题..."

### 问题分析
1. **成本浪费**：LLM 调用已经发生（检索、意图分析、重写），直接拒绝导致这些计算被浪费
2. **用户体验**：用户得不到任何信息，无法判断是系统问题还是自己的问题
3. **过度保守**：1 条高质量证据 + 模型知识可能足以给出有价值的回答

### 约束条件
- 必须保持 RAG 系统的"可追溯"核心特性
- 必须明确区分"知识库证据"和"模型知识"
- 不能破坏现有 API 结构（只添加新字段）
- 前端升级是可选的（后端返回兼容的数据结构）

## 目标 / 非目标

**目标：**
1. 在证据不足时，允许使用模型知识补充回答
2. 通过内联标注明确区分内容来源
3. 向前端返回结构化的置信度提示数据
4. 记录用户偏好，避免重复提示
5. 保持向后兼容（现有前端仍可正常工作）

**非目标：**
1. 完全禁用证据检查（仍然需要至少 1 条证据）
2. 修改前端 UI（只提供数据，渲染由前端决定）
3. 支持用户自定义证据阈值
4. 在回答中显示具体的证据评分

## 决策

### 1. 证据阈值：从 2 条放宽到 1 条
**理由**：
- 1 条高质量证据 + 模型知识足以回答许多开放性问题
- 减少不必要的拒绝，提升用户体验
- 仍然保持"可追溯"特性（至少有 1 个引用）

**替代方案**：
- 保持 2 条（过于保守，不符合目标）
- 完全禁用阈值（风险太高，可能导致纯模型幻觉回答）

### 2. 回答模式：`answer_mode` 字段
**方案**：新增 `answer_mode` 字段区分回答类型

```python
answer_mode: "evidence_only" | "low_confidence_with_model_knowledge"
```

**理由**：
- 向后兼容：现有代码检查 `decision == "answer"` 仍然有效
- 新代码可以通过 `answer_mode` 做更细粒度的处理
- 前端可以根据 `answer_mode` 渲染不同的 UI

**替代方案**：
- 修改 `decision` 字段添加新值（破坏现有代码）
- 使用 `output_warnings` 推断模式（不够明确）

### 3. 内联标注格式
**方案**：使用 `[📄 chunk_id]` 和 `[🤖 模型推测]` 格式

**示例**：
```
深度学习在NLP领域取得了突破性进展[📄 chunk_001]。
Transformer架构可能是未来的主流[🤖 模型推测]。
BERT模型于2018年提出[📄 chunk_002]。
```

**理由**：
- 📄 = 明确可追溯
- 🤖 = 明确来自模型知识
- 格式简洁，不破坏阅读体验
- 便于前端解析（正则匹配）

**替代方案**：
- 纯文本标注（"根据知识库..."）过于冗长
- HTML 标签（`<span class="evidence">`）需要前端解析 HTML
- 脚注格式（`[1]`）与现有引用格式冲突

### 4. 用户行为记忆
**方案**：在 session store 中新增 `user_honesty_preferences` 字段

```python
user_honesty_preferences = {
    "hide_low_confidence_warnings": bool,  # 是否隐藏提示
    "acknowledged_at": str,  # ISO 8601 时间戳
    "acknowledgment_count": int,  # 点击"继续查看"的次数
}
```

**触发逻辑**：
- 首次证据不足：显示提示
- 用户点击"不再提示"：`hide_low_confidence_warnings = true`
- 新会话或 24 小时后：重置提示

**理由**：
- 避免重复打扰用户
- 简单有效，不需要复杂的状态机
- 与现有 session 机制集成

**替代方案**：
- 按用户 ID 持久化（需要用户系统，当前项目没有）
- 每次都不提示（可能让用户忽视证据质量问题）

### 5. 结构化提示数据格式
**方案**：在 trace 中新增 `honesty_disclosure` 字段

```json
{
  "honesty_disclosure": {
    "should_show": true,
    "type": "insufficient_evidence",
    "severity": "warning",
    "title": "回答基于有限证据",
    "message": "当前检索到的证据较少，回答结合了模型训练知识",
    "evidence_stats": {
      "count": 1,
      "coverage": 0.2
    },
    "suggested_actions": [
      {"type": "clarify", "text": "提供更具体的论文线索"},
      {"type": "dismiss", "text": "继续查看回答"}
    ]
  }
}
```

**理由**：
- 结构化数据便于前端灵活渲染
- 支持不同严重级别（info/warning/error）
- 可扩展性强（未来可添加更多类型）

## 风险 / 权衡

**[风险] 模型知识可能产生幻觉** → **缓解措施**：
- 要求内联标注，明确区分来源
- 证据不足时降低回答置信度
- 提示用户"谨慎参考"

**[风险] 用户可能忽视提示** → **缓解措施**：
- 提供"不再提示"选项，避免过度打扰
- 置信度提示使用明显的视觉样式（由前端决定）
- 回答内容中强制包含来源标注

**[风险] 现有前端无法处理新字段** → **缓解措施**：
- 向后兼容：只添加字段，不修改现有字段
- 现有前端忽略新字段即可正常工作
- `answer` 字段格式不变（只是内容多了标注）

**[权衡] 回答质量 vs 拒绝率**
- 接受：更多问题得到回答，但部分回答包含模型推测
- 拒绝：回答更可靠，但用户体验差
- 决策：通过标注和提示，让用户自己判断可信度

## 开放问题

1. **模型标注的准确性**：需要测试 LLM 是否能准确区分"来自证据"和"来自模型知识"
   - 解决：在测试集中验证标注准确率

2. **用户偏好的持久化**：当前方案只在 session 中存储，用户关闭浏览器后重置
   - 接受：项目目前没有用户系统，无法持久化到用户级别

3. **性能影响**：生成内联标注可能增加 LLM 输出长度
   - 监控：添加指标追踪回答长度变化

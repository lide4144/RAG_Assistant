## 为什么

当前 RAG 系统在证据不足时采用"生硬拒绝"策略，直接返回"证据不足，请提供论文标题..."等提示。这种方式存在两个问题：

1. **用户体验差**：用户提出问题后得不到任何实质性回答，感觉系统"不可用"
2. **成本浪费**：系统已经调用了 LLM 进行检索和意图分析，直接拒绝导致这些计算成本被浪费

通过引入"软性证据不足处理"机制，系统可以在证据有限时，结合模型知识给出"诚实的不确定回答"，既充分利用已投入的算力，又通过明确标注帮助用户理解回答的可信度。

## 变更内容

1. **放宽证据阈值**：从最少 2 条证据放宽到 1 条即可
2. **新增回答模式**：当证据不足时，允许使用模型知识补充回答（`low_confidence_with_model_knowledge`）
3. **内联来源标注**：在回答中明确标注内容来源（📄 知识库 / 🤖 模型推测）
4. **结构化置信度提示**：向前端返回结构化的诚实提示数据，供 UI 渲染
5. **用户行为记忆**：记录用户是否选择"不再提示"，避免过度打扰

**BREAKING**: `sufficiency_gate` 的返回值将新增 `answer_mode` 字段，调用方需要处理新的回答模式

## 功能 (Capabilities)

### 新增功能
- `soft-evidence-handling`: 软性证据不足处理机制，包含证据阈值调整、模型知识补充、内联标注、用户偏好记忆等功能

### 修改功能
- `sufficiency-gate`: 修改证据不足时的决策逻辑，从"拒绝"改为"低置信度回答"
- `answer-generation`: 支持在证据不足时结合模型知识生成回答，并添加内联来源标注

## 影响

- **app/sufficiency.py**: 修改证据阈值和决策逻辑
- **app/qa.py**: 修改回答生成流程，支持模型知识补充
- **app/session_state.py**: 新增用户偏好存储字段
- **API Response**: `trace` 中新增 `answer_confidence` 和 `honesty_disclosure` 字段
- **Frontend**: 需要新增置信度提示组件（可选，现有字段不影响旧版前端）

## 上下文

当前代码库已具备 PDF 入库（M1）与 chunk 清洗标注（M1.5）能力，数据侧已有 `chunks_clean.jsonl` 与 `content_type` 标签，可直接支持 M2 的检索与问答基线。缺口在于：尚未形成统一的索引构建、检索融合和 QA CLI 出口，导致无法稳定评估“问题 -> 证据 -> 回答”的最小链路。

## 目标 / 非目标

**目标：**
- 建立 BM25 与向量双索引，统一基于 `clean_text` 建索引。
- 提供 `python -m app.qa --q "..." --mode dense|bm25|hybrid` 最小问答命令。
- 在 hybrid 融合阶段对 `table_list` 候选降权（默认 *0.5），并排除 `watermark`。
- 输出 Answer + Top-5 evidence（`chunk_id + section/page + quote`），其中 quote 来自原始 `text`。
- 产出 `reports/m2_baseline.md` 作为阶段性评估记录。

**非目标：**
- 不引入复杂 reranker/agent 流程（后续里程碑处理）。
- 不引入在线服务化部署或 UI。
- 不做高精度答案质量优化，仅保证链路可运行与可评估。

## 决策

1. 检索数据源与索引过滤
- 决策：统一从 `data/processed/chunks_clean.jsonl` 读取；仅索引 `content_type != "watermark"` 的记录。
- 原因：保证与 M1.5 清洗约束一致，降低噪声并减少索引污染。
- 备选：从 `chunks.jsonl` 建索引；会丢失清洗收益且更易被水印干扰。

2. 索引模块职责拆分
- 决策：`app/index_bm25.py` 与 `app/index_vec.py` 各自构建与加载索引，`app/retrieve.py` 统一编排三种模式。
- 原因：与当前目录约定一致，便于单测与后续替换后端（如 FAISS 参数升级）。
- 备选：单文件混合实现；耦合较高，维护成本大。

3. 融合与降权策略
- 决策：hybrid 对候选执行融合后，针对 `table_list` 应用可配置降权（默认 0.5），但不剔除。
- 原因：符合 M1.5 “可引用但降权”约束，兼顾列表问题召回。
- 备选：直接过滤 `table_list`；会损失“有哪些项”类问答能力。

4. QA 输出与证据来源
- 决策：回答阶段最小化生成，强制输出 Top-5 evidence；quote 从原始 `text` 截取。
- 原因：保证证据可追溯，满足“检索用 clean_text、引用用 text”的统一原则。
- 备选：仅输出答案不附证据；不满足验收与可解释性。

5. 运行可复现与评估记录
- 决策：在 runs 轨迹中记录 query、mode、top-k、最终 evidence；并用 `reports/m2_baseline.md` 记录至少 10 个样例。
- 原因：支持回放、对比与里程碑验收。
- 备选：只保留终端输出；不可审计，不利于后续优化。

## 风险 / 权衡

- [向量模型或依赖缺失导致 dense 不可用] -> 提供清晰降级提示，保底支持 bm25/hybrid（若向量索引存在则启用）。
- [hybrid 分数尺度不一致] -> 先采用简单归一化与权重配置，后续再做更稳健融合。
- [quote 截取不稳定] -> 固定最大长度与边界清洗规则，保证输出可读。
- [30 问题主观评估成本较高] -> 先要求 10 条落盘样例，30 条作为阶段验收清单执行。

## 迁移计划

1. 完成 BM25 与向量索引构建与加载接口。
2. 在 `retrieve` 中打通 dense/bm25/hybrid 与 `table_list` 降权。
3. 新增 `app/qa.py` 命令入口，输出 Answer + Top-5 evidence。
4. 补齐单元与集成测试（模式返回、过滤/降权、输出结构）。
5. 生成 `reports/m2_baseline.md`（>=10 条）并执行 30 问题验收。

回滚策略：
- 若 M2 引入问题，可保留 M1.5 数据与清洗流程，仅停用 `app.qa` 入口与新索引产物，不影响已有入库链路。

## 开放问题

- BM25 选型最终使用轻量本地实现（Whoosh/自研）还是接入 ES（当前建议先本地实现）。
- 向量编码模型与维度是否固定到配置中（建议在 M2 内先固定一个默认模型）。
- `reports/m2_baseline.md` 的样例问题是否需要按能力分类（事实问答/列表问答/比较问答）。

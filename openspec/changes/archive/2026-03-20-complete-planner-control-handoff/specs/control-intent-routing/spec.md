## MODIFIED Requirements

### 需求:系统必须在检索前执行控制意图路由
系统必须将 `control intent（控制意图）` 识别作为 `planner / policy（规划器 / 策略层）` 的正式职责，并在检索前输出结构化结果（至少包含 `intent_type（意图类型）` 与必要参数）；禁止仅依赖关键词/正则列表作为唯一识别手段。对于统一 planner 模式，控制意图不得再误入证据型 QA 主链，也不得由 `sufficiency gate（充分性门控）` 或 `evidence gate（证据门）` 直接裁决其最终交互姿态。

#### 场景:表达变体仍可识别控制意图
- **当** 用户使用非预设措辞表达格式控制意图（如“换成表格展示”）
- **那么** 系统必须识别为控制意图并输出结构化参数

#### 场景:控制意图不进入证据型 QA 主链
- **当** 用户输入“用中文回答我”或等价的样式控制指令
- **那么** 系统必须由 `planner / policy（规划器 / 策略层）` 直接处理该控制意图，而不得将其送入证据型 QA 主链并触发基于证据的拒答

#### 场景:控制意图与主能力共存时保持主能力不丢失
- **当** 用户输入“先列出昨天上传的论文，再用表格展示方法差异”
- **那么** 系统必须保留主能力为跨文档对比，并将“表格”作为结构化参数而不是把整轮请求降格为纯格式控制

### 需求:系统必须提供意图路由可观测字段
系统必须输出意图路由观测字段，至少包含路由来源（`semantic / model / rule_fallback（语义 / 模型 / 规则回退）`）、命中意图、置信度、回退标识、`primary_capability（主能力）`、`planner_confidence（规划置信度）` 与 `interaction_decision_source（交互决策来源）`。当控制意图被识别时，trace 中还必须能够证明其未误入证据型 QA 主链。

#### 场景:控制意图路由可审计
- **当** 一次请求完成控制意图判定
- **那么** trace 中必须包含控制意图路由结果及其最终决策来源，并能证明是否绕开了证据型 QA 主链

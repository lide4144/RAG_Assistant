## 上下文

当前系统的 answer 路径已经支持运行时模型配置持久化与动态生效，但 embedding/rerank 仍主要依赖静态配置与固定环境变量约定。该不对称导致：
- 运维层面：回答模型可热切换，但检索与重排仍受单一 provider/key 约束。
- 稳定性层面：embedding 或 rerank 异常时，降级语义在不同模块分散且不一致。
- 可观测层面：缺少统一的三路依赖健康视图，排障需要跨日志和运行轨迹拼接。

本次设计面向跨模块改造（`config`、`llm_routing`、`retrieve`、`rerank`、`qa`、`kernel_api`），目标是在不破坏现有主流程的前提下统一“配置-路由-降级-健康检查”闭环。

## 目标 / 非目标

**目标：**
- 将 answer、embedding、rerank 统一到一个可扩展的阶段化路由中心（stage-aware routing）。
- 为 embedding 与 rerank 引入与 answer 对齐的动态配置能力（provider/model/api_base/api_key_env）。
- 定义并落地阶段化降级策略：
  - embedding：失败后优先回退词频检索（TF-IDF/BM25），必要时上抛明确异常。
  - rerank：失败后静默穿透，保持上游候选顺序并保证数据结构兼容。
- 新增 `/health/deps`，统一返回 answer/embedding/rerank 三路可用性状态及失败摘要。
- 明确 embedding 维度一致性约束，避免 fallback 向量模型与主索引维度不匹配导致检索崩溃。

**非目标：**
- 不在本变更中替换底层向量索引实现（FAISS/Milvus/Qdrant 适配逻辑保持现状）。
- 不改变现有 RAG 召回、图扩展、证据选择算法本身的排序公式。
- 不引入新的供应商 SDK 作为强依赖（以现有 HTTP/路由抽象为主）。

## 决策

### 决策 1：将 `app/llm_routing.py` 升级为统一 AI 路由中心

- 方案：在同一模块中支持 `stage=answer|embedding|rerank`，输出统一的阶段路由决策对象（provider、endpoint、model、api_key_resolver、fallback_policy）。
- 原因：避免 answer 与 embedding/rerank 各自维护分裂的配置解析与失败处理逻辑。
- 备选方案：
  - 分别新增 `embedding_routing.py` 与 `rerank_routing.py`：短期改动小，但长期策略漂移风险高。
  - 仅在调用处拼接配置：改动分散，难以形成统一可观测语义。

### 决策 2：配置采用“阶段前缀族 + 向后兼容映射”

- 方案：在配置层新增 `embedding_*` 与 `rerank_*` 对齐字段族，并兼容读取现有嵌套块（`embedding.*`、`rerank.*`）进行映射。
- 原因：满足统一配置诉求，同时控制存量配置迁移成本。
- 备选方案：
  - 直接替换旧字段：迁移风险高，易造成现网配置失效。
  - 继续沿用旧字段：无法形成与 answer 一致的运行时管理心智。

### 决策 3：Embedding 降级采用“维度守卫 + 词频回退”双保险

- 方案：
  - 当 `stage=embedding` 调用主/备模型失败，或检测到向量维度与目标索引维度不一致时，禁止进入向量检索分支。
  - 优先降级到 TF-IDF/BM25；若当前运行上下文不允许降级，则抛出结构化不可恢复异常（由上层决定答复策略）。
- 原因：维度不一致是硬错误，继续执行只会在向量库检索时报错并放大故障面。
- 备选方案：
  - 自动重建索引以匹配新维度：代价高、时延不可控，不适合作为在线失败路径。
  - 忽略维度检查直接尝试：不可接受，失败不可预测。

### 决策 4：Rerank 失败采用“静默穿透 + 分数字段补齐”

- 方案：
  - rerank 调用失败（超时/5xx/网络错误）后直接返回上游检索候选顺序。
  - 输出结构中强制补齐兼容字段：至少保证 `score_rerank` 可用（默认复用 `score_retrieval`），并保持下游 `qa.py` 所需契约字段存在。
- 原因：避免下游在依赖 `relevance_score/score_rerank` 时崩溃，同时保持行为可解释。
- 备选方案：
  - 失败即抛错终止：用户体验抖动大，不符合“静默穿透”目标。
  - 赋固定常数分：可行但可解释性弱，优先复用检索分。

### 决策 5：新增 `/health/deps` 汇总三路状态

- 方案：在内核 API 增加依赖健康接口，分别探测 answer/embedding/rerank 的配置完整性、连通性与最近失败摘要，并包含 embedding 维度一致性检查结果。
- 原因：把“是否可用、为何不可用、是否处于降级模式”前置到可观测面。
- 备选方案：
  - 仅依赖 run trace：排障滞后，不能作为启动与发布门禁。

## 风险 / 权衡

- [风险] 阶段化配置与旧配置并存可能引发优先级歧义
  → 缓解：明确优先级顺序（运行时 > 新前缀字段 > 旧嵌套字段 > 默认值），并在日志输出最终生效来源。

- [风险] embedding 频繁降级到词频检索会带来效果波动
  → 缓解：在 trace 与 `/health/deps` 输出降级计数/状态，便于告警和容量治理。

- [风险] rerank 穿透后分数语义改变（`score_rerank == score_retrieval`）
  → 缓解：增加 `rerank_used_fallback=true` 或等价标记，便于下游判断当前分数含义。

- [风险] 健康检查探测本身引入额外外部调用开销
  → 缓解：支持轻量模式（配置校验）与深度模式（真实探测）分离，默认轻量。

## 迁移计划

1. 引入统一路由抽象与阶段枚举，但保持现有调用路径可运行。
2. 配置层加入新字段及映射逻辑，输出兼容告警，不立即移除旧字段。
3. 接入 embedding 维度守卫与降级策略；在检索链路打通异常语义。
4. 接入 rerank 静默穿透与分数字段补齐。
5. 增加 `/health/deps` 并补充测试。
6. 观察期后再评估是否逐步弃用旧配置键。

回滚策略：
- 保留旧配置解析路径与原有 rerank/embedding 调用分支开关，出现问题时可快速切回静态行为。

## 开放问题

- `/health/deps` 是否需要区分“配置可用”与“实时连通”两级状态码（如 `degraded`）？
- embedding 维度检查来源优先级：以索引元数据为准，还是以首次在线探测结果为准？
- rerank 穿透时是否需要统一写入 `score_rerank_source` 字段（`retrieval_fallback`）供前端解释？

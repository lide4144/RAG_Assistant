## 上下文

当前系统在 UI 已有 Library 工作区，但其职责偏“导入入口”，不是“目录管理入口”。同时，导入编排通过临时目录驱动 `ingest`，而 `paper_id` 由文件绝对路径哈希生成，导致 UI 重复导入同一文件时标识不稳定。入库产物目前为覆盖写入，天然不满足增量预期。

约束：
- 必须保持现有 QA 与 topic scope 机制可用。
- 必须保证导入/索引互斥锁语义不退化。
- 先优先稳定性，再追求索引层极致增量性能。

## 目标 / 非目标

**目标：**
- 提供可持续维护的论文目录管理能力，而非一次性导入清单。
- 将 Web 导入语义改为增量追加，默认保留历史语料。
- 建立稳定 `paper_id` 策略，降低重复导入与映射漂移风险。
- 明确导入结果分类：新增、已存在跳过、冲突需确认、失败。

**非目标：**
- 本次不实现复杂权限系统与多租户目录。
- 本次不强制实现 BM25/向量索引的物理级 append-only 真增量。
- 本次不重构 QA 检索策略与门控算法。

## 决策

### 决策 1：采用“目录主数据 + 增量合并”而非全量重建
- 方案 A（采纳）：导入后将新论文与既有论文进行合并写回，保留历史记录。
- 方案 B（未采纳）：继续全量覆盖，要求用户手工维护完整输入集。
- 原因：B 与 Web 导入心智冲突，且易造成历史语料丢失。

### 决策 2：稳定标识不再依赖临时路径
- 方案 A（采纳）：对 PDF 采用内容指纹（或规范化源 URI）生成稳定 `paper_id`。
- 方案 B（未采纳）：继续使用绝对路径哈希。
- 原因：UI 临时目录会变化，B 导致同内容多 ID。

### 决策 3：索引更新采用“两阶段策略”
- 方案 A（采纳）：阶段一“数据增量 + 索引重建”；阶段二再扩展真增量索引。
- 方案 B（未采纳）：一次性交付真增量索引。
- 原因：A 实施风险更低、可快速兑现用户价值。

### 决策 4：管理页聚焦目录操作，不混入底层调参
- 方案 A（采纳）：管理页提供列表、筛选、专题操作、导入状态与冲突处理。
- 方案 B（未采纳）：在同页暴露全部底层管线开关。
- 原因：目标用户关注“收录与维护”，不是系统调参。

## 风险 / 权衡

- [风险] 稳定 `paper_id` 切换后，旧 topic 映射可能失联。
  -> 缓解：提供兼容映射层或迁移脚本，过渡期同时识别旧/新 ID。

- [风险] 增量合并后，索引重建耗时上升。
  -> 缓解：先在 UI 给出明确进度与状态；后续引入真增量索引。

- [风险] 重复识别策略过严导致误判“冲突”。
  -> 缓解：导入结果分级展示，允许用户二次确认。

- [风险] 目录字段扩展导致测试基线变化。
  -> 缓解：补充回归测试覆盖导入分类、目录列表、topic 关联。

## 迁移计划

1. 定义并落地稳定 `paper_id` 生成规则与兼容读取逻辑。
2. 将入库写入改为“读取现有 -> 合并 -> 原子写回”。
3. 升级 Library 为目录管理页，新增列表筛选与状态呈现。
4. 升级导入反馈结构（新增/跳过/冲突/失败）。
5. 完成回归测试与手工验收后默认启用。

回滚策略：
- 保留开关切回旧导入路径（覆盖式）与旧 ID 读取逻辑，确保故障可快速降级。

## 开放问题

- 稳定 `paper_id` 的首选来源是否固定为“文件内容哈希”，还是“source_uri + 内容哈希”混合策略？
- 首版是否需要支持“下线论文（软删除）”并同步影响专题范围？
- 真增量索引阶段是否需要引入后台任务队列以避免 UI 阻塞？

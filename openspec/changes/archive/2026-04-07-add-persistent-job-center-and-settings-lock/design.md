## 上下文

当前项目已经有三类接近但未统一的运行态模型：

- 聊天问答使用页面级 `WebSocket + React state` 展示增量内容，组件卸载即失去当前连接与内存态。
- 图构建与论文导入已经具备 `taskId + 状态轮询/事件` 的后台任务雏形，但任务模型仍偏向非聊天长任务。
- 模型设置页可以直接修改运行态配置，但没有统一的“活动任务占用锁”，导致运行中的聊天/规划任务可能与后续保存的配置发生漂移。

这次变更跨越前端、Gateway、Kernel/API 与持久化层，属于新的架构模式引入：从“页面拥有执行态”转为“后台任务拥有执行态，页面只负责订阅与恢复”。受影响的利益相关方包括：

- 聊天用户：希望切换页面、刷新或短暂断网后仍能看到同一回答继续生成。
- 管理员：希望在任务运行中查看设置，但不希望误改配置影响正在执行的任务。
- 开发与排障人员：希望流式事件、任务状态、配置版本与调试信息都能对齐到同一任务标识上。

约束如下：

- 现有前端路由为独立页面组件，`/chat`、`/pipeline`、`/settings` 之间切换会卸载各自页面状态。
- 现有 Gateway 已统一分发聊天流式事件，适合继续承载任务订阅与恢复事件。
- 现有 Kernel 已有 `task` 基础设施与 `runId/traceId` 概念，可以扩展而不必新建完全独立的执行栈。
- 当前项目更适合先使用本地持久化（SQLite 或等价轻量存储）实现任务与事件恢复，再视需要升级到 Redis/Postgres。

## 目标 / 非目标

**目标：**

- 为聊天问答、规划问答、图构建、论文导入建立统一后台任务中心，并以稳定 `jobId` 作为主标识。
- 为长任务提供可顺序回放的事件日志，使页面切换、刷新和短暂断连后能够恢复同一任务。
- 为任务启动时冻结有效配置版本，保证任务执行期间的模型位点、地址与凭据来源不漂移。
- 为模型设置页与保存接口引入活动任务占用锁，实现“可查看、不可修改”的一致语义。
- 让前端在 `AppShell` 级持有任务中心状态，而不是把运行态局限在单页组件内部。

**非目标：**

- 不在本次设计中重做完整的身份认证或多租户隔离体系。
- 不要求第一阶段就引入 Redis、Postgres 或分布式队列；单进程/单机可恢复实现即可满足当前目标。
- 不把所有历史 run trace 全量迁移成任务事件日志；本次只要求新任务链路具备统一任务记录与事件回放。
- 不在本次设计中定义新的前端视觉风格，只定义状态模型、接口与交互约束。

## 决策

### 决策 1：以统一 `job` / `job_event` 模型承载所有长任务

选择：

- 新增统一后台任务实体 `job`
- 新增顺序事件实体 `job_event`
- 聊天、规划、图构建、导入都映射到该模型

核心字段建议：

- `job_id`
- `kind`：`chat_answer` / `planner_chat` / `graph_build` / `library_import`
- `state`
- `session_id`
- `trace_id`
- `run_id`
- `config_version_id`
- `latest_output_text`
- `progress_stage`
- `updated_at`

事件字段建议：

- `job_id`
- `seq`
- `type`
- `payload`
- `created_at`

原因：

- 统一模型才能让聊天与现有任务共用“恢复、取消、占用锁、排障”的能力。
- `job_event` 天然适合表达流式增量、来源集合、错误与完成事件，比把所有内容压到单个状态对象里更易恢复。

备选方案：

- 保持聊天继续使用页面级 WebSocket，仅把图构建/导入作为后台任务。
  - 否决原因：无法解决聊天切页/刷新丢失状态，也无法形成统一设置锁。
- 仅保存最终结果，不保存事件。
  - 否决原因：无法恢复中间流式内容，页面返回后只能看到最终答案，不能满足“继续看同一任务”的目标。

### 决策 2：实时事件继续由 Gateway 分发，但事件必须先落库

选择：

- Kernel 先把 `job_event` 持久化，再经 Gateway 转发给当前订阅客户端
- 前端重新连接时先拉历史事件，再接续实时事件

恢复流程：

1. 前端持有 `jobId` 与最近已消费 `seq`
2. 页面恢复时调用 `GET /api/jobs/{jobId}/events?after_seq=<lastSeq>`
3. Gateway/前端补齐缺失事件
4. 再继续消费新的实时事件

原因：

- 只做实时推送会在切页或刷新后丢失事件。
- 先落库再分发，可以保证恢复与调试共用同一事实来源。

备选方案：

- 仅靠 WebSocket 保持单连接长驻于 `AppShell`
  - 否决原因：只能覆盖切页，不能覆盖刷新、断网、浏览器重开；同时仍缺历史回放。
- 仅靠轮询任务状态，不保留事件流
  - 否决原因：无法恢复细粒度流式回答与来源/调试分段。

### 决策 3：配置保存采用“活动任务锁 + 配置冻结”双保险

选择：

- 任务启动时解析并记录 `config_version_id`
- 设置页保存时检查是否存在活动任务
- 有活动任务时：
  - 前端设置页进入只读
  - 后端保存接口返回 `409 SETTINGS_LOCKED_BY_ACTIVE_JOB`

原因：

- 只有前端禁用按钮不可靠，接口仍可能被直接调用。
- 只有后端拒绝保存也不够，用户会困惑为什么页面看起来能改但保存失败。
- 配置冻结保证“正在运行的任务”即使之后允许新版本保存，也不会漂移到新配置。

备选方案：

- 只做前端只读锁
  - 否决原因：安全性和一致性不足。
- 允许任务运行中保存配置，并让任务动态读取最新配置
  - 否决原因：会导致回答链路中途切换模型或地址，结果不可复现。

### 决策 4：前端引入 `TaskCenterProvider`，页面变为任务消费者

选择：

- 在 `AppShell` 或 `RootLayout` 层引入全局任务中心
- `ChatShell`、`PipelineShell`、`SettingsShell` 只消费任务中心状态
- 任务中心负责：
  - 记录活动任务
  - 保存 `jobId -> lastSeq`
  - 恢复任务
  - 对设置页提供活动锁状态

原因：

- 现有页面组件切换会卸载本页状态，必须把活动任务状态上移到跨页面生命周期的位置。
- `AppShell` 已是全站公共壳层，最适合承载任务中心。

备选方案：

- 每个页面各自维护自己的任务与恢复逻辑
  - 否决原因：会重复实现恢复、锁判断与错误展示，且容易状态不一致。

### 决策 5：第一阶段使用轻量本地存储实现任务恢复

选择：

- 第一阶段使用 SQLite 或等价轻量本地数据库持久化 `job`、`job_event`、`config_version`
- 保持当前单机部署模型不变

原因：

- 现有项目本地开发与单机部署为主，SQLite 已足以支撑任务恢复与事件回放。
- 这能最小化运维成本，同时为后续升级到 Redis/Postgres 保留迁移空间。

备选方案：

- 直接上 Redis + Postgres
  - 否决原因：对当前项目阶段过重，且会在需求尚未收敛前引入额外运维复杂度。
- 继续仅使用进程内内存
  - 否决原因：刷新/重启后不可恢复，无法满足完整版目标。

## 风险 / 权衡

- [任务与事件持久化增加实现复杂度] → 通过复用现有 `task` 基础设施和 `runId/traceId`，先支持最小必要字段，避免一次性大拆。
- [聊天流式改造成后台任务后，链路会更长] → 保持 Gateway 事件格式兼容，在前端先做“恢复增强”而不是完全重写 UI。
- [活动任务锁可能影响管理员快速调参] → 通过明确展示锁来源、任务摘要和结束后可保存提示，减少“为什么不能改”的困惑。
- [配置冻结会引入版本管理成本] → 第一阶段只保存任务启动时的有效配置快照或版本号，不急于实现复杂的版本回滚界面。
- [事件日志体量增长] → 对 `message_delta` 采用合理分片，并为历史事件保留清理策略，例如按时间或任务完成状态归档/裁剪。
- [前后端过渡期间新旧任务模型并存] → 采用阶段化迁移，先让聊天/规划并入统一任务模型，再逐步收敛旧接口。

## Migration Plan

1. 数据层准备
- 增加 `job`、`job_event`、`config_version`（或等价配置快照）持久化结构。
- 为现有图构建/导入任务补齐统一字段映射。

2. Kernel/API 扩展
- 新增聊天/规划任务创建接口与任务查询/事件查询接口。
- 任务启动时固化有效配置并记录 `config_version_id`。
- 配置保存接口增加活动任务占用检查，返回结构化拒绝错误。

3. Gateway 扩展
- 将实时聊天流式事件映射到统一 `job_event` 模型。
- 支持按 `jobId` 恢复与继续订阅。

4. 前端任务中心落地
- 在 `AppShell` 层引入 `TaskCenterProvider`。
- `ChatShell` 改为发起并恢复 `chat_answer` / `planner_chat` 任务。
- `PipelineShell` 接入同一任务中心，保持现有任务 UI 能力。

5. 设置锁落地
- `SettingsShell` 读取活动任务锁状态，进入只读模式。
- 保存失败时展示 `SETTINGS_LOCKED_BY_ACTIVE_JOB` 等结构化原因。

6. 调试与可观测性收敛
- 让任务、运行日志、模型调试记录在 `jobId/traceId/runId` 上对齐。
- 确保任务恢复后仍能查看调试上下文。

回滚策略：

- 保留现有非任务化聊天接口作为短期兜底开关。
- 若新任务模型在生产前验证中不稳定，可先关闭“聊天任务恢复”能力，仅保留设置锁与任务记录。

## Open Questions

- 第一阶段配置冻结是保存“版本号引用”还是直接保存“完整配置快照”更合适；这取决于当前运行时配置结构是否稳定。
- 活动任务锁是否需要区分“严格锁全部设置”与“仅锁会影响运行态的字段”；当前设计默认严格锁。
- 任务事件清理策略是否按时间、任务数量还是任务状态做裁剪，需要结合实际运行量再定。
- 是否需要在第一阶段就把 LLM 调试记录完全并入 `job_event`，还是先维持独立调试接口再逐步合并。

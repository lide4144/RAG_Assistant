## 新增需求

### 需求:流式回答必须支持按任务恢复
系统必须将流式回答与稳定任务标识绑定，并允许前端在页面切换、刷新或短暂断连后按该任务标识恢复后续流式内容。禁止把流式回答仅实现为“当前 WebSocket 连接上的一次性输出”而没有恢复语义。

#### 场景:页面切换后继续查看同一回答
- **当** 用户在流式回答尚未结束时切换到其他页面后返回聊天页
- **那么** 前端必须能够继续显示同一 `jobId` 的既有回答内容并接收后续增量，而不是把这次回答显示为中断或消失

## 修改需求

### 需求:前端必须消费并展示回答增量流
当回答流式模式开启时，系统必须在 UI 侧持续渲染增量内容，禁止等待完整回答后一次性展示。前端还必须能够在页面切换、刷新或连接重建后恢复同一回答任务的增量内容，并保持同一条 assistant 消息的连续展示。

#### 场景:UI 实时展示流式内容
- **当** 回答进入流式生成路径
- **那么** 用户必须在最终完成前看到持续更新的文本输出

#### 场景:重连后恢复同一条消息
- **当** 流式回答已经输出部分内容且前端重新建立连接
- **那么** 系统必须让前端恢复已有增量并继续追加后续内容，而不是重新创建另一条重复消息

### 需求:流式链路中断必须降级并保持上下文连续
系统必须在流式连接中断、超时或空流时返回可恢复结果并保留当前会话上下文，禁止清空或打断对话链。若任务本身仍在后台继续执行，系统必须允许客户端稍后按任务标识恢复；若任务已经失败，则必须返回可审计失败状态与可继续提问入口。

#### 场景:流式中断后恢复交互
- **当** 回答流式连接在完成前异常中断
- **那么** 系统必须输出降级提示与可继续提问入口，且历史消息保持可用

#### 场景:后台任务仍在运行时恢复
- **当** 客户端连接中断但后台回答任务尚未结束
- **那么** 系统必须允许前端稍后继续订阅该任务并补齐中断期间的回答增量

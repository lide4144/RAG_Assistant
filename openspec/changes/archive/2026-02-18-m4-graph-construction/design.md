## 上下文

当前仓库已经完成 M1.5 的 chunk 清洗与标注，并在 M2/M2.2/M2.3 建立了检索与输出治理链路，但尚未提供结构化图数据来支持“相邻上下文扩展”和“实体链路扩展”。M4 的目标是先完成独立可用的图构建与查询能力，不直接改造主检索打分路径。

输入以 `data/processed/chunks_clean.jsonl` 为准，且需要遵循现有内容类型治理：`watermark` 不可用、`front_matter` 默认不参与核心检索。输出需可落盘、可重载、可查询，作为后续 Milestone 扩展基础。

## 目标 / 非目标

**目标：**
- 构建统一图产物，覆盖两类边：adjacent（结构邻接）与 entity（实体共现）。
- 提供稳定查询接口：`neighbors(chunk_id, type, hop)`，满足后续扩展调用。
- 在输入缺失 `entities` 时提供规则抽取兜底，保证实体图可构建。
- 控制图规模，避免实体边爆炸（`top_m` 截断）。

**非目标：**
- 不在本阶段接入在线向量索引或外部图数据库。
- 不引入重型 NER 模型，不增加外部服务依赖。
- 不在本阶段改写 M2 主检索排序逻辑，只产出图与查询 API。

## 决策

### 决策 1：图存储使用“轻量 JSON + 内存索引”
- 方案：默认输出 `graph.json`（节点元信息 + 邻接表），加载后构建内存查询索引；保留 `graph.pkl` 作为可选兼容。
- 原因：可读、可调试、易与现有数据目录协作；不增加部署依赖。
- 备选：
  - 仅 pkl：读写快但可观测性弱，不利于验收排障。
  - 引入 NetworkX/GDB：能力更强但超出当前里程碑需要。

### 决策 2：节点过滤前置，统一应用于两类边
- 方案：先过滤出 eligible 节点（`suppressed=false`、非 `watermark`、默认非 `front_matter`），再分别构建 adjacent/entity 边。
- 原因：避免不同边构建路径出现规则不一致，简化验证。
- 备选：边构建时各自过滤，易产生口径漂移。

### 决策 3：adjacent 边按论文内稳定序建立
- 方案：同一 `paper_id` 内按 `(page_start, chunk_id)` 排序；若 `section` 存在，仅在同 section 邻接，否则按页序邻接。
- 原因：对 PDF 噪声更稳健；兼容无 section 数据。
- 备选：仅按 chunk_id 排序，跨页/跨段误连风险更高。

### 决策 4：entity 边采用“共享实体计数 + top_m 截断”
- 方案：同一论文内，共享实体数 >= `entity_overlap_threshold` 建边；每节点仅保留前 `entity_top_m` 个邻居。
- 原因：在保证语义连通性的同时控制图密度。
- 备选：不截断会在高频实体场景下导致爆炸。

### 决策 5：实体抽取采用规则兜底
- 方案：当输入无 entities 时，从 `clean_text` 抽取三类模式：
  - 全大写缩写（2~10）
  - 连字符数字词（如 `GUESS-18`）
  - CamelCase
- 原因：零依赖、与当前里程碑一致、可快速迭代。
- 备选：统计词典或模型 NER，复杂度更高，暂缓到后续。

## 风险 / 权衡

- [实体过稀导致 entity 邻居为空] → 在验收记录中区分“实体稀疏”与“阈值过高”，并暴露阈值配置。
- [实体过密导致噪声边偏多] → 保留 `top_m` 截断并记录每节点截断统计。
- [section 质量不稳定] → 缺失或异常时回退页序邻接，保证图可构建。
- [JSON 文件体积增大] → 邻接表仅存 chunk_id 和轻量权重，不重复存全文。

## 迁移计划

1. 新增/扩展 `app/graph_build.py`：读取清洗数据、过滤节点、构建两类边、落盘图文件。
2. 新增图查询 API（同模块或工具模块），提供 `neighbors` 与可选 `neighbors_with_weight`。
3. 增加配置项（阈值、top_m、front_matter 开关）并给出默认值。
4. 增加单元测试：过滤规则、邻接正确性、实体边上限、图加载查询。
5. 生成 M4 验收记录与运行说明。

回滚策略：若图构建上线后发现质量问题，可关闭图使用路径并回退到仅文本检索；图文件是附加产物，不影响既有索引文件。

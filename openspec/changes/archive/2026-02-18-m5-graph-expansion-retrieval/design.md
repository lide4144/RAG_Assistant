## 上下文

当前 `app/retrieve.py` 仅返回初检 top-k 候选，`app/qa.py` 后续流程（证据组织、答案生成）依赖该集合。M4 已提供 `graph.neighbors(chunk_id, type=adjacent|entity, hop=1)` 能力，但尚未接入召回阶段，导致多跳和跨段对比问题容易缺失关键上下文。M5 需要在不破坏现有检索模式（dense/bm25/hybrid）的前提下，将图扩展作为初检后的补充召回层，并对候选规模和噪声类型进行硬约束。

约束：
- 保持现有 CLI 兼容（`python -m app.qa --q ... --mode ...`）。
- 扩展仅做 1-hop，边类型仅限 `adjacent` 与 `entity`。
- 扩展候选必须遵守内容类型过滤与数量上限，避免召回爆炸。

## 目标 / 非目标

**目标：**
- 在初检 top-k 上执行图扩展，补全关键上下文。
- 对 `watermark/front_matter/reference` 执行查询意图驱动的过滤策略。
- 保证扩展后候选规模可控：`<= top_k * (1 + alpha)` 且全局上限 `<= 200`。
- 在运行记录中沉淀扩展统计，支持 M5 人工验收。

**非目标：**
- 不引入 2-hop+ 扩展或图随机游走。
- 不替换现有 BM25/向量/融合打分方式，仅补充候选集合。
- 不在本阶段引入外部图数据库或在线图服务。

## 决策

### 决策 1：采用“两阶段召回”
- 方案：先用现有检索得到 `seed_candidates=top_k`，再以每个 seed 调用图邻居 API 生成扩展候选，合并去重后进入后续排序/证据阶段。
- 原因：最小侵入、可复用 M2/M3 检索逻辑，并直接对齐 M5“初检 top-k 上扩展”的要求。
- 备选：将图信号并入首轮打分；实现复杂且不利于隔离验证。

### 决策 2：扩展候选过滤在“入池前”执行
- 方案：对每个扩展候选先过滤再入池：
  - `watermark` 永久剔除
  - `front_matter` 默认剔除，仅 query 命中作者/机构词时放行
  - `reference` 默认不扩展，仅 query 命中“引用/出处/验证/量表”等词时放行
- 原因：在规模控制之前先去噪，减少无效占位。
- 备选：扩展后统一过滤；会浪费预算并降低有效候选比例。

### 决策 3：预算控制采用“双阈值”
- 方案：扩展上限为 `expand_budget = min(top_k * alpha, global_max - top_k)`，最终总候选 `<= min(top_k*(1+alpha), global_max)`，默认 `alpha=2`、`global_max=200`。
- 原因：既满足相对比例约束，也避免在大 top-k 下超出系统可处理规模。
- 备选：仅使用比例上限；当 top-k 较大时仍可能过量。

### 决策 4：去重键使用 `chunk_id`，并记录来源
- 方案：以 `chunk_id` 去重；在 payload 记录 `retrieval_source`（seed/adjacent/entity）和命中过滤原因统计。
- 原因：保证下游证据结构稳定，并支持验收阶段追溯“补到的上下文来自何类扩展”。
- 备选：按文本去重；成本高且不稳定。

### 决策 5：按“先邻接后实体”分配扩展预算
- 方案：遍历 seed 时优先尝试 `adjacent`，再尝试 `entity`；预算耗尽即停止。
- 原因：邻接边通常提供更直接上下文，先保证局部连续性，再补语义跳转。
- 备选：混合随机或实体优先；可解释性较弱。

## 风险 / 权衡

- [实体边噪声导致候选质量下降] → 保留 `reference/front_matter` 扩展门控并记录过滤统计，必要时下调 `alpha`。
- [扩展过强掩盖初检排序] → seed 候选固定保留在结果中，扩展只补充不替换。
- [不同 query 意图词命中不稳定] → 使用中英双语词表并将词表配置化，允许后续热更新。
- [图中缺失节点导致扩展收益不稳定] → 遇到无邻居时回退为仅 seed 集合，不影响主流程可用性。

## 迁移计划

1. 在 `app/retrieve.py`（或新增 `app/expand.py`）实现图扩展函数：输入 seed candidates + query + graph API + 配置，输出扩展后候选与统计。
2. 在 `app/qa.py` 的检索阶段接入扩展流程，并在 run trace 中写入扩展统计字段。
3. 在 `configs/default.yaml` 增加 `graph_expand_alpha`、`graph_expand_max_candidates`、扩展意图词表等配置。
4. 增加单元测试：过滤规则、预算上限、去重行为、无邻居回退。
5. 补充 M5 验收记录：至少 10 个多跳/对比问题的扩展前后上下文对比。

回滚策略：通过配置关闭图扩展（`graph_expand_alpha=0` 或 feature flag），系统回退到现有初检召回路径。

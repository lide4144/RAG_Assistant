## 上下文

当前系统的论文域已经跨越多个模块：导入流程在 `app/ingest.py` 和 `app/library.py` 中生成 `papers.json`、`chunks.jsonl`、`paper_summary.json`、`structure_index.json` 与索引文件；检索与问答通过 `paper_id` 将 chunk、summary、structure 和 citation 串联；目录与任务界面则通过 `app/kernel_api.py` 聚合最近导入报告与任务状态。

现状存在三个结构性问题：

- 论文主记录并非权威真相源。论文身份、状态与专题归属散落在多个文件中，缺少可约束的一致性边界。
- 单篇论文生命周期没有稳定模型。当前更像“最近一次批任务状态”，而不是“某篇论文当前状态”。
- 向量检索后端直接耦合到文件索引产物，未来如需切换为 Qdrant，将被迫同时改动导入、重建、检索与删除链路。

约束如下：

- 当前项目是本地优先、单机运行的研究助理系统，优先降低部署复杂度。
- 现有文件产物与检索链路已被多个模块依赖，不能一次性重写为全新存储栈。
- 未来希望演进到 SQLite + Qdrant，但当前阶段不希望引入新的长期运行服务作为前置条件。

主要利益相关者包括：

- Library / Pipeline 用户，需要看到单篇论文级的稳定状态与可维护元数据。
- 检索与问答链路，需要稳定拿到 `paper`、`chunk` 与向量后端元信息。
- 后续实现者，需要在不推翻上层语义的前提下把向量后端从文件索引切换到 Qdrant。

## 目标 / 非目标

**目标：**

- 让 SQLite 成为论文域的权威存储，至少覆盖论文主记录、生命周期状态、专题归属、派生产物归属与 chunk 归属。
- 保持原始 PDF/URL 内容与大体积索引产物留在文件系统或派生产物层，避免首阶段迁移成本过高。
- 定义统一向量后端契约，要求文件索引后端与未来 Qdrant 后端共享一致的能力边界。
- 让导入、目录、检索、删除、重建都围绕稳定的 `paper_id` 与数据库状态运行。
- 为现有文件产物提供兼容过渡路径，使系统能够渐进迁移而不是一次性切断。

**非目标：**

- 首阶段不要求立即引入 Qdrant 或任何新的常驻服务。
- 首阶段不要求把所有派生产物都搬入 SQLite，例如 embedding 向量数组、graph 文件全文或原始 PDF 二进制。
- 首阶段不要求一次性重写全部检索算法，只要求先稳定存储边界与向量后端抽象。
- 首阶段不追求多用户、远程协作或分布式一致性能力。

## 决策

### 决策 1：以 SQLite 作为论文域的权威存储

系统将新增论文域 SQLite 模式，用数据库表而不是多个 JSON 文件承接以下事实：

- 论文主记录：`paper_id`、来源、稳定路径、指纹、状态、创建/更新时间
- 论文元数据：标题、作者、年份、DOI、摘要等可维护字段
- 阶段状态：去重、导入、解析、清洗、索引、图构建、删除、重建
- 归属关系：topic、chunk、summary、structure、索引构建版本、图构建版本

选择原因：

- 当前项目已使用 SQLite 维护任务存储，证明其运行模型与部署复杂度适配本项目。
- 论文域天然是关系型问题，核心需求是约束、一致性和可查询性，而不是单纯文档落盘。
- SQLite 能在保持单机低复杂度的同时提供事务、唯一约束和索引。

考虑过的替代方案：

- 继续扩展 JSON/JSONL 文件：实现成本最低，但无法解决一致性与单篇生命周期问题。
- 直接上 PostgreSQL：统一能力更强，但对当前单机项目而言引入了不必要的运维复杂度。

### 决策 2：原始文件与大体积派生产物继续保留在文件系统

系统不会把原始 PDF、URL 原文缓存或首阶段的大体积向量数据直接写入 SQLite。数据库记录这些对象的稳定路径、指纹、版本与状态，而文件系统继续承载原文与可重建产物。

选择原因：

- 原始文件天然适合以文件形式管理，数据库更适合作为其索引和控制面。
- 当前已有 `data/raw/imported`、`data/indexes/*`、`data/processed/*` 等产物布局，可在最小扰动下继续复用。
- 这样可以先把“真相源”问题解决掉，再逐步判断哪些派生产物需要进一步数据库化。

考虑过的替代方案：

- 将 PDF 二进制与向量全部写入 SQLite：一致性强，但首阶段迁移代价过大，且不利于本地文件级调试与兼容已有产物。
- 继续让文件产物担任权威真相源：会延续当前一致性问题。

### 决策 3：定义论文生命周期表，而不是继续复用批任务状态

系统将为单篇论文建立稳定的生命周期状态，而不是仅依赖 `import_latest` 或批处理报告。生命周期至少包含：

- `dedup`
- `import`
- `parse`
- `clean`
- `index`
- `graph_build`
- `ready`
- `failed`
- `deleted`
- `rebuild_pending`

这些状态可以通过专门的阶段状态表表达，也可以由论文表上的汇总状态配合阶段明细表表达。

选择原因：

- 用户真正关心的是“这篇论文处理好了没有”，不是“最近一次导入批次是否结束”。
- 删除、重建、失败重试都需要单篇粒度的稳定状态。

考虑过的替代方案：

- 继续以任务中心为主要状态来源：只能表达任务，不足以表达论文实体的当前状态。

### 决策 4：向量检索通过统一后端契约暴露，而不是让上层直接依赖文件索引格式

系统将抽象一个向量后端接口，至少覆盖以下能力：

- `upsert_chunks(chunks, embeddings, metadata)`
- `delete_papers(paper_ids)`
- `search(query_embedding, filters, top_k)`
- `rebuild(snapshot_or_scope)`
- `describe_backend()`

首个实现为文件向量索引后端，继续写入和读取当前文件索引产物；未来 Qdrant 后端必须满足相同的输入输出语义。

选择原因：

- 后续从文件索引迁移到 Qdrant 的关键不是“换一个存储位置”，而是“上层是否只依赖稳定契约”。
- 导入、删除、重建和检索都需要后端无关的编排接口。

考虑过的替代方案：

- 继续把文件索引逻辑直接嵌在检索实现中：短期简单，但未来切换 Qdrant 会引发横向重构。
- 现在就直接接入 Qdrant：会提前引入运行时依赖，偏离当前阶段“先稳住论文域”的目标。

### 决策 5：文件向量索引作为第一阶段后端，但必须记录后端元信息与版本

即使首阶段仍使用文件向量索引，也必须把“当前使用哪种向量后端、构建于何时、基于哪些 chunk 快照、是否过期”记录在 SQLite 中，作为论文域与索引层之间的桥梁。

选择原因：

- 没有后端元信息，重建、故障诊断和迁移切换都只能靠猜当前索引文件对应什么语料。
- 未来接入 Qdrant 时，可以保持数据库侧的索引状态模型不变，只替换后端实现。

考虑过的替代方案：

- 仅通过文件名或 mtime 判断索引状态：过于脆弱，不能支撑单篇重建或迁移审计。

### 决策 6：保留兼容导出文件，但降级为派生产物

`papers.json`、`library_topics.json`、`paper_summary.json`、`structure_index.json` 等现有文件在过渡期可以继续存在，但其角色从“权威事实来源”降级为：

- 兼容导出
- 调试快照
- 可重建派生产物

上层需要查询论文状态、目录与归属关系时，应优先读取 SQLite。

选择原因：

- 可以降低迁移期间对既有模块的冲击。
- 允许现有测试和调试工具在过渡期继续工作。

考虑过的替代方案：

- 一次性彻底移除旧文件：风险过高，且难以回滚。

## 风险 / 权衡

- [数据库与文件双写过渡期可能出现不一致] → 明确 SQLite 为权威源；旧文件仅由数据库或稳定快照导出生成，禁止并行人工修改两边。
- [首阶段仍使用文件向量索引，单篇增量删除和重建能力有限] → 在向量后端契约中先定义删除与重建语义，文件后端允许以“局部重建或整库重建”实现，后续由 Qdrant 提升体验。
- [迁移过程中历史 `papers.json` 路径和来源信息可能不可信] → 迁移脚本优先使用 `source_uri`、指纹与 `data/raw/imported` 中稳定文件重新建立映射；对无法确认来源的记录标记为需人工修复。
- [数据库模式过早设计过细会拖慢落地] → 首阶段只覆盖论文主记录、生命周期、chunk 归属、topic 归属与索引后端状态，不把所有派生产物一次性细拆。
- [引入抽象层可能增加实现复杂度] → 限制首阶段接口面，只抽象导入、删除、查询、重建和元信息读取五类能力，避免过度泛化。

## 迁移计划

1. 新建 SQLite 模式与访问层，定义论文主记录、阶段状态、topic 归属、chunk 归属、artifact / index 状态。
2. 编写一次性迁移与校验逻辑，从现有 `papers.json`、`library_topics.json`、`chunks*.jsonl`、`paper_summary.json`、`structure_index.json` 中导入可恢复事实，并标记不可信或缺失字段。
3. 修改导入流程，使新导入论文优先写入 SQLite，再生成兼容导出文件与清洗、索引、图谱派生产物。
4. 修改目录查询与状态接口，使 Library 与相关 API 优先读取 SQLite。
5. 引入向量后端抽象，并将现有文件向量索引改造成第一个后端实现。
6. 为删除、重建和失败恢复流程增加基于论文生命周期与后端元信息的编排。
7. 在系统稳定后，逐步将依赖旧文件真相源的逻辑切换为数据库读取，并把旧文件定位为导出或缓存。

回滚策略：

- 若新链路失败，可暂时保留现有文件产物读取逻辑作为只读回退路径。
- SQLite 迁移应设计为追加式或可重建式，避免覆盖原始文件产物。
- 在完全切换前，保持能够通过现有文件产物重建数据库。

## 开放问题

- 作者、DOI、年份、摘要等元数据首阶段是否全部结构化入表，还是先保留部分 JSON 扩展字段？
- 文件向量后端首阶段是否支持真正的单篇增量删除，还是先以“标记过期 + 批量重建”为主？
- `paper_summary` 与 `structure_index` 首阶段应直接数据库化，还是先仅记录其 artifact 状态与快照版本？
- 与现有 `job_store.sqlite3` 的关系如何处理：复用同一数据库文件还是保持论文域数据库独立？

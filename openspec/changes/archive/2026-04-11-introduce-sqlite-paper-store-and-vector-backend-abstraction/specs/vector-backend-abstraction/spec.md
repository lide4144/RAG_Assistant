## 新增需求

### 需求:系统必须提供统一的向量后端契约
系统必须通过统一的向量后端契约执行 chunk embedding 写入、删除、检索、重建与后端元信息查询。系统禁止让导入、检索、删除或重建流程直接依赖某一种具体向量后端的存储格式或 SDK 细节。

#### 场景:导入流程写入向量后端
- **当** 导入流程完成某批 chunk 的 embedding 生成
- **那么** 系统必须通过统一的向量后端接口执行写入，而不是直接假定目标一定是文件索引

#### 场景:检索流程切换向量后端
- **当** 系统从文件向量后端切换到另一种后端实现
- **那么** 上层检索编排必须保持相同的输入输出语义，包括 `paper_id` 过滤、`top_k`、返回 chunk 元数据与错误语义

### 需求:文件向量索引必须作为首个受支持后端接入统一契约
系统必须将现有文件向量索引实现纳入统一向量后端契约，作为首个正式支持的后端。系统禁止为了抽象后端而放弃当前本地文件索引运行模式。

#### 场景:当前环境未启用外部向量服务
- **当** 当前部署未配置任何外部向量服务
- **那么** 系统必须能够选择文件向量后端并继续完成 embedding 写入、检索与重建流程

#### 场景:文件向量后端报告元信息
- **当** 系统查询当前向量后端状态
- **那么** 文件向量后端必须返回后端类型、索引路径、构建时间、适用快照或等价元信息

### 需求:未来新增向量后端必须保持论文级语义一致
系统在未来新增 Qdrant 或其他向量后端时，必须保持论文级删除、按 `paper_id` 过滤检索、重建语义与后端元信息查询语义一致。系统禁止因后端切换而改变论文管理、目录查询或检索结果中对 `paper_id` 的使用方式。

#### 场景:按 paper_id 删除向量记录
- **当** 某篇论文被删除或重建
- **那么** 无论当前使用文件后端还是未来的 Qdrant 后端，系统都必须能够按 `paper_id` 清理对应向量记录

#### 场景:按论文范围过滤检索
- **当** 聊天侧或目录侧限定论文范围进行检索
- **那么** 无论当前使用哪种向量后端，系统都必须支持按论文范围执行过滤并返回符合范围的 chunk 候选

### 需求:系统必须记录向量后端类型与构建状态
系统必须在权威存储中记录当前向量后端类型、版本或配置指纹、最近构建时间、适用的语料快照与过期状态。系统禁止仅通过索引文件名或运行日志判断当前向量后端状态。

#### 场景:索引状态可查询
- **当** 用户或后台任务查询当前向量索引状态
- **那么** 系统必须返回当前后端类型、最近构建时间、适用语料范围与是否需要重建

#### 场景:切换后端时保留审计信息
- **当** 管理员将向量后端从文件索引切换到另一种实现
- **那么** 系统必须记录切换前后的后端类型与新后端初始化状态，供后续审计与回滚使用

## 新增需求

### 需求:系统必须提供 SQLite 论文主记录作为权威真相源
系统必须使用 SQLite 持久化每篇论文的主记录，并将其作为论文身份、生命周期状态与归属关系的权威真相源。系统禁止继续以 `papers.json`、任务报告或其他派生文件作为论文存在性与当前状态的最终判断依据。

#### 场景:根据来源与指纹确认论文是否已存在
- **当** 用户再次导入一篇来源 URI 或内容指纹与现有记录匹配的论文
- **那么** 系统必须从 SQLite 论文主记录中判断该论文已存在，并返回稳定的 `paper_id` 与去重结果

#### 场景:目录与检索查询论文状态
- **当** 目录页、任务页或聊天侧需要读取某篇论文的当前状态
- **那么** 系统必须从 SQLite 论文主记录或其关联状态表读取状态，而不是从派生文件反推

### 需求:系统必须维护单篇论文生命周期状态
系统必须为每篇论文维护可查询的生命周期状态，至少覆盖 `dedup`、`import`、`parse`、`clean`、`index`、`graph_build`、`ready`、`failed`、`deleted` 与 `rebuild_pending`。系统必须能够同时表达汇总状态与阶段级状态明细，禁止仅暴露批任务级笼统状态。

#### 场景:单篇论文导入成功后进入 ready
- **当** 一篇论文完成去重、导入、解析、清洗与索引所需阶段且无阻塞错误
- **那么** 系统必须将该论文标记为 `ready`，并保留各阶段最近更新时间

#### 场景:单篇论文中途失败
- **当** 某篇论文在解析、清洗、索引或图构建阶段失败
- **那么** 系统必须将该论文标记为 `failed` 或相应失败阶段，并保存错误原因且不影响其他论文状态查询

### 需求:系统必须记录论文与派生产物的归属关系
系统必须记录每篇论文与其 chunk、summary、structure、向量索引快照、图构建状态等派生产物之间的归属关系与版本信息。系统禁止在删除、重建或诊断时仅依赖文件名、mtime 或目录扫描猜测产物归属。

#### 场景:查询论文派生产物
- **当** 用户或后台任务需要查看某篇论文有哪些派生产物
- **那么** 系统必须返回该论文关联的产物类型、状态、版本标识与最近构建时间

#### 场景:单篇论文标记为需要重建
- **当** 某篇论文的 chunk、向量索引或图构建产物已过期
- **那么** 系统必须能够仅将该论文及其相关产物标记为 `rebuild_pending`，而不是把整库状态一并置为未知

### 需求:系统必须支持论文级删除与重建编排
系统必须支持以 `paper_id` 为单位执行删除、软删除或重建编排，并确保生命周期状态、归属关系与向量后端状态保持一致。系统禁止在缺少归属确认的情况下直接删除共享派生产物。

#### 场景:删除单篇论文
- **当** 用户删除一篇论文
- **那么** 系统必须更新其生命周期状态为 `deleted` 或等价删除状态，并编排清理该论文关联的 chunk、向量记录与可重建产物归属

#### 场景:重建单篇论文索引
- **当** 用户请求重建一篇论文的索引
- **那么** 系统必须基于该论文的主记录与归属关系重新生成相关派生产物，而不是要求全量重建整个知识库

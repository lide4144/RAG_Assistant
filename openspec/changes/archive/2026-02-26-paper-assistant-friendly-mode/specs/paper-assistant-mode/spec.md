## 新增需求

### 需求:系统必须提供论文助理默认模式
当用户提出开放式论文知识库问题（如“总结方向”“下一步问什么”）时，系统必须优先输出可追溯的主题化总结与后续提问建议，禁止直接返回空泛拒答文案。

#### 场景:自然首问返回主题地图
- **当** 用户首轮输入“帮我总结这个知识库里最值得关注的方向”
- **那么** 系统必须返回 3~5 个主题方向，并为每个方向给出至少 1 条可追溯证据引用

#### 场景:返回可执行追问建议
- **当** 系统完成主题化总结
- **那么** 系统必须额外返回 1~3 个建议追问问题，且每个问题必须与已给主题相关

### 需求:系统必须在证据不足时执行最小澄清
当当前证据不足以支撑完整回答但仍可确定澄清方向时，系统必须输出最小澄清问题（1 条），禁止一次输出多条无优先级澄清清单。

#### 场景:可补齐缺口时仅问一条澄清
- **当** 系统识别缺失关键约束且可以通过单一问题补齐
- **那么** `decision` 必须为 `clarify`，且 `clarify_questions` 数量必须为 1

#### 场景:无法判断澄清方向时拒答
- **当** 证据与问题主题完全不匹配且无法生成有效澄清路径
- **那么** 系统必须输出 `refuse`，并在 `reason` 中明确主题不匹配原因

### 需求:系统必须限制连续澄清并在上限后给出可用回答
系统必须跟踪当前会话主题下的连续澄清次数；当达到上限（默认 2）时，下一轮不得继续输出 `clarify`，必须输出低置信但可追溯的主题化回答，并明确不确定性。

#### 场景:连续澄清达到上限后强制回答
- **当** 同一主题会话连续两轮 `decision=clarify`
- **那么** 第三轮系统必须输出可追溯主题化回答（含引用），并在结果中标记低置信/待验证，而不是再次澄清

#### 场景:主题切换后重置澄清计数
- **当** 系统判定用户已切换到新主题
- **那么** 连续澄清计数必须重置，允许按新主题重新执行最小澄清策略

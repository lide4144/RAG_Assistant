## 修改需求

### 需求:Sufficiency Gate 决策输出
系统必须在最终回答前基于 `question` 与 `top_n evidence` 产出单一决策对象，字段至少包含 `decision` 与 `reason`；其中 `decision` 必须为 `answer`、`refuse`、`clarify` 三者之一。当 `decision=clarify` 时，系统必须额外输出 `clarify_questions`，且数量必须为 1 个。系统必须输出可供 UI 审查的告警字段（含 `output_warnings` 或等价结构），并确保 `reason` 与告警字段可序列化且可直接展示。

#### 场景:输出结构合法
- **当** 系统收到一次问答请求并完成证据聚合
- **那么** 系统必须返回包含 `decision` 与 `reason` 的决策对象，且 `decision` 取值仅限 `answer/refuse/clarify`

#### 场景:clarify 输出问题数量固定为 1
- **当** Sufficiency Gate 判定为 `clarify`
- **那么** 输出必须包含 `clarify_questions`，且问题数量必须恰好为 1

#### 场景:连续澄清超限时禁止再次澄清
- **当** 会话级连续澄清计数达到上限（默认 2）
- **那么** Gate 必须禁止输出 `clarify`，并改为输出低置信可追溯回答路径（或不可澄清风险拒答）

### 需求:主题不匹配必须触发不足
当 `top_n evidence` 与问题主题相关性低于门限时，系统必须判定为证据不足，且禁止直接进入 `answer` 决策。对于可通过单一澄清补齐主题约束的场景，系统必须优先输出 `clarify` 而非 `refuse`；仅当无法形成有效澄清路径时才允许 `refuse`。对于控制意图轮次，系统必须使用锚定主题查询（`anchor_query` 或等价字段）作为主题匹配输入来源，而非直接使用控制语句本身。

#### 场景:相关性低但可澄清时优先澄清
- **当** 证据主题与问题主题不匹配且系统可以识别单一缺口
- **那么** 系统必须输出 `clarify`，并在 `reason` 中说明缺失主题约束

#### 场景:相关性低且不可澄清时拒答
- **当** 证据主题与问题主题不匹配且无法确定有效澄清问题
- **那么** 系统必须输出 `refuse`，并在 `reason` 中说明“主题不匹配/相关性不足”

#### 场景:助理模式下不可连续两轮模板化拒答
- **当** `assistant_mode_used=true` 且存在可引用证据
- **那么** 系统不得在相邻两轮都输出模板化拒答，至少一轮必须进入可追溯回答或最小澄清

### 需求:关键要素缺失必须触发不足
当问题要求关键要素（如数值、方法步骤、实验条件、时间/主体限定）而证据未覆盖时，系统必须判定为证据不足，且禁止直接进入 `answer` 决策。对开放式总结问题，系统必须禁止无依据推断“必须包含 numeric”并触发误拒答。

#### 场景:缺失数值或方法细节触发不足
- **当** 问题明确要求数值或方法细节，且 `top_n evidence` 未提供对应要素
- **那么** 系统必须输出 `refuse` 或 `clarify`，并在 `reason` 中指明缺失的关键要素类型

#### 场景:开放式总结不得误触发 numeric 缺口
- **当** 用户输入主题化总结类问题且未请求具体指标
- **那么** 系统不得仅因 numeric 缺失直接拒答，必须先尝试给出可追溯总结或最小澄清

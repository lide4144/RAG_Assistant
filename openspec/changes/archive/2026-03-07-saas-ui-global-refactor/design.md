## 上下文

当前前端在信息架构上仍以功能堆叠为主，聊天、流水线、模型配置的入口与职责边界不清晰，导致操作路径长、页面噪音高、状态反馈不一致。项目现有技术栈已具备 Next.js 前端、Node 网关、Python Kernel 分层能力，适合通过“布局壳层 + 页面职责重分配 + 统一交互规范”完成产品化升级。

本次变更是跨模块重构：将影响全局布局、路由、聊天页、设置页、流水线页、共享组件库和状态反馈机制。关键约束包括：
- 不改变既有后端主流程契约，仅重组前端消费方式。
- UI 需全站汉化并统一术语（"Pipeline" 统一为“知识库构建流水线”）。
- 动效增强必须可控，不能牺牲可读性与操作效率。

## 目标 / 非目标

**目标：**
- 建立稳定的 SaaS 外壳：左侧沉浸式导航 + 右侧主体路由容器。
- 将模型配置完全收敛到设置页，聊天页与流水线页不再承载配置输入。
- 建立流水线状态可视化标准：统计、阶段状态、任务日志三层展示。
- 提升聊天体验：干净主视图、Markdown 渲染、空白态、未配置防呆。
- 统一设计令牌与加载反馈，形成可复用 UI 基线。

**非目标：**
- 不改造后端算法（检索/重排/生成）逻辑。
- 不新增复杂权限系统与多租户隔离。
- 不在本次变更内引入全量暗黑主题体系。

## 决策

### 决策 1：采用“App Shell + 子页面路由”作为一等结构
- 选择：构建全局 Layout（Sidebar + Main），导航项映射到 `/chat`、`/pipeline`、`/settings`，使用客户端路由无刷新切换。
- 理由：将跨页视觉与交互统一放入壳层，减少重复代码并降低后续扩展成本。
- 备选：
  - 保持单页切换面板：实现快但可维护性差。
  - 每页独立布局：灵活但一致性弱、维护成本高。

### 决策 2：模型配置集中化并设置“前端防呆 + 后端状态确认”双重门禁
- 选择：聊天页仅消费“是否已配置”状态；真正配置编辑只在设置页完成。
- 理由：避免聊天主流程被配置项干扰，同时保证用户在错误入口获得明确跳转路径。
- 备选：
  - 每页保留局部配置：学习成本高且状态分裂。

### 决策 3：流水线页面采用“三段式可观测结构”
- 选择：顶部统计（Bento + Number Ticker）+ 中部阶段状态（Pulsating Dot）+ 底部终端日志。
- 理由：同时覆盖宏观进度、阶段健康与排障细节，兼顾业务视图与工程可调试性。
- 备选：
  - 仅保留线性进度条：信息密度不足，难以定位失败环节。

### 决策 4：采用“Shadcn 基底 + Magic 轻动效”与统一设计令牌
- 选择：背景 `slate-50`、卡片 `white`、统一 loading/成功/失败反馈、保留轻量动画组件。
- 理由：保持专业稳重的 SaaS 气质，同时引入适度动效提高状态感知。
- 备选：
  - 完全静态样式：稳定但交互反馈弱。
  - 强动效风格：视觉冲击强但易干扰核心任务。

## 风险 / 权衡

- [风险] 路由切换后状态丢失（如未保存表单、未读日志） → [缓解] 为关键页建立本地状态持久策略（query/state store）并提供离开前提示。
- [风险] 动效过多影响性能或可访问性 → [缓解] 统一动画时长与触发条件，尊重 reduced-motion 设置。
- [风险] 前后端配置字段对齐不完整导致保存失败提示混乱 → [缓解] 定义前端字段映射表与错误码映射字典。
- [风险] 流水线日志更新频繁导致渲染抖动 → [缓解] 采用增量追加与节流渲染策略。

## Migration Plan

1. 引入全局 App Shell 与三页路由骨架，保证旧功能入口可映射到新页面。
2. 迁移模型配置组件到设置页并接通保存/测试反馈；原页面入口改为跳转。
3. 迁移流水线状态区与日志区，完成阶段状态枚举统一。
4. 清理聊天页非对话配置项，接入未配置告警与空白态。
5. 回归验证（无配置、配置成功、流水线执行中/失败、聊天发送中）。

回滚策略：
- 保留旧页面组件分支与路由开关，出现关键缺陷时可切回旧入口。

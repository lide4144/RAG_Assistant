## 新增需求

### 需求:后端模块必须通过数据库访问层读写论文状态
后端各模块（导入、目录、任务、删除/重建）必须通过统一的论文存储访问层与 SQLite 交互，禁止直接操作 `papers.json` 或其他文件产物。系统必须确保所有论文状态查询与更新都经过数据库访问层，以保持数据一致性。

#### 场景:导入流程查询论文是否存在
- **当** 导入流程需要确认某篇论文是否已存在
- **那么** 系统必须通过数据库访问层查询 SQLite 论文主记录，而不是读取 `papers.json`

#### 场景:目录查询获取论文列表
- **当** 目录模块需要获取论文列表
- **那么** 系统必须通过数据库访问层从 SQLite 查询，禁止直接读取 `papers.json` 或拼接文件产物

### 需求:后端必须支持论文生命周期状态转换
后端必须支持单篇论文生命周期状态的完整转换链路，包括从 `dedup` 到 `ready` 的正常流程，以及 `failed`、`deleted`、`rebuild_pending` 等异常或维护状态的设置与查询。

#### 场景:导入流程更新论文阶段状态
- **当** 导入流程完成某阶段的处理
- **那么** 系统必须更新该论文在 SQLite 中的阶段状态与更新时间

#### 场景:删除流程标记论文状态
- **当** 用户删除一篇论文
- **那么** 系统必须将该论文标记为 `deleted` 状态，并编排清理相关派生产物

#### 场景:重建请求设置重建状态
- **当** 用户请求重建某篇论文的索引
- **那么** 系统必须将该论文标记为 `rebuild_pending` 状态，供后台任务调度

### 需求:后端必须维护派生产物归属一致性
后端在创建、更新或删除论文的派生产物（chunk、summary、structure、向量索引）时，必须同步更新 SQLite 中的产物归属记录。系统禁止在文件系统操作与数据库归属记录之间出现不一致。

#### 场景:索引构建后记录产物归属
- **当** 某篇论文的向量索引构建完成
- **那么** 系统必须在 SQLite 中记录索引产物状态、版本与构建时间

#### 场景:删除论文时清理产物归属
- **当** 论文被删除或重建
- **那么** 系统必须更新相关产物归属记录，并编排实际的文件清理或标记过期

## 修改需求

### 需求:入库流程必须写入 SQLite 论文主记录
入库流程必须在生成或确认 `paper_id` 后，将论文主记录、稳定来源信息、指纹与生命周期初始状态写入 SQLite。系统禁止仅通过 `papers.json` 或临时导入目录中的文件路径保存论文主记录。

#### 场景:本地文件导入后记录稳定来源
- **当** 一篇本地文件导入成功进入论文主记录
- **那么** 系统必须为其写入稳定的来源路径或来源 URI，并禁止把临时目录路径作为长期主记录路径

#### 场景:URL 导入后记录来源
- **当** 一篇 URL 文档导入成功
- **那么** 系统必须在 SQLite 论文主记录中写入稳定的来源 URI、来源类型与去重所需标识

### 需求:入库流程必须更新论文级阶段状态
入库流程必须在去重、解析、清洗、索引与图构建阶段更新单篇论文的阶段状态与错误信息。系统禁止只更新整批任务状态而不回写单篇论文生命周期。

#### 场景:导入中单篇论文解析失败
- **当** 一批导入中的某篇论文在解析阶段失败
- **那么** 系统必须将该论文的解析状态标记为失败并记录原因，同时继续处理其他论文

#### 场景:导入后进入索引阶段
- **当** 某篇论文完成导入与清洗并进入索引构建
- **那么** 系统必须更新该论文的阶段状态为索引中或等价状态，供目录与任务查询复用

### 需求:目录查询必须基于 SQLite 权威论文记录
目录查询、筛选与详情展示必须以 SQLite 中的权威论文记录与生命周期状态为准。系统禁止继续依赖 `papers.json`、导入报告或临时聚合结果作为目录页主数据源。

#### 场景:目录页按状态筛选论文
- **当** 用户在目录页按 `ready`、`failed`、`rebuild_pending` 或等价生命周期状态筛选论文
- **那么** 系统必须直接基于 SQLite 论文记录返回结果

#### 场景:目录页显示稳定来源信息
- **当** 用户查看某篇论文的目录详情
- **那么** 系统必须展示该论文的稳定来源信息、生命周期状态与更新时间，而不是展示临时导入路径

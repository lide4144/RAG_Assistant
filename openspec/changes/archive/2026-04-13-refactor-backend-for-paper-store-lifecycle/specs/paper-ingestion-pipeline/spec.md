## 修改需求

### 需求:入库流程必须写入 SQLite 论文主记录
入库流程必须在生成或确认 `paper_id` 后，将论文主记录、稳定来源信息、指纹与生命周期初始状态写入 SQLite。系统禁止仅通过 `papers.json` 或临时导入目录中的文件路径保存论文主记录。

#### 场景:本地文件导入后记录稳定来源
- **当** 一篇本地文件导入成功进入论文主记录
- **那么** 系统必须为其写入稳定的来源路径或来源 URI，并禁止把临时目录路径作为长期主记录路径

#### 场景:URL 导入后记录来源
- **当** 一篇 URL 文档导入成功
- **那么** 系统必须在 SQLite 论文主记录中写入稳定的来源 URI、来源类型与去重所需标识

### 需求:入库流程必须更新论文级阶段状态
入库流程必须在去重、解析、清洗、索引与图构建阶段更新单篇论文的阶段状态与错误信息。系统禁止只更新整批任务状态而不回写单篇论文生命周期。

#### 场景:导入中单篇论文解析失败
- **当** 一批导入中的某篇论文在解析阶段失败
- **那么** 系统必须将该论文的解析状态标记为失败并记录原因，同时继续处理其他论文

#### 场景:导入后进入索引阶段
- **当** 某篇论文完成导入与清洗并进入索引构建
- **那么** 系统必须更新该论文的阶段状态为索引中或等价状态，供目录与任务查询复用

### 需求:入库流程必须将兼容文件视为派生产物
入库流程在保留 `papers.json`、`paper_summary.json` 或其他兼容性文件导出时，必须将这些文件视为由 SQLite 论文主记录与关联数据生成的派生产物。系统禁止在数据库与兼容导出文件之间并行维护两个权威真相源。

#### 场景:兼容导出文件重建
- **当** 系统需要重新生成 `papers.json` 或等价兼容文件
- **那么** 系统必须以 SQLite 论文主记录与其关联数据为输入生成，而不是反向以文件覆盖数据库

#### 场景:兼容文件缺失但数据库存在
- **当** 兼容导出文件丢失而 SQLite 论文主记录仍完整存在
- **那么** 系统必须能够重建兼容文件且不丢失论文主记录与状态

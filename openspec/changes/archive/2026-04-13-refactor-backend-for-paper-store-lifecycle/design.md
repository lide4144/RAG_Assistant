## 上下文

SQLite 论文主记录与生命周期状态已在数据库层就绪，但后端各模块（`app/ingest.py`、`app/library.py`、`app/kernel_api.py`）仍依赖文件产物（`papers.json`、`library_topics.json`）作为主要数据源。这导致：

- 论文生命周期状态无法被稳定读写，单篇论文状态查询不可靠
- 删除、重建等操作无法基于数据库状态正确编排
- 文件产物与数据库之间存在双写不一致风险
- 兼容性文件（`papers.json`）仍被当作权威来源，违背架构设计

约束如下：

- 不能一次性重写全部后端逻辑，必须保持现有 API 契约稳定
- 需要支持平滑过渡：旧文件产物在迁移期继续可用，但角色降级为派生产物
- 必须保证导入、目录、任务查询等核心链路的可用性

主要利益相关者包括：

- Library / Pipeline 用户，依赖稳定的论文状态查询
- 聊天/问答链路，依赖目录接口返回的论文元数据
- 后续实现者，需要清晰的后端与存储层边界

## 目标 / 非目标

**目标：**

- 让 `app/ingest.py` 的导入流程优先写入 SQLite，并维护论文生命周期状态
- 让 `app/library.py` 的目录查询从 SQLite 读取，支持按生命周期状态筛选
- 让 `app/kernel_api.py` 的任务与论文状态聚合关联到单篇论文级数据库状态
- 为删除、重建流程提供基于数据库状态的编排能力
- 保持现有 API 响应契约不变，平滑过渡文件产物依赖

**非目标：**

- 不要求一次性移除所有文件产物读取逻辑（允许保留兼容路径）
- 不要求立即重写检索与问答链路的全部实现（只要求稳定存储边界）
- 不要求在首阶段实现真正的单篇向量索引增量删除（允许整库重建）

## 决策

### 决策 1：导入流程通过数据库访问层写入论文主记录

`app/ingest.py` 及相关导入模块将通过新引入的论文存储访问层（paper store access layer）与 SQLite 交互，不再直接操作 `papers.json`。

修改点：

- 去重阶段：查询 SQLite 确认论文是否已存在，而非读取 `papers.json`
- 导入成功后：通过访问层写入论文主记录（`paper_id`、来源、指纹、初始状态）
- 各阶段完成：更新单篇论文的阶段状态（`parse` → `clean` → `index` → `graph_build` → `ready`）
- 兼容导出：`papers.json` 等文件由数据库状态生成，禁止并行维护

选择原因：

- 确保论文身份与状态以数据库为唯一权威源
- 为单篇生命周期管理提供稳定基础

考虑过的替代方案：

- 继续双写文件与数据库：会延续一致性问题

### 决策 2：目录查询通过数据库访问层读取

`app/library.py` 的目录列表、筛选与详情接口将改为从 SQLite 查询，支持按生命周期状态（`ready`、`failed`、`rebuild_pending` 等）筛选。

修改点：

- 列表查询：从 SQLite 论文主记录分页读取，支持状态筛选
- 详情展示：从数据库读取论文元数据、阶段状态与更新时间
- 专题归属：从数据库读取 topic 关联，而非 `library_topics.json`
- 兼容路径：保留从文件读取的降级逻辑，但优先使用数据库

选择原因：

- 让用户看到与数据库一致的论文状态
- 支持按处理状态筛选（如只看失败的论文）

考虑过的替代方案：

- 继续从 `papers.json` 读取：简单但违背架构设计

### 决策 3：任务状态聚合关联到单篇论文级数据库状态

`app/kernel_api.py` 的批任务视图将不再仅返回 `recent_items`，而是能够关联到具体论文的数据库状态。

修改点：

- 任务完成回写：将单篇论文的处理结果（成功/失败/阶段）写入数据库
- 状态聚合：查询某批任务涉及的论文列表及其当前生命周期状态
- 失败诊断：能够按 `paper_id` 查询某篇论文的失败原因与重试状态

选择原因：

- 用户关心的是"这篇论文处理好了没有"，不是"任务是否结束"
- 支持单篇级的失败重试与重建

考虑过的替代方案：

- 继续仅维护任务级状态：无法支持单篇论文管理

### 决策 4：删除与重建流程基于数据库状态编排

删除、重建等维护操作将基于论文生命周期状态执行，并更新相关产物归属记录。

修改点：

- 删除操作：标记论文为 `deleted` 状态，编排清理 chunk、向量索引归属
- 重建操作：标记论文为 `rebuild_pending` 状态，供后台任务调度重建
- 产物清理：实际文件清理或标记过期，同步更新数据库归属记录

选择原因：

- 确保删除/重建操作与数据库状态一致
- 支持可审计的产物生命周期管理

考虑过的替代方案：

- 直接删除文件不更新数据库：会导致归属关系丢失

### 决策 5：保留文件产物的兼容生成路径

`papers.json`、`library_topics.json` 等文件在过渡期继续存在，但由数据库状态生成，角色降级为兼容导出。

修改点：

- 新增导出逻辑：从 SQLite 查询并生成兼容格式的 JSON 文件
- 禁止并行修改：文件仅由数据库导出生成，禁止人工直接修改
- 回滚能力：如数据库链路故障，可暂时回退到文件读取逻辑

选择原因：

- 降低迁移风险，允许渐进切换
- 保持现有测试与调试工具的兼容性

## 风险 / 权衡

- [数据库与文件双写过渡期可能出现不一致] → 明确写入顺序：先数据库后文件；文件仅用于兼容读取，禁止作为写入目标
- [旧代码路径依赖文件产物格式] → 保留文件读取作为降级路径，但标记为 deprecated，逐步替换
- [数据库查询性能可能影响目录响应] → 为常用查询字段（状态、来源、时间）添加索引；保留分页与缓存策略
- [迁移期间新旧数据可能冲突] → 设计一次性迁移脚本，将现有文件产物导入数据库并标记来源；缺失字段标记为需修复

## 迁移计划

1. 实现论文存储访问层，封装 SQLite 读写操作
2. 修改 `app/ingest.py`，使导入流程优先写入数据库，再生成兼容导出文件
3. 修改 `app/library.py`，使目录查询优先从数据库读取，保留文件降级路径
4. 修改 `app/kernel_api.py`，使任务状态与论文生命周期状态关联
5. 实现删除、重建等维护操作的基于数据库状态的编排
6. 验证核心链路（导入→目录→查询）在数据库优先模式下的稳定性
7. 逐步标记文件产物读取逻辑为 deprecated，最终移除

回滚策略：

- 如数据库链路出现故障，可通过配置切换回文件产物读取模式
- 保持文件产物生成逻辑，确保数据库可重建

## 开放问题

- 作者、DOI、年份等元数据在导入时的完整性校验策略
- 大规模导入时的数据库写入性能优化（批量插入 vs 逐条）
- 与现有 `job_store.sqlite3` 的关系：是否复用连接池还是保持独立
- 是否需要为目录查询引入缓存层以减少数据库压力

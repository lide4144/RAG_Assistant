## 上下文

当前仓库已经具备从论文入库到 QA 的完整链路，但主交互形态偏向开发调试：大量参数与工程术语暴露在主路径，普通科研同学需要理解系统内部机制才能稳定使用。目标用户是“非工程背景的普通科研同学”，核心任务是“平时收集论文，灵感阶段快速提问并沉淀可执行研究想法”。

现状约束：
- 必须保留现有证据可追溯、Sufficiency Gate、会话状态与运行审计能力。
- 不能破坏现有 CLI 与评测链路可用性。
- 新体验优先在 UI 与默认行为层落地，高级调试能力继续保留但分层隐藏。

## 目标 / 非目标

**目标：**
- 将用户主路径收敛为 `导入论文 -> 问答 -> 生成灵感卡片 -> 保存复用`。
- 在 UI 中提供稳定的信息架构：`Library / Chat / Ideas`。
- 默认输出面向科研任务，保留引用证据并降低认知负担。
- 建立灵感卡片最小数据模型与生命周期，支持后续追问和沉淀。

**非目标：**
- 本变更不重写底层检索算法与模型策略。
- 本变更不引入多用户权限系统或远程后端服务。
- 本变更不替代现有 `runs/*` 调试产物，仅调整其在默认 UI 的呈现方式。

## 决策

### 决策 1：采用“双层产品面”而非移除调试能力
- 方案 A（采纳）：默认用户层只展示科研任务流，高级设置中保留检索与调试开关。
- 方案 B（未采纳）：完全删除开发者能力。
- 原因：当前系统已有较完整实验链路，删除会削弱迭代能力；分层隐藏可同时满足易用性与研发需求。

### 决策 2：以“灵感卡片”作为核心沉淀对象
- 方案 A（采纳）：每轮回答可一键结构化为卡片，支持编辑与状态流转。
- 方案 B（未采纳）：仅保留聊天记录与 run trace。
- 原因：科研用户需要可复用成果，不是一次性答案；卡片更适合后续比较、筛选、推进实验。

### 决策 3：文案与交互语义改为任务导向
- 方案 A（采纳）：前台使用“导入论文/证据来源/灵感卡片”等术语。
- 方案 B（未采纳）：延续 ingest/index/retrieval 术语。
- 原因：目标用户非工程导向，术语需贴近其任务模型。

### 决策 4：定义可量化的用户体验验收门槛
- 方案 A（采纳）：2 分钟完成“导入 5 篇 -> 提问 -> 保存 1 张卡片”。
- 方案 B（未采纳）：仅追踪技术指标（命中率、延迟）。
- 原因：本次变更核心是产品可用性升级，必须以任务完成率与时间衡量。

## 风险 / 权衡

- [风险] 默认界面隐藏参数后，部分高级用户可能感到控制能力下降。
  -> 缓解：提供“高级设置”入口并保留原参数映射。

- [风险] 卡片自动结构化质量不稳定，导致内容泛化。
  -> 缓解：卡片默认可编辑，保存前允许人工修订。

- [风险] 新增卡片持久化字段可能与现有会话存储产生耦合复杂性。
  -> 缓解：卡片存储独立于 session_store，按最小字段模型先行。

- [风险] 前台体验改造可能影响现有 UI 测试基线。
  -> 缓解：补充以“用户任务闭环”为导向的 UI 集成测试，并保留关键调试视图回归用例。

## 迁移计划

1. 新增 UI 信息架构与导航分区，先接线现有能力，避免一次性替换。
2. 增加灵感卡片模型与存储接口，支持从当前回答生成卡片草稿。
3. 将调试项迁移至高级设置并保持默认关闭（或折叠）。
4. 增加用户任务验收脚本与手工验收清单。
5. 灰度验证后将“用户层”设为默认入口，保留原调试入口回退方案。

回滚策略：
- 保留原 UI 入口与配置开关，若用户流程出现严重回归，可临时切回旧默认布局。

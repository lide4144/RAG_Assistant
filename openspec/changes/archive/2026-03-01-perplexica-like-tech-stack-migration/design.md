## 上下文

当前系统以 Python 为核心，已具备较强的本地论文 RAG 能力（检索、重排、证据门控、流式回答、多轮会话），但产品交互承载于 Streamlit，主要服务开发调试与审查。目标体验对标 Perplexica 时，短板集中在产品壳层、实时通信协议、模式化交互和来源展示一致性。

此外，算法链路当前呈现“规则主导、语义兜底”特征：rewrite 与 intent 校准阶段存在较多硬阈值和词法拼接，容易压制 embedding/reranker 的语义优势；回答生成 prompt 偏保守，导致结果可读性与信息密度不足。GraphRAG 在检索层已参与召回，但 UI 当前仅暴露 `graph_expand` 标签，尚无图结构与路径可视化。

约束：
- 必须保留现有 Python RAG 能力资产，避免一次性重写造成质量回退。
- 必须支持流式回答与引用追溯，不降低现有证据可信度策略。
- 必须提供可回滚的分阶段迁移，避免大爆炸上线。

## 目标 / 非目标

**目标：**
- 构建与 Perplexica 体验接近的产品化前端与实时通信栈。
- 建立 Node 网关编排层，统一前端协议并屏蔽后端异构实现。
- 在 V1 打通 `Local / Web / Hybrid` 三模式与统一引用展示。
- 保持现有 Python RAG 作为智能内核，优先复用而非替换。
- 将算法策略升级为“语义优先、规则护栏”，降低硬规则对语义检索的破坏。
- 补齐 GraphRAG 在 UI 的可视化表达与可解释路径展示。

**非目标：**
- 本次不一次性交付完整 Perplexica 六类 focus modes。
- 本次不迁移全部历史调试页面到新前端。
- 本次不重写 Python 检索/门控核心算法。
- 本次不引入重量级图数据库或在线图计算引擎（先基于现有图产物实现可视化）。

## 决策

### 决策 1：采用“Next.js 前端 + Node 网关 + Python 内核”三层架构
- 方案 A（采纳）：前端与网关使用 JS/TS 技术栈，RAG 内核保留 Python。
- 方案 B（未采纳）：继续单体 Streamlit 叠加功能。
- 方案 C（未采纳）：全量迁移到 Node/TS 重写 RAG。
- 原因：A 在体验升级与资产复用之间平衡最佳，实施风险可控。

### 决策 2：前后端通信统一为 WebSocket 事件协议
- 方案 A（采纳）：定义 `message`、`sources`、`messageEnd`、`error` 事件模型。
- 方案 B（未采纳）：以 HTTP 轮询模拟流式。
- 原因：A 可稳定支持打字机输出、多阶段来源推送和低延迟交互。

### 决策 3：V1 模式收敛到 `Local / Web / Hybrid`
- 方案 A（采纳）：先收敛三模式，先达成主要体验闭环。
- 方案 B（未采纳）：首版直接实现全部细分 focus modes。
- 原因：A 可显著降低范围膨胀风险，先验证路由与引用协议稳定性。

### 决策 4：建立统一来源/引用契约
- 方案 A（采纳）：对本地证据与 Web 来源输出同构结构（id、title、url/source_uri、snippet、score、source_type）。
- 方案 B（未采纳）：分别渲染两套来源系统。
- 原因：A 可避免引用编号漂移，降低前端渲染复杂度和用户认知负担。

### 决策 5：采用“壳层先行、能力渐进”迁移策略
- 阶段 1：新前端 + 网关上线，后端主要调用现有 Python Local 能力。
- 阶段 2：接入 Web 检索编排并实现 Local/Web/Hybrid 路由。
- 阶段 3：扩展细分模式与高级配置面板。
- 原因：按价值密度与风险分层交付，保证每阶段可独立验收。

### 决策 6：rewrite 与 intent 从“硬拦截”改为“打分仲裁”
- 方案 A（采纳）：并行保留 rule_query 与 llm_query，通过检索质量信号（top-k 分布、rerank margin、引用覆盖率）选择最终 query。
- 方案 B（未采纳）：延续当前硬阈值校验后直接回退规则查询。
- 原因：A 能减少对 embedding/reranker 的语义破坏，避免 query 过度词法化。

### 决策 7：回答生成采用“答案规划 + 证据绑定”两段式
- 方案 A（采纳）：先生成可验证的 claim 草案，再做 claim-citation 对齐，最后渲染用户答案。
- 方案 B（未采纳）：保持单次 LLM JSON 输出并依赖后置修补。
- 原因：A 在可读性、稳定性、引用一致性上更可控，便于质量评估。

### 决策 8：GraphRAG 可视化采用“子图 + 证据联动”的轻量实现
- 方案 A（采纳）：前端展示 query 命中子图（seed 节点、扩展节点、边类型、证据映射），与回答中的 citation 双向联动。
- 方案 B（未采纳）：仅继续展示 `graph_expand` 文本标签。
- 方案 C（未采纳）：首版即引入全图库浏览器与复杂图分析交互。
- 原因：A 能显著提升可解释性，且可在不引入重依赖前提下快速落地。

## 风险 / 权衡

- [风险] 多服务部署复杂度上升（前端、网关、Python 内核）。
  -> 缓解：提供统一本地启动脚本与健康检查；明确端口与依赖矩阵。

- [风险] 网关增加后，流式延迟与错误链路更复杂。
  -> 缓解：为每轮请求注入 trace_id，统一日志字段并加网关级超时与重试策略。

- [风险] Local 与 Web 混合时引用编号可能错位。
  -> 缓解：在网关层做最终 citation 对齐校验，失败时降级为显式来源列表并标注告警。

- [风险] 迁移期双 UI 并存导致维护成本上升。
  -> 缓解：限定并存窗口，设定退役里程碑；保留 Streamlit 仅作开发审查模式。

- [风险] 路由误判（时效问题走 Local）造成“像假搜索”。
  -> 缓解：规则+语义双路由，时效词硬规则优先，持续回放评估样本。

- [风险] 打分仲裁引入后，线上行为可能波动，影响历史回归基线。
  -> 缓解：保留旧策略开关，灰度发布并构建离线评测集（rewrite 通过率、nDCG、citation 一致率）。

- [风险] Graph 可视化在大证据集下性能下降，影响首屏交互。
  -> 缓解：默认仅展示 top-N 子图，支持惰性展开与采样渲染。

- [风险] 提升回答表达后可能增加幻觉风险。
  -> 缓解：坚持 claim-citation 绑定校验，不通过则降级为保守回答并给出不确定性提示。

## 迁移计划

1. 定义并冻结网关-前端事件协议与来源数据契约。
2. 搭建 Next.js 前端壳层与基础聊天交互（输入、流式展示、来源卡片）。
3. 搭建 Node 网关并接入现有 Python Local QA 能力，完成端到端打通。
4. 引入 rewrite/intent 打分仲裁与回答两段式生成，完成策略灰度与离线评测。
5. 引入 Web 检索编排与 Local/Web/Hybrid 路由，统一引用输出。
6. 增加 GraphRAG 子图可视化与 citation 联动交互。
7. 完成稳定性与体验验收后，将新前端设为默认入口，Streamlit 降为开发模式。

回滚策略：
- 保留现有 Streamlit 启动链路与 Python 直连问答入口；当网关或前端不可用时可一键切回旧路径。

## 开放问题

- Web 模式首版检索源采用 SearxNG 自建还是外部搜索 API 托管？
- Python 内核与 Node 网关之间采用 HTTP 还是 gRPC（考虑实现成本与流式支持）？
- 统一引用中，Web 来源是否需要抓取页面正文做二次证据切片，还是先展示搜索摘要？
- 新前端是否在 V1 同步提供开发者 Inspector，还是先独立保留 Streamlit 审查页？
- 仲裁评分中的主指标权重如何设定（检索分、重排分、引用可覆盖率）？
- GraphRAG 可视化首版采用何种前端渲染方案（Canvas/SVG）以平衡性能与可维护性？

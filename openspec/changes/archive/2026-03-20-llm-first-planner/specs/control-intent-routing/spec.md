## 新增需求

### 需求:系统必须由 LLM planner 统一解释控制意图与多轮关系
系统必须由顶层 `LLM planner` 统一解释控制意图、追问/换题关系与结构化参数，并将这些结果写入正式 `planner decision`；禁止在正式执行链中再使用独立规则路由器为控制意图或多轮关系生成另一份顶层解释结果。

#### 场景:控制意图解释不再依赖独立规则链
- **当** 用户输入“换成表格展示，并继续上一轮比较”或等价表达
- **那么** 系统必须由 `LLM planner` 同时产出控制意图参数和多轮关系判断，而不得在正式执行链中再调用独立规则路由器重解释该请求

## 修改需求

### 需求:系统必须提供意图路由可观测字段
系统必须输出意图路由观测字段，至少包含路由来源、命中意图、置信度、`primary_capability`、`planner_confidence`、`interaction_decision_source` 与是否进入受控结束路径；当控制意图被识别时，trace 中必须能够证明其解释来源于正式 `LLM planner` 决策，而不是来自旧规则链或尾部补写。

#### 场景:控制意图来源可审计
- **当** 一次请求完成控制意图与多轮关系判定
- **那么** trace 中必须能够证明这些结果来自正式 `LLM planner decision`，并记录是否进入受控结束路径

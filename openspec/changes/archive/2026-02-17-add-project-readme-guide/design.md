## 上下文

当前仓库已具备论文入库、清洗、检索、问答与运行日志等能力，但根目录缺少统一 README。新加入的开发者与使用者通常需要翻阅多个脚本与历史变更才能理解“从 0 到可运行”的流程，导致上手成本高、里程碑能力不可见。

## 目标 / 非目标

**目标：**
- 在根目录提供单一 `README.md`，覆盖已实现能力、目录说明、环境准备与关键命令。
- 形成“按里程碑（M1/M1.5/M2/M2.1/M3）组织的使用指南与示例”。
- 将 README 更新纳入规范，后续里程碑交付时可持续维护。

**非目标：**
- 不改动现有算法、检索排序与数据格式。
- 不引入网站文档系统或多语言文档站点。

## 决策

### Decision 1: README 采用“能力总览 + 快速开始 + 分里程碑命令”结构
- 原因：兼顾首次上手与按阶段查阅。
- 备选：仅提供命令清单。
  - 放弃原因：缺少上下文，难以理解输入/输出与文件位置。

### Decision 2: 命令示例必须绑定真实路径与产物
- 约定示例统一使用 `data/papers`、`data/processed`、`runs/`、`reports/`。
- 原因：降低“命令能跑但不知道结果在哪”的问题。

### Decision 3: 文档维护通过规范约束而非脚本强制
- 原因：当前阶段优先轻量治理；通过规范要求在里程碑变更时同步 README。
- 备选：增加 CI 校验 README 章节完整性。
  - 放弃原因：当前收益不足，后续可在需要时添加。

## 风险 / 权衡

- [风险] README 与实际实现漂移
  → 缓解：在 `pipeline-development-conventions` 中新增维护要求，里程碑完成时同步更新。

- [风险] README 过长影响可读性
  → 缓解：主文档保留最小可用内容，深入细节放到 `reports/` 或后续子文档。

- [风险] 示例依赖环境差异导致失败
  → 缓解：明确 venv 与依赖安装步骤，并标注输入/输出前置条件。

## 迁移计划

1. 新增 README 规范能力 `project-readme-guide`。
2. 修改 `pipeline-development-conventions`，加入 README 同步维护要求。
3. 在实现阶段创建/更新根目录 `README.md` 并补示例命令。
4. 验证 README 命令与当前 CLI 一致。

回滚：若内容不完整，可先回退到最小 README（仅能力清单与快速开始），再增量补齐。

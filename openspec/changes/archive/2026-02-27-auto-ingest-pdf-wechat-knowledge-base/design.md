## 上下文

当前系统已具备以 PDF 为核心的入库、清洗、分块、检索与证据引用链路，但用户真实输入来源包含大量网页链接（尤其是公众号文章）。现有检索主要依赖 chunk 级召回，缺少论文级摘要索引，导致“先选文档范围、再看证据细节”的交互效率不足。

本变更需在不削弱既有证据门控（Sufficiency Gate / Evidence Policy Gate）的前提下，扩展输入类型并引入论文级摘要层，支持更自然的个人知识库构建流程。

约束：
- 不能破坏现有 chunk 证据可追溯与拒答机制。
- 需要与既有 `papers/chunks` 产物兼容，避免一次性重构。
- 链接抓取存在源站限制与不稳定性，必须提供可解释失败路径。

## 目标 / 非目标

**目标：**
- 支持统一导入 `PDF + URL`，并输出一致的文档元数据。
- 为每篇文档生成并持久化 `paper summary`，形成论文级检索层。
- 检索采用“两阶段”：summary 粗召回 -> chunk 精检索/重排。
- 保持回答事实引用来自 chunk 证据，摘要层仅用于召回与组织。
- 升级多轮会话能力：引入 Redis 会话存储、分层记忆与显式状态机，提升并发稳定性与跨轮语义连续性。
- 使用 LLM 独立问题重写与语义意图路由，替换高误判率的硬编码规则链。

**非目标：**
- 不在本变更中引入远程多用户知识库服务。
- 不在本变更中重写底层 embedding/rerank 算法。
- 不承诺首版覆盖所有网页站点解析规则（优先公众号常见场景）。

## 决策

### 决策 1：采用统一文档抽象层（Document Record）
- 方案 A（采纳）：将 PDF 与 URL 导入统一映射到文档记录（`doc_id`, `source_type`, `source_uri`, `title`, `ingest_status` 等），后续清洗/分块复用同一流程。
- 方案 B（未采纳）：PDF 与 URL 各自维护独立下游管线。
- 原因：统一抽象可减少双轨维护成本，便于后续检索与 UI 汇总。

### 决策 2：新增论文级摘要存储作为独立产物
- 方案 A（采纳）：新增 `paper_summary` 产物（每文档一条），字段至少包含摘要、要点、关键词、来源、版本与摘要新鲜度标识。
- 方案 B（未采纳）：查询时临时从 chunks 现算摘要，不落盘。
- 原因：落盘能提升检索稳定性、减少重复计算，并支持摘要版本化与过期检测。

### 决策 3：检索改为双层检索
- 方案 A（采纳）：先基于 summary 召回候选文档，再在候选文档内执行 chunk 检索与 rerank。
- 方案 B（未采纳）：继续全库 chunk 级一次性检索。
- 原因：双层检索更贴合“先定位文档主题、再验证细节证据”的用户任务路径，并能降低无关 chunk 干扰。

### 决策 4：摘要层不得直接作为最终引用来源
- 方案 A（采纳）：回答引用必须来自 chunk 证据；summary 可在 UI 层展示，但不得进入 `answer_citations`。
- 方案 B（未采纳）：允许 summary 直接被引用。
- 原因：摘要信息属于二次组织层，直接引用会放大摘要偏差风险，削弱防幻觉护栏。

### 决策 5：URL 抓取失败采用“可解释降级”
- 方案 A（采纳）：抓取失败时输出结构化失败原因（网络、权限、解析失败、空正文），并保留重试入口。
- 方案 B（未采纳）：统一报“导入失败”。
- 原因：可解释失败可降低用户挫败感，便于后续排障与策略优化。

### 决策 6：会话状态采用 Redis + 分层记忆
- 方案 A（采纳）：会话记录写入 Redis，按 `recent turns + summary memory + semantic recall memory` 组装上下文，并引入显式状态字段（如 `normal/need_clarify/waiting_followup`）。
- 方案 B（未采纳）：继续使用本地 JSON 文件 + 固定滑窗。
- 原因：Redis 方案可解决并发写入冲突，分层记忆可降低长对话遗忘与检索污染。

### 决策 7：查询重写采用小模型独立问题改写
- 方案 A（采纳）：使用小模型将当前轮问题改写为不依赖上下文的完整问题，再进入检索。
- 方案 B（未采纳）：沿用历史实体拼接与规则重写。
- 原因：独立问题改写在 Dense Retrieval 下更稳定，可明显改善指代、省略与中英混杂问题。

### 决策 8：意图路由采用语义路由与结构化输出
- 方案 A（采纳）：路由输出结构化意图对象（`intent + params`），并以语义相似度进行分类。
- 方案 B（未采纳）：关键词/正则模式匹配。
- 原因：语义路由覆盖表达变体能力更强，减少规则维护成本与漏判。

## 风险 / 权衡

- [风险] 网页正文抽取质量不稳定，可能混入导航/广告噪声。  
  -> 缓解：增加正文长度阈值与噪声规则；低质量内容打标并降权。

- [风险] 论文摘要生成质量波动，可能偏离原文重点。  
  -> 缓解：摘要用于召回，不用于最终引用；保留摘要版本与人工覆盖能力。

- [风险] 双层检索可能增加处理复杂度与调参成本。  
  -> 缓解：首版采用最小参数集，保留回退到单层检索的配置开关。

- [风险] 新增数据产物可能影响历史数据兼容。  
  -> 缓解：采用增量字段与旁路文件，不覆盖旧产物结构。

- [风险] 会话层重构引入迁移复杂度，可能出现上下文丢失。  
  -> 缓解：保留旧会话读取兼容层与灰度切换开关。

- [风险] LLM 重写与语义路由增加调用开销。  
  -> 缓解：采用小模型与结果缓存，设置超时后回退到最小可用路径。

## 迁移计划

1. 增加 URL 导入入口与统一文档记录层，保持 PDF 入口不变。  
2. 在入库后新增 summary 生成与落盘步骤，不改变现有 chunk 文件语义。  
3. 在检索链路接入 summary 粗召回，并提供配置开关可回退旧路径。  
4. 调整引用校验，显式禁止 summary 进入 `answer_citations`。  
5. 引入 Redis 会话存储与分层记忆组装，保留旧会话兼容读取。  
6. 接入小模型独立问题重写与语义路由，增加可观测字段。  
7. 补充回归测试：URL 导入、summary 命中、双层检索、会话并发稳定性、重写质量、语义路由准确率与引用合规。  

回滚策略：关闭双层检索与 URL 导入开关，恢复 PDF + chunk 单层路径；保留新增产物文件但不参与运行。

## 开放问题

- 公众号链接首版是否需要登录态/反爬兼容策略，还是仅支持无鉴权可访问页面？
- `paper_summary` 的更新触发条件采用“每次重建”还是“基于 chunk 变更检测（hash）”？
- summary 召回层是用 BM25、向量索引，还是混合召回作为默认？
- URL 来源的元数据最小字段集是否需要包含发布时间与作者信息（若可抽取）？

### 决策 9：证据充分性主判定采用轻量语义相似度
- 方案 A（采纳）：默认使用轻量 embedding 模型计算 query-evidence 余弦相似度，替换纯 Token Overlap 主判定；当 embedding 不可用（如密钥缺失/网络失败）时回退到轻量词向量近似方案并记录可观测字段。
- 方案 B（未采纳）：继续沿用纯规则 Token 交集。
- 原因：语义相似度可显著改善同义词与中英混杂场景下的误杀与漏放，同时保持低时延。

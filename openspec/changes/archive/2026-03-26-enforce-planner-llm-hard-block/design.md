## 上下文

当前仓库已经在规范和实现上将 `LLM-first planner` 定义为线上正式顶层决策来源，但系统仍保留了一种混合语义：

- 对单轮请求来说，planner LLM 不可用时会进入 `controlled_terminate`
- 对系统运行态来说，聊天入口、健康检查和前端交互仍可能表现为“服务在线”

这种设计在“planner LLM 只是增强能力”时还说得过去，但与当前项目的架构定位已经冲突。现在 planner LLM 已经承担：

- 顶层意图理解
- tool / skill 选择
- 执行顺序规划
- 最终交互姿态来源

因此它不再只是单次请求中的一个可替换组件，而是正式聊天链路的基础设施依赖。利益相关者包括：

- 最终用户：需要在基础设施不可用时得到明确阻断，而不是模糊失败
- 运维与管理员：需要从健康检查和设置页一眼判断系统是否可服务
- 开发者：需要明确哪些 fallback 仍然合法，哪些已经升级为系统阻断

## 目标 / 非目标

**目标：**

- 将 `planner LLM` 不可用从“请求级受控结束”提升为“系统级阻断”。
- 明确区分系统级阻断与请求级失败收束，避免二者混淆。
- 在 Kernel、Gateway、前端和健康检查中暴露统一的阻断语义。
- 收紧配置治理，消除“正式模式下关闭 planner LLM”这一无效状态，并让关键 planner 配置缺失被视为阻断态。
- 保留受控结束语义用于“planner 已可用，但单轮决策或执行失败”的情况。

**非目标：**

- 不在本设计中重新定义 planner decision schema。
- 不在本设计中引入新的 planner provider 或新的 tool 类型。
- 不要求在此变更中重做整个健康检查 API 结构，只要求其稳定表达阻断态。
- 不把开发/诊断模式完全移除；但必须让其与正式可服务模式显式区分。

## 决策

### 决策 1：建立“系统级阻断”与“请求级失败收束”两层失败模型

系统将失败分为两层：

- `system_blocked`：planner LLM 基础设施未满足正式运行前置条件，正式聊天入口不可服务
- `request_terminated`：planner 基础设施已就绪，但某一轮请求在 validation、tool、constraint 或 runtime 中受控结束

原因：

- 当前 `controlled_terminate` 语义过载，同时承担“系统没准备好”和“本轮请求不能继续”两类含义。
- 对用户和运维来说，这两类失败的处理动作完全不同：前者需要修复配置或依赖，后者只影响单轮请求。

考虑过的替代方案：

- 方案 A：继续只用 `controlled_terminate`，靠 reason code 区分
  - 未选用，因为这仍会让上层把系统阻断误当成普通请求失败。
- 方案 B：让前端自己根据 fallback reason 猜测是不是阻断
  - 未选用，因为阻断判定应由后端真相源统一给出。

### 决策 2：正式模式下删除 `planner.use_llm` 作为可切换开关

在正式模式下，`planner.use_llm` 不应继续存在为一个可切换的 runtime 布尔开关。对于正式聊天链路，planner 默认且必须使用 LLM；系统是否可服务应由“配置是否完整、依赖是否健康”来决定，而不是由一个布尔值决定。

若未来仍需保留无 planner LLM 的开发、离线评测或诊断场景，应通过独立模式字段表达，例如 `service_mode=production|diagnostic`，而不是通过 `planner.use_llm=false` 复用正式配置面。

原因：

- `planner.use_llm` 让“基础设施缺失”伪装成“合法配置选择”。
- 这会让治理模型、健康检查和前端设置页都陷入歧义：系统到底是被关闭、被降级，还是其实已经不可服务。

考虑过的替代方案：

- 方案 A：保留 `planner.use_llm`，但在正式模式下把 `false` 视为阻断
  - 可行，但仍保留了一个本质上无效的开关，增加长期治理复杂度。
- 方案 B：保留 `planner.use_llm` 仅供开发环境使用
  - 未选用，因为同一字段在不同模式下语义差异过大，容易继续污染正式配置面。

### 决策 3：正式聊天入口在执行前进行 planner 基础设施门禁检查

在正式模式下，聊天请求进入主 planner runtime 之前，系统必须先完成一次轻量门禁检查。至少覆盖：

- planner model 是否存在
- planner API key 是否可解析
- planner 运行时配置是否形成合法正式配置
- 健康状态是否已标记 planner 基础设施不可用

若任一关键条件不满足，系统直接返回系统级阻断响应，而不是继续进入 planner validation 或 tool 执行。

原因：

- 当前很多失败是在 planner runtime 内部才暴露，导致表现上仍像一次正常请求。
- 基础设施级缺失应尽量在入口处短路，避免污染单轮 trace。

考虑过的替代方案：

- 方案 A：允许请求先进入 runtime，再由 runtime 统一受控结束
  - 未选用，因为这会继续把系统级依赖缺失降格为单轮请求错误。

### 决策 4：健康检查必须显式输出 planner LLM 阻断态

依赖健康检查和运行态概览需要新增或稳定化 planner 阻断语义，例如：

- planner 是否已启用正式模式
- planner 配置是否完整
- planner LLM 是否满足基础可调用条件
- 当前阻断原因代码
- 该阻断是否影响正式聊天可用性

健康检查不是为了证明 planner 一定能回答所有请求，而是为了证明“正式聊天链路可以开始运行”。

原因：

- 只在聊天请求时才暴露阻断，运维成本太高。
- 该系统已有多服务边界，阻断态需要先被 Gateway 和前端读取，而不是靠用户试错。

考虑过的替代方案：

- 方案 A：仅在聊天 API 返回错误，不调整健康检查
  - 未选用，因为这会让部署系统错误地继续把服务判定为健康。

### 决策 5：前端与 Gateway 仅转发阻断真相，不自行推断

阻断判定由 Kernel 给出；Gateway 和前端负责保持语义不失真：

- Gateway 区分 `system_blocked` 与普通 `planner/tool fallback`
- 前端在阻断态下不再展示可继续发送的正常聊天姿态，而是明确提示不可服务原因和修复方向

原因：

- 多服务系统里，语义漂移最容易发生在中间层和 UI 层。
- 如果 Gateway 或前端自行基于 reason code 猜测，未来容易和后端真相源脱节。

考虑过的替代方案：

- 方案 A：前端只看到普通失败消息
  - 未选用，因为这无法形成稳定的服务级阻断体验。

### 决策 6：保留显式开发/诊断模式，但必须与正式模式隔离

本设计允许保留无 planner LLM 的开发/诊断场景，但要求：

- 正式模式下，不再通过 `planner.use_llm=false` 表达“关闭 planner”
- 若保留开发/诊断模式，必须显式标识为非正式服务模式
- 非正式模式不得被健康检查或前端误报为“正式可服务”

原因：

- 开发和离线诊断仍可能需要无 LLM 运行环境
- 但这些场景不能再与正式模式共享相同可用性语义

考虑过的替代方案：

- 方案 A：彻底禁止所有无 planner LLM 运行方式
  - 未选用，因为会压缩开发与诊断弹性。

## 风险 / 权衡

- [风险] 行为收紧后，现有默认配置 `planner_use_llm=false` 会让更多环境直接进入阻断。
  → 缓解措施：在迁移期提供兼容读取，但在正式模式下忽略该开关并输出明确迁移提示。

- [风险] 系统级阻断与请求级失败若字段设计不清晰，仍可能在 Gateway 和前端被误映射。
  → 缓解措施：使用单独阻断语义字段，而不是继续复用普通 fallback reason。

- [风险] 若仅基于静态配置判断可用性，可能高估 planner 真正可服务性。
  → 缓解措施：区分“配置就绪阻断”与“运行期调用失败阻断”，允许健康检查逐步演进为更强依赖探测。

- [风险] 显式保留开发模式会留下边界模糊空间。
  → 缓解措施：要求正式模式和诊断模式在健康状态、前端文案与可观测字段上完全可区分。

## Migration Plan

1. 先在 specs 中补充系统级阻断与请求级失败的语义边界。
2. 收紧 planner runtime 配置治理，从正式配置面移除 `planner.use_llm`，并定义新的模式表达方式或兼容迁移规则。
3. 在 Kernel 聊天入口与健康检查中引入 planner 基础设施门禁判断。
4. 调整 Gateway 事件与前端状态映射，使阻断态与普通 fallback 可区分。
5. 为历史环境提供迁移窗口，但正式模式下不再接受 `planner.use_llm=false` 作为合法运行配置。
6. 如上线后出现误阻断，可临时回滚到旧语义，但必须保留新观测字段以定位根因。

## Open Questions

- 是否需要一个显式的 `service_mode=production|diagnostic` 顶层配置字段，作为替代 `planner.use_llm` 的模式表达？
- 健康检查是否应增加轻量 planner provider 探测，还是先以配置完备性为阻断门槛？
- 前端阻断态是否需要区分“管理员可修复”和“宿主机/密钥问题”两类提示？

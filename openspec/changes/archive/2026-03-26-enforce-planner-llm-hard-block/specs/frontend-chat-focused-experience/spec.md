## 新增需求

### 需求:系统必须在 planner LLM 阻断态下阻止正常聊天姿态
聊天页在读取运行态摘要、健康状态或聊天入口阻断信号后，必须能够识别 planner LLM 基础设施未就绪的服务阻断态。处于该状态时，系统必须优先呈现明确的不可服务提示、阻断原因或管理员修复指引；禁止继续向普通用户呈现看似可正常发送并获得正式回答的聊天姿态。

#### 场景:聊天页识别 planner 阻断并禁用正式发送
- **当** 前端检测到 planner LLM 基础设施阻断状态
- **那么** 页面必须明确展示不可服务提示，并且不得继续将当前聊天入口表现为正式可用会话

## 修改需求

### 需求:系统必须在 llm-first planner 迁移期保持聊天体验兼容稳定
聊天页在 llm-first planner 与 shadow mode 共存的迁移期，必须继续以标准聊天协议闭合会话，并允许高层 planner / tool / fallback 事件作为增强层存在；禁止因 shadow 对比或 planner source 切换导致用户可见消息链路中断、重复闭合或必须依赖调试字段才能正常使用。对于系统级 planner LLM 阻断，页面必须优先展示服务不可用状态，而不是继续模拟普通受控结束对话体验。

#### 场景:shadow 模式下聊天主体验保持不变
- **当** 后端在 shadow mode 下额外生成诊断记录
- **那么** 聊天页必须仍以单一主回答流展示本轮结果，而不是向普通用户展示任何竞争性的 planner 结果或诊断细节

#### 场景:planner 进入受控结束后聊天仍正常闭合
- **当** LLM planner 决策被拒绝并进入受控结束路径
- **那么** 聊天页必须仍能收到受控的标准消息闭合事件，并保持历史会话和输入区可继续使用

#### 场景:服务阻断优先于普通闭合体验
- **当** 后端声明 planner LLM 基础设施未就绪并阻断正式聊天
- **那么** 聊天页必须优先展示服务阻断态，而不得把该状态渲染为一条普通 AI 拒答消息

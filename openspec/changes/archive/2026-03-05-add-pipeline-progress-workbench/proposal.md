## 为什么

当前图构建等长耗时任务缺少用户可见进度，用户容易误判系统卡死并中断操作。同时前端主界面仅覆盖聊天与模型配置，缺少“导入论文→切块→清洗→图构建”的产品化入口，导致后端能力难以被非开发用户使用。

## 变更内容

- 增加统一任务进度可见性，优先覆盖图构建流程，向前端提供阶段状态、已处理数量、耗时与失败信息。
- 扩展前端信息架构，新增面向用户的流水线工作台入口，串联导入论文、切块清洗、图构建与聊天验证。
- 在网关/内核协议中引入任务型事件或接口契约，使任务状态与聊天事件解耦，避免混淆。
- 保持现有聊天与引用交互能力不回退，新增能力以渐进方式落地（Phase 1: 进度可见；Phase 2: 工作台入口）。

## 功能 (Capabilities)

### 新增功能
- `pipeline-task-observability`: 定义非聊天任务的统一状态机、进度事件与错误恢复约定，支持长任务实时可见。

### 修改功能
- `research-assistant-workbench`: 扩展工作台导航与任务中心，补充从 Library 到 Pipeline 到 Chat 的连续流程体验。
- `library-quick-ingestion`: 增加导入后处理链路可见性与下一步引导，强化“导入后可继续动作”规范要求。
- `node-gateway-orchestration`: 在统一编排层补充任务进度事件契约，明确与现有 chat 四事件并行共存策略。

## 影响

- 前端：新增工作台入口/页面区域、任务状态组件、进度条与任务日志摘要。
- 网关：扩展事件类型或任务通道，增加任务生命周期管理与状态转发。
- Python Kernel：为图构建等流程暴露任务启动、状态查询、进度推送与最终结果接口。
- 测试：新增任务状态机与事件协议回归用例，补充前端可见性 E2E 断言。

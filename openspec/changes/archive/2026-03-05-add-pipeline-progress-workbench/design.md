## 上下文

当前系统以聊天链路为主，网关协议仅覆盖 `message/sources/messageEnd/error` 四类事件。图构建等长耗时任务主要通过后端脚本运行，缺少对前端可消费的进度状态，导致用户在等待期间无法判断系统是否仍在工作。与此同时，前端入口聚焦聊天与模型配置，尚未形成“导入论文-处理-问答验证”的连续工作流。

本变更跨越 Frontend、Gateway、Python Kernel 三个域，目标是在不破坏现有聊天体验的前提下，引入任务型可观测能力并扩展工作台入口。

约束：
- 保持现有聊天事件协议兼容，避免前端既有聊天流程回归。
- 任务进度必须优先覆盖图构建（最主要长耗时环节），再扩展到导入流水线。
- 面向普通用户的文案必须使用任务语义，避免暴露底层工程术语。

## 目标 / 非目标

**目标：**
- 定义统一任务状态机与进度事件契约，支持长任务实时可见。
- 为图构建提供可消费的阶段进度反馈（阶段、已处理数量、耗时、失败信息）。
- 在前端建立 Pipeline Workbench 入口，连通导入、处理、建图、聊天验证流程。
- 通过分阶段发布降低风险：Phase 1 先图构建可见性，Phase 2 再扩展前端工作台能力。

**非目标：**
- 本变更不重写检索算法、重排逻辑或引用分配策略。
- 本变更不替换现有聊天 UI 的核心交互结构，仅做入口扩展与任务面板补充。
- 本变更不引入新的外部任务队列中间件（如 Celery/RQ/Kafka）。

## 决策

### 决策 1：引入独立于聊天事件的任务进度事件类型
- 决策：在网关协议中新增任务事件（如 `taskProgress`、`taskState`、`taskResult`），不复用聊天 `message` 事件。
- 理由：聊天事件语义面向回答流，任务进度语义面向后台作业生命周期，混用会造成前端状态机歧义。
- 备选方案：
  - 方案 A：将任务进度伪装为聊天 `message` 文本流。缺点是前端难以结构化渲染，且污染会话历史。
  - 方案 B：仅靠轮询日志文件。缺点是实时性与一致性差，用户感知滞后。

### 决策 2：统一任务状态机为 `idle/queued/running/succeeded/failed/cancelled`
- 决策：所有流水线任务复用统一状态机，前端组件基于状态机渲染。
- 理由：可降低前后端耦合，避免每个任务定义一套私有状态。
- 备选方案：按任务自定义状态。缺点是维护成本高，前端组件难复用。

### 决策 3：图构建进度按“阶段 + N/N”双层粒度输出
- 决策：阶段级进度覆盖全流程，实体抽取阶段额外输出 chunk 级 `processed/total`。
- 理由：阶段级进度可快速上线；chunk 级进度可显著缓解“卡住感”。
- 备选方案：只输出百分比。缺点是百分比可信度不足，难解释耗时变化。

### 决策 4：前端新增 Workbench 入口并保留 Chat 默认主流程
- 决策：新增 Pipeline 工作台入口，Chat 保持快速提问路径；任务操作集中到 Workbench。
- 理由：兼顾现有用户习惯与新用户的全流程可见性。
- 备选方案：把所有任务控件直接塞进 Chat 页面。缺点是页面复杂度升高、心智负担加大。

## 风险 / 权衡

- [风险] 任务事件与聊天事件并发导致前端竞态 → [缓解] 以 `taskId` 与 `traceId` 分域管理状态，事件处理器分通道注册。
- [风险] 进度数据不准确导致用户误解 → [缓解] 明确区分“阶段状态”与“估算进度”，无法估算时展示耗时与心跳而非假精度百分比。
- [风险] 重复触发任务导致资源竞争 → [缓解] 对同类任务增加并发互斥与幂等策略，返回“已有运行中任务”提示。
- [风险] 分阶段发布造成用户体验不一致 → [缓解] 在 Phase 1 先提供图构建入口与状态面板，保证最主要痛点优先解决。

## Migration Plan

1. Phase 1（后端可见性优先）
- 在 Python Kernel 提供图构建任务启动与状态查询能力。
- 网关转发任务状态事件至前端。
- 前端增加最小任务面板展示图构建进度。

2. Phase 2（工作台入口扩展）
- 在前端新增 Pipeline Workbench（导入、切块清洗、建图、结果摘要）。
- 扩展任务中心以统一展示多任务历史与失败重试入口。
- 与 Chat 页面建立“处理完成后继续提问”的流转入口。

3. 回滚策略
- 回滚时保留原有聊天链路与四事件协议。
- 新增任务事件与工作台入口可独立关闭，不影响已有聊天能力。

## Open Questions

- Pipeline Workbench 与 Chat 的路由关系采用并列页面还是单页分区切换？
- 图构建失败后的重试策略是否允许“从失败阶段继续”而非全量重跑？
- 任务历史保存周期与清理策略如何定义，避免本地存储无限增长？

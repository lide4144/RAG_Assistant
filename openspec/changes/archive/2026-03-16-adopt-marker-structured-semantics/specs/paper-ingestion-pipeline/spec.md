## 修改需求

### 需求:论文元数据映射输出
系统必须为每篇已处理文档写入统一映射记录，并在原有 `paper_id`、`title`、`path` 基础上扩展 `source_type` 与 `source_uri`（或等价字段）以区分 PDF 与 URL 来源。对于 PDF 来源，系统必须额外记录解析引擎与标题质量信息（至少包括 `parser_engine`、`title_source`、`title_confidence` 或等价字段），并在 Marker 成功解析时按结构化信任路径决策标题。系统禁止将命中标题黑名单的候选直接写入 `title`，也禁止在 Marker 已提供合规顶层标题时退回采用更低优先级来源。

#### 场景:PDF 映射包含结构化标题来源
- **当** 一篇 PDF 文档通过 Marker 成功解析并导入成功
- **那么** 文档映射记录必须包含可定位到结构化层级的标题来源语义，并可用于排查标题误识别

#### 场景:结构化信任阻止误用章节标题
- **当** Marker 同时识别出论文主标题与正文章节标题
- **那么** 系统必须将主标题写入映射记录，不得因章节标题文本更短或规则分更高而覆盖主标题

#### 场景:标题候选命中黑名单
- **当** 当前优先层的标题候选命中占位或版权黑名单模式
- **那么** 系统必须拒绝该候选并继续尝试下一层合规候选或安全降级标题，不得直接写入黑名单标题

### 需求:成功解析后的结构化输入状态必须落盘
当 PDF 通过 Marker 成功解析时，入库流程必须落盘该文档有哪些结构化输入已成功保留、哪些仅部分消费、哪些因兼容性原因尚未进入优先链路。系统禁止把“结构语义存在但未消费”和“结构语义本身不存在”混为同一状态。

#### 场景:block 语义保留状态可审计
- **当** Marker 成功解析并输出 block 类型
- **那么** 入库元数据或报告必须能表明这些 block 语义是否已保留到统一中间表示

#### 场景:markdown 仅部分消费时显式披露
- **当** markdown 输出存在但当前仅被用于局部用途，例如标题候选
- **那么** 入库元数据或报告必须显式标记该 markdown 为“部分消费”，而不是默认为“已完整采用”

### 需求:结构块语义必须贯通到 chunking
当 PDF 通过 Marker 成功解析并识别出表格块、公式块或其他显著结构块时，入库流程必须在 chunk 生成阶段保留这些块语义，使对应 chunk 能携带结构类型。系统禁止在 chunk 层把结构块完全扁平为不可区分的普通正文。

#### 场景:表格块生成可区分 chunk
- **当** Marker 识别出表格块并进入 chunking
- **那么** 生成的 chunk 必须保留其表格语义，供后续检索与 citation 使用

#### 场景:公式块生成可区分 chunk
- **当** Marker 识别出公式块并进入 chunking
- **那么** 生成的 chunk 必须保留其公式语义，供后续检索与 citation 使用

### 需求:历史标题修复路径
系统必须提供能够复用结构化标题信号的历史标题修复路径。对于历史 PDF 记录，系统必须优先使用已存在的结构化候选或可重跑得到的 Marker 结果重新执行同一标题决策树；系统禁止仅以当前落盘标题作为 metadata 重新评分并将其称为结构化修复。

#### 场景:历史修复复用结构化候选
- **当** 某篇历史论文存在可访问的 Marker 结构化候选或可重跑得到该候选
- **那么** 修复流程必须沿结构化信任路径重算标题，并更新对应标题来源与置信度

#### 场景:历史修复缺少结构化输入
- **当** 某篇历史论文既无可用结构化候选也无法安全重跑 Marker
- **那么** 系统必须显式标记该论文无法进行结构化修复，而不是静默沿用旧标题完成“修复”

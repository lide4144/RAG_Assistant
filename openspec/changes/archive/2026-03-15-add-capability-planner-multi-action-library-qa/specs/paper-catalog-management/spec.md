## 新增需求

### 需求:系统必须支持聊天侧复用目录筛选结果
系统必须允许聊天侧通过目录查询能力获取论文集合，并将结果作为后续总结或问答步骤的输入范围；禁止要求聊天侧仅通过正文 chunk 检索反推论文列表。

#### 场景:聊天侧先筛选再对比
- **当** 用户输入“列出我昨天上传的 3 篇大模型论文，并用表格对比一下它们的方法差异”
- **那么** 系统必须能够先返回满足条件的论文集合，并将该集合提供给后续对比步骤

### 需求:系统必须对目录结果集施加硬上限并披露截断
当目录查询命中数量超过后续执行上限时，系统必须裁剪为受控的 `selected_paper_set`，并输出 `matched_count`、`selected_count` 与 `truncated`；禁止将未经裁剪的大结果集直接交给后续总结或问答流程。

#### 场景:目录命中过大时披露截断
- **当** 目录查询命中 500 篇论文而下游执行上限为 20
- **那么** 系统必须仅返回受控的 20 篇论文集合并披露已发生截断

## 修改需求

### 需求:系统必须展示论文可维护元数据
目录页必须展示每篇论文的稳定标识、标题、来源类型、导入时间与当前状态，支持用户判断是否重复导入或待处理。对于聊天侧复用场景，目录查询返回结果也必须至少包含上述核心元数据，以支持后续计划执行与结果披露。

#### 场景:用户查看论文详情
- **当** 用户打开某论文详情
- **那么** 系统必须展示核心元数据与专题归属信息

#### 场景:聊天侧目录筛选返回可执行元数据
- **当** 聊天侧通过目录筛选得到论文集合
- **那么** 返回结果必须包含 `paper_id`、标题、导入时间、状态等可用于计划执行的核心元数据

## 上下文

本次变更聚焦“机器语义到人类语义”的前端体验重构。当前系统功能可用，但页面存在三类系统性问题：
1. 直接暴露底层变量、枚举和 JSON，用户难以理解。
2. 状态表达分散且风格不一致，缺乏跨页面统一语义。
3. 页面视觉层级和信息密度失衡，关键操作与关键状态不突出。

变更范围覆盖全局壳层、知识处理页、模型设置页、聊天页，属于跨模块 UI 架构演进，需统一组件策略与状态映射约束。

## 目标 / 非目标

**目标：**
- 建立全局 `StatusMapper`，统一状态词典、图标和文案映射。
- 将 4 个目标页面重构为符合现代 B 端 SaaS 的信息层级与语义化交互。
- 在不改变后端 API 合约的前提下完成主要体验升级。
- 移除面向用户的原始 JSON 渲染路径，避免技术噪音泄露。

**非目标：**
- 不重构后端业务流程或新增复杂数据计算逻辑。
- 不替换既有路由与导航信息架构。
- 不在本次变更中引入全量设计系统迁移（仅做必要组件抽象）。

## 决策

1. 决策：引入前端全局 `StatusMapper` 工具层并集中管理映射。
- 方案 A（选中）：`StatusMapper` 作为单一来源，页面通过 key 获取文案/图标/色彩 token。
- 方案 B：各页面各自写 switch 映射。
- 理由：A 可避免语义漂移并减少重复逻辑，便于后续扩展状态集。

2. 决策：告警和连接状态采用“信息组件”而非“操作按钮”范式。
- 方案 A（选中）：告警用可折叠 Alert；连接用无边框状态指示器。
- 方案 B：延续按钮式视觉承载状态。
- 理由：A 能减少误点击暗示，增强语义正确性。

3. 决策：知识处理页采用“指标卡 + 横向步骤条 + 终端日志 + 历史表格”的四层结构。
- 方案 A（选中）：通过 Steps 组件串联四阶段，并将日志与历史信息分层。
- 方案 B：维持独立卡片平铺。
- 理由：A 更符合流水线心智模型，降低扫描成本。

4. 决策：模型设置采用“默认继承 + 显式 Override”交互。
- 方案 A（选中）：辅助模型默认继承全局凭据，按需展开独立配置。
- 方案 B：所有模型默认展示完整凭据输入。
- 理由：A 可显著降低表单噪音与误配置概率。

5. 决策：聊天页空态引导与摘要标签化并行落地。
- 方案 A（选中）：空态卡片支持点击即发，摘要使用语义标签组。
- 方案 B：保留空白容器与字符串拼接摘要。
- 理由：A 降低冷启动门槛并提高可读性。

## 风险 / 权衡

- [风险] 统一映射后遗漏旧状态值，可能出现未映射渲染。
  → 缓解：StatusMapper 提供 unknown fallback 与日志告警。

- [风险] 页面结构变化较大，可能引入视觉回归。
  → 缓解：按页面分步提交并补充截图回归与交互检查。

- [风险] “禁止 JSON 直出”可能影响开发期排障效率。
  → 缓解：保留开发模式受控调试面板，生产默认关闭。

- [风险] Override 逻辑引入条件分支，可能导致保存载荷不一致。
  → 缓解：在提交层统一规范化 payload，并为继承/覆盖两条路径补充测试。

## Migration Plan

1. 新增 `StatusMapper` 与状态 UI 原子组件（Badge/Indicator/StepState）。
2. 重构全局壳层告警条和连接状态显示。
3. 重构知识处理页：统计卡、步骤条、日志控制台、历史表格。
4. 重构模型设置页：继承/覆盖交互、未保存徽章、移除 JSON 回显。
5. 重构聊天页：空态建议卡、模型摘要标签。
6. 补充单元测试与 E2E 验证，完成视觉回归。

回滚策略：
- 每个页面改造保留独立提交点，可按页面回滚。
- `StatusMapper` 若异常，可临时回退到原页面内映射（受特性开关控制）。

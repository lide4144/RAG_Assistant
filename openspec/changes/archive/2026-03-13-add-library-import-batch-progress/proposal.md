## 为什么

当前知识处理页的“处理状态”区域主要表达四阶段流程是否推进，却没有可靠表达“这一批论文已经处理了多少、还有多少、哪几篇卡住了”。当用户一次导入多篇 PDF 且后台持续执行入库与清洗时，页面会长期停留在“处理中/待处理”的语义上，导致用户无法判断任务是否仍在前进、距离完成还有多远。

现在需要把导入进度从“只有阶段状态”提升为“批次进度 + 阶段进度 + 单篇状态”的可读反馈，避免多论文长任务在前端表现为静态等待。

## 变更内容

- 为知识处理页的导入任务增加面向批量论文的进度可视化，区分“总体完成度”和“当前阶段状态”。
- 将导入任务进度契约从单一阶段消息扩展为可用于前端渲染批次进度的信息，包括总量、已完成、处理中、失败与当前处理项。
- 调整导入状态卡的表达方式，使其优先回答“处理了多少篇/剩多少篇/失败多少篇”，而不是只展示 4 阶段流程。
- 为多篇论文场景补充逐项状态明细与失败原因展示，便于用户理解长时间运行中的真实工作量。

### 修改功能
- `library-quick-ingestion`: 导入入口的任务反馈必须从“阶段摘要”提升为“批次进度 + 阶段流 + 单篇状态”的可理解进度视图。
- `pipeline-task-observability`: 任务进度事件必须支持导入批次的结构化统计，允许前端渲染真实工作量进度，而不是仅靠阶段名称推导伪百分比。

## 影响

- 前端知识处理工作台中的导入面板与处理状态卡布局。
- 导入任务状态查询接口及其前端类型定义。
- 后端导入任务进度采集与聚合逻辑。
- 与导入进度相关的前端/后端契约测试与交互测试。

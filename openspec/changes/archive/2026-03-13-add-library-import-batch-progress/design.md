## 概览

本次变更只关注“导入任务如何被用户理解”，不改变导入主流程本身。设计目标不是新增一个装饰性进度条，而是把当前以阶段为中心的状态卡改造成面向批量任务的进度面板。

进度面板分为三层：

1. 批次总体进度：回答“这一批处理到哪里了”。
2. 阶段流程进度：回答“当前走到导入、清洗、索引、图构建中的哪一步”。
3. 单篇论文状态：回答“哪几篇已完成、哪几篇处理中、哪几篇失败”。

## 问题重述

当前界面的主要问题不是完全没有百分比，而是百分比语义不可信：

- 现有主卡以 `completedStageCount/4` 推导流程百分比，更接近“阶段完成度”而不是“工作量完成度”。
- 多篇论文场景下，用户更关心 `processed/total/failed`，而不是仅知道当前阶段是“清洗”还是“索引”。
- 如果继续用固定阶段权重生成总百分比，会在长时清洗或部分失败场景中制造“界面在走，任务却没实质推进”的错觉。

## 目标界面模型

```text
┌─────────────────────────────────────────────┐
│ 处理状态                              2/5完成 │
│ 当前：清洗中 · 已耗时 02:14 · 1 篇失败        │
│                                             │
│ 总体进度                                    │
│ ████████████░░░░░░░░ 40%                    │
│ 5 篇论文 · 已完成 2 · 处理中 2 · 失败 1       │
│                                             │
│ 阶段流                                      │
│ [导入 ✓]──[清洗 ▶ 3/5]──[索引 ○]──[图构建 ○]  │
│                                             │
│ 论文明细                                    │
│ A.pdf     清洗中      65%                    │
│ B.pdf     已完成      完成                    │
│ C.pdf     失败        bad pdf                │
└─────────────────────────────────────────────┘
```

## 信息架构

### 顶部摘要

- 显示 `已完成/总数` 作为首要结论。
- 辅助显示 `处理中数量`、`失败数量`、累计耗时或最近刷新时间。
- 当无批量统计时，允许退化到“任务已受理，正在处理中”的文案，但不得显示伪造的完成百分比。

### 总体进度条

- 仅在存在可靠的 `completed` 与 `total` 统计时渲染确定性进度条。
- 总体百分比以“已进入终态的论文数 / 总论文数”计算。
- `failed` 计入已处理工作量，但在文案上单独高亮，避免“失败也算完成”被误读。

### 阶段流

- 保留四阶段流程卡，用来表达当前系统所处环节。
- 当前阶段可额外显示该阶段内部的 `processed/total`，例如“清洗 3/5”。
- 阶段流只表达流程位置，不再承担整批任务百分比的唯一来源。

### 单篇论文列表

- 面向多论文任务显示最近若干条论文状态，至少包含文件名、当前状态、所在阶段、失败原因。
- 当批次很大时允许折叠，只展示最慢/失败/正在处理的条目。
- 单篇进度缺失时可只显示状态标签，不要求对每篇都提供百分比。

## 数据契约

### 前端必需字段

导入任务进度事件至少需要支持：

- `batch_total`
- `batch_completed`
- `batch_running`
- `batch_failed`
- `current_stage`
- `current_item_name`
- `stage_processed`
- `stage_total`
- `recent_items[]`，每项包含 `name`、`state`、`stage`、`message`

### 兼容策略

- 若后端仅提供旧版 `stage/processed/total/message`，前端继续渲染简化版阶段卡。
- 若后端提供批次统计，则升级为完整的批次进度面板。
- 前端不得根据四阶段固定权重生成“看似精确”的总体百分比来冒充真实批次进度。

## UI 组件策略

本次推荐使用 magic-mcp 方向的组合组件，而不是单独套一个普通 progress bar：

- `task progress dashboard`: 用于顶部批次进度和统计摘要。
- `multi step progress`: 用于中部四阶段流程。
- `timeline upload` 或同类列表：用于底部论文逐项状态。

组件选择原则：

- 主视觉以横向信息密度为先，不使用圆形进度作为核心布局。
- 重点是长时间盯看的可读性，而不是瞬时动画效果。
- 颜色用于区分 `running/succeeded/failed`，不用于制造夸张的完成感。

## 风险与未知数

### 批次统计来源

当前任务状态机已支持 `processed/total`，但更像阶段级统计；是否能稳定提供“按论文粒度”的完成数，需要在实现时确认导入链路的插桩点。

### 单篇状态成本

如果每篇论文的清洗/索引状态需要穿透多个子流程，后端可能先只能提供“最近若干项”而非完整批次明细。规范允许渐进增强，但至少要先满足总量进度可信。

### 图构建阶段语义

图构建可能是整批任务的后处理步骤，而不是逐篇独立完成。界面上应允许出现“论文已全部入库，但图构建仍在收尾”的状态，避免把二者混成同一进度含义。

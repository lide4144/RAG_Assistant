## 新增需求

### 需求:批量论文摄取命令
系统必须提供命令 `python -m app.ingest --input <dir> --out <dir>`，用于批量处理输入目录中的 PDF 文件并在输出目录写入结果文件。

#### 场景:命令参数有效时执行成功
- **当** 用户执行 `python -m app.ingest --input data/papers --out data/processed`，且输入目录存在可读取的 PDF 文件
- **那么** 系统必须完成批处理并在输出目录生成 `chunks.jsonl` 与 `papers.json`

### 需求:输入文件发现与过滤
系统必须仅处理输入目录下扩展名为 `.pdf` 的文件，且必须忽略非 PDF 文件。

#### 场景:目录包含混合文件
- **当** 输入目录同时包含 `a.pdf`、`b.txt`、`c.md`
- **那么** 系统必须仅将 `a.pdf` 纳入处理集合

### 需求:论文元数据映射输出
系统必须为每篇已处理论文分配稳定的 `paper_id`，并在 `papers.json` 中记录 `paper_id`、标题、源文件路径的映射关系。

#### 场景:论文处理完成后写入映射
- **当** 一篇 PDF 被成功纳入批处理
- **那么** `papers.json` 中必须存在对应条目，且包含 `paper_id`、`title`、`path` 字段

### 需求:chunk 数据结构
系统必须将解析结果写入 `chunks.jsonl`，且每个 chunk 记录必须包含 `chunk_id`、`paper_id`、`page_start`、`text` 字段。

#### 场景:任意 chunk 记录结构校验
- **当** 读取 `chunks.jsonl` 的任意一行并解析为对象
- **那么** 该对象必须包含 `chunk_id`、`paper_id`、`page_start`、`text`，且 `text` 不得为空字符串

### 需求:分块策略
系统必须优先依据可识别章节标题进行分段，并在分段过长时使用滑窗策略进行切块，目标窗口大小必须落在 300-500 token 近似区间，且相邻窗口必须包含约 50 token 重叠。

#### 场景:超长段落触发滑窗
- **当** 某段文本超过单个 chunk 目标长度
- **那么** 系统必须将其拆分为多个重叠 chunk，且相邻 chunk 保持约 50 token 重叠

### 需求:页码可追溯性
系统必须保证每个 chunk 的 `page_start` 可追溯到原始 PDF 的实际起始页。

#### 场景:人工抽样校验页码
- **当** 从已处理论文中抽样 chunk 并回查 PDF 源页
- **那么** `page_start` 必须与文本起始页一致，允许少量噪声但不得系统性错位

### 需求:异常容错
系统必须在遇到单页解析异常时记录日志并跳过异常页，且不得因单页异常中止整篇或整批任务。

#### 场景:单页解析失败
- **当** 某 PDF 的某一页发生解析异常
- **那么** 系统必须记录该页异常信息并继续处理后续页面与后续文件

### 需求:最小稳定性验收
系统必须支持对至少 20 篇论文完成处理且流程无崩溃，以满足 M1 的最小可用验收要求。

#### 场景:20 篇论文批处理
- **当** 输入目录中存在至少 20 篇可处理论文
- **那么** 系统必须完成运行并输出结果文件，且进程不得异常退出

## 上下文

M1 需要把本地论文 PDF 批量转成后续检索可直接消费的结构化 chunk 数据。当前仓库尚未形成统一入库链路，风险主要在 PDF 文本质量不稳定、标题提取不可靠、以及单篇异常导致全量失败。约束是先以最小可用方案落地，优先保证 20+ 论文可稳定处理，并提供可人工抽检的数据输出。

## 目标 / 非目标

**目标：**
- 提供命令行入口 `python -m app.ingest --input <dir> --out <dir>`。
- 对 `data/papers/*.pdf` 批量解析，生成统一 chunk 结构与论文元数据映射。
- 输出 `chunks.jsonl` 与 `papers.json`，满足字段与可读性、页码可追溯要求。
- 对异常页/异常论文具备容错能力并记录日志，避免流程崩溃。
- 统一运行追踪日志格式与输出目录，保证实验过程可复现。
- 将关键超参数集中在 `configs/default.yaml` 管理，避免散落硬编码。

**非目标：**
- 不在 M1 引入向量索引、检索排序或在线服务。
- 不保证所有 PDF 完美 OCR 级质量；允许少量噪声。
- 不集成 GROBID 或复杂版面恢复，仅预留后续扩展点。
- 不在 M1 完成全部检索链路实现，但需预留统一日志字段与模块结构。

## 决策

1. 解析引擎采用 PyMuPDF（fitz）
- 原因：本地依赖轻、速度快、对常见学术 PDF 兼容性较好。
- 备选：pdfplumber（布局提取更细但速度/稳定性折中）、GROBID（质量高但引入服务运维复杂度）。

2. chunk 策略采用“标题边界优先 + 滑窗兜底”
- 原因：学术论文通常包含可识别章节标题，先按标题切分可保留语义结构；对超长段落再按 token 近似窗口（300-500，overlap 50）切分，控制块大小。
- 备选：纯固定窗口（实现简单但语义边界差）、纯段落切分（长度波动大，召回质量不稳定）。

3. 标题提取采用 metadata 优先、首页规则兜底
- 原因：metadata 有则直接可用；无 metadata 时用首页前若干行规则提取可快速上线。
- 备选：训练/调用标题识别模型（准确率更高但超出 M1 范围）。

4. 输出采用文件落盘（JSONL + JSON）
- 原因：符合 M1 目标，便于调试、抽检、后续导入向量库。
- 备选：直接入库（数据库/向量库）会增加耦合与环境依赖。

5. 错误处理采用“页级跳过 + 文档级继续 + 汇总日志”
- 原因：最大化批处理成功率，避免单点失败导致任务中断。
- 备选：失败即终止（定位简单但不满足稳定性目标）。

6. 运行可复现采用“单次运行目录 + 分阶段 JSON 轨迹”
- 原因：统一落盘到 `runs/YYYYMMDD_HHMM/` 可支持回放与问题定位；即使某阶段未实现也可输出空字段保持 schema 稳定。
- 备选：只打控制台日志（快速但不可结构化检索与对比）。

7. 超参数采用 YAML 集中配置
- 原因：`chunk_size`、`overlap`、检索与融合阈值等参数变化频繁，集中配置有利于实验对照。
- 备选：代码常量（实现快但难以复现实验）。

8. 模块按约定目录预留
- 原因：提前稳定 `app/` 中 ingest/index/retrieve/expand/rerank/judge/generate 的边界，减少后续里程碑重构成本。
- 备选：按当前最小代码临时放置（短期快，长期迁移成本高）。

## 风险 / 权衡

- [PDF 文本乱码或提取为空] -> 记录页级告警并跳过该页；统计每篇有效页比例供人工复核。
- [章节标题正则误判] -> 增加回退逻辑：标题切分失败时改用固定滑窗；保留原始页码与文本片段便于抽检。
- [token 估算与真实 tokenizer 偏差] -> M1 使用字符/词近似窗口，后续在检索阶段接入统一 tokenizer 校准。
- [标题提取粗糙] -> 优先保证 `paper_id` 与路径稳定映射，标题允许近似，后续可离线修正。
- [统一日志字段与当前实现阶段不一致] -> 采用固定 schema，未启用阶段写入空数组或 null，避免后续字段破坏性变化。
- [配置项增加导致兼容问题] -> 对 `configs/default.yaml` 设置默认值与缺省回退规则。

## 迁移计划

1. 增加 `app.ingest` CLI 与 ingest pipeline 模块（parser/chunker/writer）。
2. 在 `data/papers` 上试跑并生成 `data/processed` 产物。
3. 执行验收：处理数量、字段完整性、抽样可读性与页码一致性。
4. 若异常率过高，先通过规则修正与日志排障；必要时在 M2 评估 GROBID。
5. 落地 `configs/default.yaml`，将 chunk 与检索相关超参数集中定义。
6. 落地 `runs/YYYYMMDD_HHMM/*.json` 的运行轨迹写入器并串接 CLI。

回滚策略：
- 该变更为增量新增，不改写线上状态；若效果不满足要求，可移除新模块并删除 `data/processed` 产物恢复到变更前状态。

## 开放问题

- token 计数是否需要在 M1 就统一为特定模型 tokenizer（如 tiktoken），还是先采用近似计数。
- 标题规则是否需要按中英文论文分别维护不同正则模板。
- 是否需要在 `chunks.jsonl` 额外预留 `page_end`、`section` 字段以降低后续迁移成本。

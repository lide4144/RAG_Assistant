## 修改需求

### 需求:计划执行必须能够基于论文生命周期状态确定可用论文范围
计划执行必须查询 SQLite 论文存储以确定哪些论文处于 `ready` 状态，并仅将这些论文纳入检索、总结或问答范围。系统禁止在论文未达到 `ready` 状态时将其纳入执行范围。

#### 场景:执行器筛选 ready 论文进行检索
- **当** 计划执行需要检索论文时
- **那么** 执行器必须查询 SQLite 并仅选择状态为 `ready` 的论文进行检索

#### 场景:执行器识别非就绪论文
- **当** 某篇论文状态为 `imported`、`parsed` 或 `failed`
- **那么** 执行器必须将该论文排除在执行范围外，并可记录原因

### 需求:计划执行必须能够处理论文级依赖
计划执行必须能够处理 action plan 中声明的论文级依赖，如等待某论文索引完成后再执行后续步骤。

#### 场景:执行器等待论文就绪
- **当** action plan 声明依赖某篇论文的 `ready` 状态
- **那么** 执行器必须等待该论文就绪或按策略处理超时

#### 场景:执行器处理论文失败依赖
- **当** action plan 依赖的论文状态为 `failed`
- **那么** 执行器必须触发 fallback 或按策略跳过该论文

### 需求:计划执行必须能够查询论文重建状态
计划执行必须能够查询论文的 `rebuild_pending` 状态，以便在必要时触发重建或调整执行策略。

#### 场景:执行器检测待重建论文
- **当** 计划涉及的论文中有状态为 `rebuild_pending` 的
- **那么** 执行器可以提示用户或自动触发重建流程

#### 场景:执行器批量处理待重建论文
- **当** 用户请求批量处理多篇待重建论文
- **那么** 执行器必须能够获取待重建论文列表并批量触发重建

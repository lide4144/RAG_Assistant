## 上下文

当前流水线在 M5 图扩展与 M6 重排后，证据数量和文本长度显著增加；同时多轮会话引入历史拼接，导致 LLM 输入上下文存在两类风险：
- 上下文超限，触发 `Token Limit Exceeded`；
- 即便未超限，也因信息过长导致“中部丢失”（Lost in the Middle）并降低答案稳定性。

现状缺少统一的预算控制点，不同模块各自追加内容，无法保证 M7.5-B 入口 prompt 始终处于安全区间。

## 目标 / 非目标

**目标：**
- 在 LLM Answer 调用前新增统一的 Context Assembler，计算并约束总 token 预算。
- 当 `total_tokens > max_context_tokens` 时，按固定优先级裁剪：先历史、后证据。
- 对证据裁剪执行来源与分数感知策略：优先丢弃 `graph_expand` 且 `score_rerank` 低分项。
- 输出 `assembled_prompt`、`discarded_evidence`、`prompt_tokens_est`，并在兜底场景返回可观测告警。

**非目标：**
- 不改写 rerank 模型与打分算法本身。
- 不在本变更内引入新的 tokenizer 服务（先采用统一估算器）。
- 不改变最终答案生成提示词策略（仅控制输入拼装与裁剪）。

## 决策

1. 引入独立阶段 `M7.7 Token Budgeting & Context Assembly`
- 选择：在 M7.5-B 之前集中组装 prompt，而不是分散在检索/重排阶段各自截断。
- 理由：集中控制可保证单一预算真值，减少模块间耦合与重复裁剪。
- 备选：在每个上游模块设置各自上限。放弃原因：全局预算不可证明，易出现叠加超限。

2. 采用“历史优先裁剪”
- 选择：超预算时从最老历史开始删除，保留当前用户问题与最近最小上下文。
- 理由：当前问题是回答底线，历史信息的时间衰减更明显。
- 备选：优先裁证据。放弃原因：会更早损伤事实依据，增加幻觉风险。

3. 采用“来源 + 分数”联合证据裁剪
- 选择：当历史仅剩 1 轮后，按 `payload.source=graph_expand` 优先且 `score_rerank` 升序裁剪。
- 理由：图扩展证据通常召回宽、噪声更高；结合 rerank 可稳定识别边缘项。
- 备选：仅按长度裁剪。放弃原因：可能误删核心高分证据。

4. 证据清空兜底短路
- 选择：若裁剪后证据 `<1`，不调用 LLM，直接返回固定失败消息并记录 `context_overflow_fallback`。
- 理由：无证据调用会放大不可信回答风险，应显式失败而非隐式降级。
- 备选：允许空证据回答。放弃原因：可解释性与可信度不足。

## 风险 / 权衡

- [风险] token 估算器与真实 tokenizer 存在偏差 -> 缓解：预留安全余量（例如 10-15%）并记录估算/实际差异指标。
- [风险] 过度裁剪历史导致多轮指代失败 -> 缓解：保留至少 1 轮历史并在日志标记 `history_trimmed_turns`。
- [风险] 过度依赖来源标签可能误删有效图扩展证据 -> 缓解：仅在低 `score_rerank` 区间优先裁剪，并保留最小核心证据数阈值。
- [权衡] 严格兜底会增加“拒答率” -> 换取不超限与可解释失败，优先系统稳定性。

## 迁移计划

1. 新增 `context_assembler` 模块及配置 `max_context_tokens`，默认值与现有模型窗口匹配。
2. 在 M7.5-B 调用前接入该模块，替换旧的直接拼装路径。
3. 增加日志与告警字段：`prompt_tokens_est`、`discarded_evidence_count`、`context_overflow_fallback`。
4. 在测试中加入极端样本（`top_n_evidence=30` + 长图扩展文本），验证无超限异常。
5. 灰度发布：先记录裁剪决策（shadow mode），确认稳定后切换为强制裁剪。

## 开放问题

- 是否需要按模型类型维护不同的 `max_context_tokens` 默认值与安全余量。
- 当历史与证据均逼近下限时，是否需要引入“摘要压缩”作为下一阶段能力，而不是继续硬裁剪。

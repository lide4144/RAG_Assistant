## 上下文

当前 `frontend/components/chat-shell.tsx` 的 LLM 设置面板仍基于单路配置心智（`api_base/api_key/model`），而后端内核与配置加载逻辑已演进为 stage 化能力（answer/embedding/rerank）。这导致：
- 前端无法完整表达后端配置能力，用户只能配置 answer 路径。
- 接口契约与 UI 展示不一致，增加误操作和排障成本。
- 测试样例聚焦单路流程，无法约束三路配置稳定性。

本设计跨越前端交互层、内核管理接口层和配置持久化层，需要在兼容旧请求的前提下完成协议升级。

## 目标 / 非目标

**目标：**
- 将 LLM 设置面板升级为 answer/embedding/rerank 三路可编辑表单。
- 升级 `/api/admin/llm-config` 请求/响应契约，使其支持三路持久化结构。
- 保留旧单路载荷兼容，避免已有调用方立即中断。
- 为三路配置补齐前端与后端测试覆盖。

**非目标：**
- 不改动聊天主流程（检索、重排、回答）的业务逻辑。
- 不引入新的管理端页面，仅在现有 chat shell 面板扩展。
- 不在本变更中移除单路兼容逻辑（仅标记为后续可弃用）。

## 决策

### 决策 1：运行时配置采用“三路对象 + 单路兼容解析”

- 方案：`configs/llm_runtime_config.json` 支持 `answer/embedding/rerank` 三个对象；若读取到旧单路结构则自动映射到三路。
- 原因：在满足新能力的同时避免历史配置失效。
- 备选方案：
  - 直接切断旧结构：实现简单但升级风险高。
  - 仅前端做三路拆分、后端保持单路：能力不完整，无法真正独立配置三路。

### 决策 2：管理接口保持路径不变，仅扩展字段

- 方案：复用 `/api/admin/llm-config`，`POST` 支持三路字段（并兼容单路字段），`GET` 返回三路摘要。
- 原因：最小化网关与前端路由改动，降低集成成本。
- 备选方案：
  - 新增 `/api/admin/llm-config/stages`：语义清晰，但需要双接口维护与迁移协调。

### 决策 3：前端 UI 采用“三段式表单 + 独立保存状态”

- 方案：在现有设置卡片中展示 answer/embedding/rerank 三组输入，阶段级错误提示与保存结果聚合反馈。
- 原因：用户能直观看到三路配置，减少配置遗漏。
- 备选方案：
  - 多 tab 分离：交互复杂度上升，且切换成本高。

### 决策 4：向后兼容策略显式化

- 方案：后端在解析时优先三路结构，回退单路结构；前端默认发送三路结构。
- 原因：确保新旧客户端并存期间系统稳定。
- 备选方案：
  - 版本分支（v1/v2）：结构更清晰，但改造范围更大。

## 风险 / 权衡

- [风险] 三路表单增加用户填写负担
  → 缓解：提供默认 provider/api_base 填充值，并在 UI 中标注“可复用 answer 配置”。

- [风险] 单路兼容路径长期保留导致代码复杂度增加
  → 缓解：在文档中明确弃用窗口，后续版本移除兼容分支。

- [风险] 前后端字段命名不一致导致保存失败
  → 缓解：增加契约测试与 e2e 测试，覆盖三路保存/读取/回显。

## 迁移计划

1. 扩展后端运行时配置 schema 与解析逻辑（含单路兼容）。
2. 扩展 `/api/admin/llm-config` GET/POST 契约并添加测试。
3. 升级前端面板为三路配置 UI，接入新契约。
4. 执行前端 e2e 与后端契约回归。
5. 文档更新（使用方式、兼容策略、后续弃用计划）。

回滚策略：
- 若前端三路 UI 出现问题，可临时回退到旧单路提交（后端仍兼容）。
- 若三路持久化解析异常，可删除运行时配置文件并回退静态默认配置。

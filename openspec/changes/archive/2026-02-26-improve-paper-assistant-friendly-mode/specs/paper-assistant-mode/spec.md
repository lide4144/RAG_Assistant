## 修改需求

### 需求:系统必须提供论文助理默认模式
当用户提出开放式论文知识库问题（如“总结方向”“下一步问什么”）时，系统必须优先输出可追溯的主题化总结与后续提问建议，禁止直接返回空泛拒答文案。系统在发布评测中必须按分桶门槛验证该能力，且门槛必须以增长优先版本为准。

#### 场景:自然首问返回主题地图
- **当** 用户首轮输入“帮我总结这个知识库里最值得关注的方向”
- **那么** 系统必须返回 3~5 个主题方向，并为每个方向给出至少 1 条可追溯证据引用

#### 场景:返回可执行追问建议
- **当** 系统完成主题化总结
- **那么** 系统必须额外返回 1~3 个建议追问问题，且每个问题必须与已给主题相关

#### 场景:增长评测首问门槛
- **当** 系统执行 `A_open_summary` 桶评测
- **那么** 必须满足已配置的增长版首问门槛（可追溯回答率与拒答率）

### 需求:系统必须限制连续澄清并在上限后给出可用回答
系统必须跟踪当前会话主题下的连续澄清次数；当达到上限（默认 2）时，下一轮不得继续输出 `clarify`，必须输出低置信但可追溯的主题化回答，并明确不确定性。发布评测必须以会话链路方式验证该行为。

#### 场景:连续澄清达到上限后强制回答
- **当** 同一主题会话连续两轮 `decision=clarify`
- **那么** 第三轮系统必须输出可追溯主题化回答（含引用），并在结果中标记低置信/待验证，而不是再次澄清

#### 场景:多轮链路增长门槛
- **当** 系统执行 `B_multi_turn` 桶评测
- **那么** 每条会话链必须在限定轮次内至少出现一次可追溯回答，且不得出现超上限连续澄清

## 上下文

当前入库链路中，PDF 标题抽取在 metadata 缺失时依赖首页首行回退，已在真实数据中出现占位文本（如 `Preprint. Under review.`）被写入索引的问题。该问题会污染 `papers.json` 与 `paper_summary.json`，并直接影响检索排序、证据展示和问答输出。我们希望在不破坏当前 ingest 可用性的前提下，引入 Marker 作为结构化解析增强能力，并保留稳定回退路径。

## 目标 / 非目标

**目标：**
- 在 PDF 入库中引入 Marker 解析路径，并形成统一中间表示供后续切块与元数据抽取使用。
- 将标题抽取升级为“候选生成 + 质量门禁 + 置信度记录”机制，阻断占位标题入库。
- 在报告与追踪中提供解析来源与回退可观测性字段，支持定位质量问题。
- 在 Marker 不可用场景维持现有解析器回退，保证导入任务稳定完成。

**非目标：**
- 不在本变更中替换整个 chunker 算法或重写检索排序逻辑。
- 不在本变更中引入外部在线元数据源（DOI/Crossref）做标题修复。
- 不承诺一次性修复历史所有脏标题，仅提供可执行的增量修复路径。

## 决策

1. 决策：采用“可选 Marker + 必须回退”的双解析器架构。
- 方案 A（选中）：Marker 作为首选解析器，失败时回退现有 `parse_pdf_pages`。
- 方案 B：仅替换为 Marker，不提供回退。
- 选择理由：A 在质量提升与运行稳定性之间更平衡，适配个人开发环境的依赖波动。

2. 决策：在解析阶段构建统一中间表示，避免后续模块直接依赖具体解析器输出。
- 方案 A（选中）：新增中间表示结构（标题候选、段落块、层级、页码映射）。
- 方案 B：在现有数据结构上打补丁字段。
- 选择理由：A 能降低后续扩展成本，避免 Marker 与 legacy 逻辑在下游分叉蔓延。

3. 决策：标题落库前执行质量门禁并输出来源与置信度。
- 方案 A（选中）：黑名单 + 结构规则 + 置信度阈值的组合门禁。
- 方案 B：仅正则过滤。
- 选择理由：A 比单纯正则更稳健，且能解释“为什么接受/拒绝某候选”。

4. 决策：对 ingest 报告扩展观测字段，而不是只写日志。
- 方案 A（选中）：在 `ingest_report.json` 与 run trace 中新增结构化字段。
- 方案 B：仅在控制台日志打印。
- 选择理由：A 可被 API 与测试直接消费，便于回归和诊断。

## 风险 / 权衡

- [风险] Marker 依赖较重，部分环境安装复杂或执行慢。
  → 缓解措施：提供配置开关、超时控制与自动回退；将 Marker 设为可选依赖。

- [风险] 新旧解析器输出差异会影响 chunk 边界稳定性，可能引起召回波动。
  → 缓解措施：保留窗口参数约束；对关键样本建立前后对比回归基线。

- [风险] 标题门禁阈值过严可能导致更多 `Untitled` 降级。
  → 缓解措施：记录 `title_source/title_confidence`，按样本迭代阈值与黑名单。

- [风险] 增加中间表示后实现复杂度上升。
  → 缓解措施：限制首版字段范围，仅覆盖标题、层级、块边界、页码映射与最小观测字段。

## Migration Plan

1. 在 `app` 中新增 Marker 解析适配模块与统一中间表示转换层。
2. 在 `app/ingest.py` 接入解析器路由：`marker -> fallback`。
3. 更新 `requirements.txt`，新增 Marker 及其运行依赖，并补充安装兼容说明。
4. 改造标题抽取流程，新增质量门禁并扩展 `papers.json`/报告字段。
5. 增加单元测试与集成测试：标题黑名单、回退逻辑、观测字段断言。
6. 小样本重跑 ingest 对比，确认质量提升后再扩大到全量导入。
7. 为历史脏标题提供离线重建脚本（按 `paper_id` 增量修复）。

回滚策略：关闭 Marker 开关并恢复 legacy 解析路径；新字段保持向后兼容，不阻断旧消费者。

## 修改需求

### 需求:论文元数据映射输出
系统必须为每篇已处理文档写入统一映射记录，并在原有 `paper_id`、`title`、`path` 基础上扩展 `source_type` 与 `source_uri`（或等价字段）以区分 PDF 与 URL 来源。对于 PDF 来源，系统必须额外记录解析引擎与标题质量信息（至少包括 `parser_engine`、`title_source`、`title_confidence` 或等价字段），并禁止将命中标题黑名单的候选直接写入 `title`。

#### 场景:PDF 映射包含解析与标题质量字段
- **当** 一篇 PDF 文档导入成功
- **那么** 文档映射记录必须包含解析引擎与标题来源/置信度字段，并可用于排查标题误识别

#### 场景:标题候选命中黑名单
- **当** 解析得到的标题候选命中占位或版权黑名单模式
- **那么** 系统必须拒绝该候选并使用其他合规候选或安全降级标题，不得直接写入黑名单标题

### 需求:分块策略
系统必须优先依据可识别章节标题进行分段，并在分段过长时使用滑窗策略进行切块，目标窗口大小必须落在 300-500 token 近似区间，且相邻窗口必须包含约 50 token 重叠。系统必须在应用配置前完成参数合法性校验，禁止使用越界参数直接执行切块。对于 PDF 来源，系统必须优先使用结构化解析器提供的标题层级与块边界进行分段，并在结构化信息缺失时回退到文本规则分段。

#### 场景:结构化边界优先切块
- **当** 结构化解析结果包含章节层级与块边界
- **那么** 系统必须优先依据结构化边界分段，再在超长分段上执行滑窗切块

#### 场景:结构化信息缺失时回退
- **当** 当前文档缺少可用结构化边界
- **那么** 系统必须回退到文本规则分段并保持现有窗口大小与重叠约束

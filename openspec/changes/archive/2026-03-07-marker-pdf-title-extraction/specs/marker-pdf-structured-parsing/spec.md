## 新增需求

### 需求:本地 Marker 解析能力
系统必须支持在本地调用 Marker 对 PDF 进行结构化解析，并将解析结果转换为入库可用的统一中间表示；当 Marker 不可用或解析失败时，系统必须回退到现有解析器以保证导入可完成。

#### 场景:Marker 可用时优先解析
- **当** 运行环境已安装 Marker 且当前文档解析成功
- **那么** 系统必须使用 Marker 结果构建中间表示并进入后续入库流程

#### 场景:Marker 失败时回退
- **当** Marker 依赖缺失、执行超时或解析报错
- **那么** 系统必须记录失败原因并自动回退到现有解析器，不得中断整批导入

### 需求:标题候选与质量门禁
系统必须基于 Marker 输出生成标题候选并执行质量门禁；系统禁止将占位文本、版权页眉或模板文案作为最终标题写入 `papers.json`。

#### 场景:命中占位标题被拒绝
- **当** 候选标题命中 `Preprint. Under review.`、`All rights reserved` 或同类黑名单模式
- **那么** 系统必须拒绝该候选并继续选择其他候选或降级为安全标题

#### 场景:高置信标题直接采用
- **当** 候选标题通过质量门禁且置信度达到配置阈值
- **那么** 系统必须将该标题写入文档映射并在追踪中记录标题来源与置信度

### 需求:结构化正文保真输出
系统必须保留 Marker 提供的结构化正文信息，并在入库分块前输出稳定的章节层级、公式文本与表格文本序列，禁止在无告警情况下静默丢弃结构信息。

#### 场景:章节层级可追踪
- **当** PDF 含显式标题层级
- **那么** 系统必须在中间表示中保留层级信息并供后续切块策略使用

#### 场景:公式与表格文本不丢失
- **当** PDF 页面包含公式块或表格区域
- **那么** 系统必须将对应文本保留到中间表示，并可在 chunk 文本中追溯

### 需求:解析可观测性字段
系统必须在 ingest 运行报告中写入解析引擎来源与质量信号，至少包含 `parser_engine`、`parser_fallback`、`title_source`、`title_confidence` 字段（或等价字段）。

#### 场景:Marker 解析成功的观测
- **当** 某文档通过 Marker 解析并完成导入
- **那么** ingest 报告必须标记 `parser_engine=marker` 且 `parser_fallback=false`

#### 场景:回退解析的观测
- **当** 某文档触发回退解析
- **那么** ingest 报告必须标记 `parser_fallback=true` 并记录可定位的失败原因

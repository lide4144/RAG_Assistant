## 新增需求

### 需求:所有用户回答必须通过LLM生成
系统禁止直接拼接证据片段生成机械化回复。所有用户查询的回答必须由大语言模型(LLM)基于提供的证据上下文生成自然语言回答。

#### 场景:用户请求开放式总结
- **当** 用户请求"给我一份最近导入论文的关键贡献对比表"
- **那么** 系统必须调用LLM，将检索到的证据作为上下文输入
- **并且** LLM生成自然语言回答，而非直接拼接证据片段

#### 场景:证据不足时的回答
- **当** 检索到的证据数量少于3条
- **那么** 系统仍必须调用LLM基于现有证据生成回答
- **并且** 在回答末尾附加风险提示，说明证据数量有限

### 需求:证据作为上下文传递
检索到的证据片段必须格式化为LLM可理解的上下文，作为系统提示词的一部分传递给LLM，而非直接输出给用户。

#### 场景:证据格式化
- **当** 系统检索到相关证据片段
- **那么** 证据必须格式化为结构化的上下文块
- **并且** 上下文块包含：论文标题、引用片段、引用编号、相关度分数

#### 场景:上下文注入
- **当** 构建LLM提示词
- **那么** 格式化后的证据上下文必须注入到系统提示词中
- **并且** 提示词明确指示LLM基于提供的证据生成回答

### 需求:LLM不可用时的降级处理
当LLM服务不可用时，系统必须返回友好的错误提示，而非崩溃或返回空回复。

#### 场景:LLM服务超时
- **当** LLM调用超时(>30秒)
- **那么** 系统返回预定义的错误消息："抱歉，智能回答服务暂时不可用，请稍后重试"
- **并且** 记录错误日志供运维排查

#### 场景:LLM服务返回错误
- **当** LLM服务返回非200状态码
- **那么** 系统返回预定义的错误消息："回答生成服务遇到错误，请稍后重试"
- **并且** 记录详细的错误信息

## 修改需求

### 需求:删除代替LLM的硬编码回答函数
系统必须删除所有直接拼接证据生成回答的函数，包括但不限于：`_build_assistant_summary_answer`、`compose_catalog_answer`的硬编码路径、`_build_claim_plan`。

#### 场景:开放式总结请求
- **当** 用户请求跨文档总结或对比
- **那么** 系统禁止调用`_build_assistant_summary_answer`函数
- **并且** 必须路由到LLM生成回答

#### 场景:目录查询请求
- **当** 用户请求查询论文目录
- **那么** 系统禁止直接格式化输出目录结果
- **并且** 必须调用LLM生成自然语言回答

## 移除需求

### 需求:_build_assistant_summary_answer函数
**Reason**: 该函数直接拼接证据片段生成机械化回复，剥夺了LLM生成自然语言的能力
**Migration**: 使用标准的LLM回答生成流程，将证据作为上下文传递

### 需求:compose_catalog_answer的硬编码输出路径
**Reason**: 目录查询结果直接格式化输出，缺乏智能感
**Migration**: 目录查询结果必须通过LLM生成自然语言总结

### 需求:_build_claim_plan函数的硬编码输出
**Reason**: 证据不足时返回固定的硬编码计划，无法根据实际情况灵活调整
**Migration**: 证据不足时由LLM基于已有证据生成回答，并附加风险提示

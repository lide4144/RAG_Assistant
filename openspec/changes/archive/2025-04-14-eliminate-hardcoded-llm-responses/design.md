## 上下文

### 当前状态

系统目前存在多处**代替 LLM 生成回答**的硬编码逻辑，主要集中在以下函数：

1. **`_build_assistant_summary_answer`** (app/qa.py:1477-1538)
   - 当用户请求开放式总结（如"对比表"、"总结"）时，直接拼接证据片段
   - 返回机械化的格式化回复："基于当前可追溯证据，优先可关注这些主题："
   - **危害**：完全绕过 LLM，剥夺了大模型生成自然语言的能力

2. **`compose_catalog_answer`** (app/capability_planner.py:638-690)
   - 目录查询直接格式化输出："基于目录元数据，匹配到这些论文："
   - **危害**：返回死板的列表而非自然语言回答

3. **`_build_claim_plan`** (app/qa.py:937-985)
   - 证据不足时返回硬编码计划："当前证据不足，无法生成可追溯的结论"
   - **危害**：固定模板无法灵活适应不同场景

4. **`build_answer`** (app/generate.py:27-31)
   - 最简单的硬编码回答："基于检索证据，问题...最相关内容见..."

### 约束条件

1. **必须保证 LLM 调用**：所有用户回答路径必须能调用 LLM
2. **降级路径**：LLM 不可用时需要优雅降级，返回友好错误提示
3. **性能考虑**：LLM 调用增加响应时间，需要考虑超时和流式输出
4. **向后兼容**：错误消息等系统级硬编码可以保留

### 利益相关者

- **终端用户**：获得更自然、智能的回答体验
- **开发者**：代码更清晰，无需维护多套回答生成逻辑
- **运维人员**：LLM 服务依赖增加，需要监控 LLM 可用性

## 目标 / 非目标

**目标：**
1. 删除所有代替 LLM 的硬编码回答函数
2. 强制所有回答路径经过 LLM 生成
3. 将证据作为上下文传递给 LLM，而非直接输出
4. 实现 LLM 不可用时优雅降级
5. 保持错误消息的系统级硬编码（非用户可见回答）

**非目标：**
1. 不修改 LLM 调用底层实现（`call_chat_completion` 等函数）
2. 不改变检索逻辑和证据召回机制
3. 不引入新的外部依赖（除现有 LLM 服务外）
4. 不改变 API 接口契约

## 决策

### 决策 1：删除 `_build_assistant_summary_answer` 函数

**选择**：完全删除该函数，将其调用点重定向到标准 LLM 回答生成流程。

**理由**：
- 该函数是主要的"毒点"，完全剥夺了 LLM 的能力
- 标准 LLM 流程已支持流式输出，用户体验更好
- 证据可以通过标准上下文组装机制传递给 LLM

**替代方案考虑**：
- **保留并重构**：保留函数但改为调用 LLM → 拒绝，函数名和语义已不适应
- **作为 LLM 的 fallback**：当 LLM 失败时返回硬编码回复 → 拒绝，这会降级用户体验

### 决策 2：重构 `compose_catalog_answer` 为 LLM 驱动

**选择**：修改 `compose_catalog_answer`，将目录结果格式化为上下文后调用 LLM 生成回答。

**理由**：
- 目录查询结果本质上是结构化数据，需要自然语言描述
- LLM 可以根据用户查询意图生成个性化的回答

**实现方式**：
```python
# 原逻辑（硬编码）
def compose_catalog_answer(catalog_result):
    lines = ["基于目录元数据，匹配到这些论文："]
    for paper in catalog_result['papers']:
        lines.append(f"- {paper['title']}")
    return "\n".join(lines)

# 新逻辑（LLM 驱动）
def compose_catalog_answer(catalog_result, query):
    context = format_catalog_context(catalog_result)
    return call_llm_for_catalog_summary(query, context)
```

### 决策 3：保留 `_build_claim_plan` 的概念但 LLM 化

**选择**：不删除概念本身，但改为调用 LLM 生成计划。

**理由**：
- 证据不足时向用户解释情况是有价值的
- 但解释方式应该由 LLM 根据具体情况生成

### 决策 4：错误消息分类处理

**选择**：将消息分为两类：
1. **系统级错误**（保留硬编码）：网络错误、配置错误等，用户无法交互
2. **用户可见建议**（LLM 生成）：证据不足提示、追问建议等

**理由**：
- 系统级错误需要稳定性和一致性
- 用户可见内容需要自然和个性化

### 决策 5：LLM 不可用时降级策略

**选择**：当 LLM 超时或返回错误时，返回预定义的友好错误消息。

**错误消息**：
- 超时："抱歉，智能回答服务暂时不可用，请稍后重试"
- 错误："回答生成服务遇到错误，请稍后重试"

**理由**：
- 避免系统崩溃或返回空回复
- 消息友好且可指导用户下一步操作
- 预定义消息确保一致性

## 风险 / 权衡

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| **LLM 调用增加导致响应时间上升** | 中 | 1. 保持流式输出让用户感知进度<br>2. 优化上下文组装减少 token 数<br>3. 设置合理的超时阈值（30秒） |
| **LLM 服务不可用导致功能完全失效** | 高 | 1. 实现优雅降级返回友好错误<br>2. 增加 LLM 健康检查<br>3. 考虑引入本地备用模型（可选） |
| **成本增加** | 中 | 1. 监控 LLM token 使用量<br>2. 优化提示词模板减少冗余<br>3. 缓存常见查询结果 |
| **开发者误用硬编码** | 低 | 1. 代码审查时检查<br>2. 添加 lint 规则检测硬编码中文回答<br>3. 文档中明确禁止模式 |
| **测试覆盖不足** | 中 | 1. 新增测试验证所有回答经过 LLM<br>2. 测试降级路径<br>3. 测试证据上下文传递正确性 |

**主要权衡**：
- **响应时间 vs 回答质量**：接受轻微延迟换取自然语言回答
- **系统复杂度 vs 用户体验**：增加 LLM 调用链路但提升智能感
- **成本 vs 质量**：接受 LLM 调用成本换取用户体验

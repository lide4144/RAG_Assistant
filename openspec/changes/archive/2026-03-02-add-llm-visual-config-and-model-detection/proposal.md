## 为什么

当前系统的 LLM 路由配置主要依赖静态配置与环境变量，前端缺少可视化配置入口，导致接入新的 OpenAI 兼容服务需要手动改文件和重启，试错成本高。现在需要提供一个可视化配置流程，支持在线检测可用模型并持久化生效，以便 LiteLLM Router 稳定复用。

## 变更内容

- 新增后端管理接口 `POST /api/admin/detect-models`：接收 `api_base`、`api_key`，通过 `httpx` 请求 `${api_base}/models`，返回规范化模型列表。
- 新增前端 LLM 设置面板：包含 API Base、API Key 输入框，提供“测试连接并获取模型”按钮并展示模型下拉选择。
- 新增保存配置能力：保存用户最终选择（`api_base`、`api_key`、`model`）到配置持久化层，供后续 LiteLLM Router 读取。
- 增强错误处理：覆盖跨域问题、超时、鉴权失败、上游返回格式异常等错误并向前端返回可理解错误码/消息。
- 修改现有 LLM 路由配置使用方式：从“仅静态配置/环境变量”扩展为“支持管理配置持久化结果作为运行时配置来源”。

## 功能 (Capabilities)

### 新增功能
- `admin-llm-model-detection`: 提供管理员模型探测接口，连接 OpenAI 兼容 `/models` 端点并返回标准化列表与错误分类。
- `frontend-llm-connection-settings`: 提供前端可视化设置面板，支持连接测试、模型选择和提交保存。
- `llm-runtime-config-persistence`: 提供 LLM 运行时配置持久化读写契约，并将已保存配置接入运行时使用链路。

### 修改功能
- `litellm-python-runtime-adapter`: 调整运行时配置来源与加载优先级，使 Router 可消费新保存的动态配置。

## 影响

- Python API 层：新增管理接口与上游模型探测逻辑（`httpx`、超时/鉴权/网络错误映射、请求参数校验）。
- 前端应用层：新增设置 UI、交互状态（测试中/成功/失败）、模型下拉和保存动作。
- 配置层：新增或扩展配置持久化结构，包含敏感信息存储策略与读取接口。
- Router 集成层：LiteLLM 路由初始化/请求阶段读取持久化配置并应用。
- 测试与运维：增加后端接口测试、前端交互测试和配置兼容性验证。

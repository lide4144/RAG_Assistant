## 新增需求

### 需求:系统必须提供管理员模型探测接口
系统必须提供 `POST /api/admin/detect-models` 接口，接收 `api_base` 与 `api_key`，并由服务端调用 `${api_base}/models` 获取模型列表。

#### 场景:管理员提交可用连接信息
- **当** 管理员提交合法 `api_base` 和 `api_key`
- **那么** 系统必须返回可选择的模型列表且每个模型至少包含 `id`

### 需求:系统必须标准化上游 models 响应
系统必须将上游 OpenAI-compatible `/models` 响应标准化为稳定输出结构，禁止将未校验的上游字段直接透传给前端。

#### 场景:上游返回标准 data 列表
- **当** 上游响应包含 `data` 数组且元素含模型标识
- **那么** 系统必须返回去重后的模型 `id` 列表并保留稳定字段顺序

### 需求:系统必须分类返回探测错误
系统必须将探测失败映射为可识别错误类别，至少覆盖参数错误、鉴权失败、超时、网络错误和上游响应格式错误。

#### 场景:鉴权失败
- **当** 上游返回 401 或 403
- **那么** 系统必须返回 `AUTH_FAILED` 错误码并提示检查 API Key

#### 场景:请求超时
- **当** 上游在超时时间内未响应
- **那么** 系统必须返回 `UPSTREAM_TIMEOUT` 错误码

### 需求:系统必须允许前端跨域调用管理接口
系统必须允许受信任前端来源通过 CORS 访问模型探测接口，禁止因预检失败导致管理功能不可用。

#### 场景:浏览器发起预检请求
- **当** 前端来源向管理接口发起 `OPTIONS` 预检
- **那么** 系统必须返回允许的来源、方法与请求头

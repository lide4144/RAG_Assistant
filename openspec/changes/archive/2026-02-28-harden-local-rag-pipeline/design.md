## 上下文

当前 RAG 本地流水线已能在中小规模数据集上工作，但关键链路仍偏“脚本化”：
- 数据写入对中断不具备原子保障，异常时可能遗留半成品。
- 大文件序列化与校验路径存在内存峰值不可控问题。
- 校验语义停留在字段存在级别，难以提前发现类型与约束错误。
- UI 侧以阻塞等待为主，长耗时操作缺少流式和进度反馈。
- 运行结果目录由“最新目录猜测”推断，存在时序脆弱性。
- 相对路径依赖导致非项目根目录启动时行为不稳定。

该变更涉及 ingestion、validation、QA 运行编排和 Streamlit UI 多模块协同，属于跨域改造，且包含性能与稳定性目标。

## 目标 / 非目标

**目标：**
- 在本地单机场景下，确保核心产物写入具备原子替换与失败清理能力。
- 为关键写入与索引更新步骤增加互斥机制，避免并发写入损坏本地数据。
- 将高体量 JSON/JSONL 处理路径改为流式或受限内存策略，控制峰值占用。
- 将 chunk 校验升级为结构+类型+语义约束校验，并支持 fail-fast 错误上限。
- 消除“猜测 run_dir”的脆弱流程，前后端共享显式 `run_id/run_dir` 生命周期。
- 在 UI 提供可观察进度、回答流式呈现与可恢复错误提示。
- 控制会话内存体积，重型 trace 数据落盘并按需读取。
- 统一路径锚定策略，支持非根目录启动。

**非目标：**
- 不在本次变更中引入分布式调度、跨主机锁或多租户隔离。
- 不重写整条 QA/ingest 业务逻辑，仅替换关键脆弱点与接口边界。
- 不强制要求在线服务形态（保持 local-first）。

## 决策

### 决策 1：关键产物采用“临时文件 + 原子替换”写入协议
- 方案：写入 `*.tmp` 后 `rename/replace` 到目标文件；失败时清理临时文件。
- 原因：可防止崩溃后暴露半成品，且在本地文件系统上成本低。
- 备选方案：
  - 直接覆盖写入：实现简单，但崩溃时风险最高。
  - 数据库存储替代文件：一致性更强，但改造范围过大，不符合本次边界。

### 决策 2：对写入与索引步骤引入进程级文件锁
- 方案：在落盘与索引更新入口加 `filelock`（或等效）互斥锁，锁粒度按目标资源路径划分。
- 原因：贴合单机多标签页/多次点击场景，最小改造即可防并发破坏。
- 备选方案：
  - 仅 UI 禁用重复点击：无法覆盖多进程或异常重入。
  - 线程锁：不能跨进程，不满足本地多实例风险。

### 决策 3：序列化与校验采用流式与上限控制
- 方案：
  - JSONL 按行写入与校验。
  - 大数组 JSON 输出使用增量拼接（或等效流式编码）而非全量 `list + dumps`。
  - 校验错误设置 `max_errors` 并 fail-fast。
- 原因：降低内存峰值并保证坏数据场景下质检自身稳定。
- 备选方案：
  - 保持全量内存构建：实现简单但无法支撑百万级数据。
  - 全面替换为新存储格式：收益高但超出当前变更范围。

### 决策 4：校验升级到模型化约束
- 方案：基于现有数据模型（如 `ChunkRecord`）执行类型与必填约束校验；保留缺字段与 JSON 语法错误诊断。
- 原因：把错误前置到 ingest/validate 阶段，避免下游索引与检索阶段爆炸。
- 备选方案：
  - 手写字典校验：可控但易漏约束、维护成本高。
  - 仅在下游失败时兜底：定位晚、恢复成本高。

### 决策 5：前后端统一显式 run 生命周期
- 方案：由调用端生成 `run_id` 并传入 pipeline，读取产物时使用显式 `run_dir`，废弃“latest dir”推断。
- 原因：消除时序竞态和目录污染带来的结果错配。
- 备选方案：
  - 继续比较修改时间：实现最省，但脆弱性无法根治。

### 决策 6：UI 采用流式输出 + 可恢复错误 + 会话瘦身
- 方案：
  - 回答链路支持增量 token/片段渲染。
  - 导入/处理链路提供进度回调。
  - 异常统一捕获并展示 `st.error`，不让页面级崩溃中断交互。
  - `st.session_state` 仅保留渲染必要字段，trace 明细按需从磁盘加载。
- 原因：单机用户首先感知的是“是否卡死”和“是否能恢复”。
- 备选方案：
  - 仅保留 spinner：交互不可观察，失败体验差。
  - 全量 trace 常驻内存：实现简单但长期会话易膨胀。

### 决策 7：路径统一锚定到项目根
- 方案：以代码文件绝对路径推导项目根并构建默认数据/运行路径。
- 原因：避免 CWD 变化导致的路径失效。
- 备选方案：
  - 继续相对路径：短期可用，打包和子目录启动不稳定。

## 风险 / 权衡

- [文件锁误用导致“假死”] → 设置锁超时与清晰报错；锁粒度最小化，避免全局大锁。
- [原子替换在跨分区目录上失败] → 约束临时文件与目标文件同目录创建；失败时明确降级日志。
- [流式改造引入接口兼容风险] → 在 API 层提供兼容包装，逐步替换 CLI 风格调用。
- [更严格校验提升拒绝率] → 提供可配置阈值与错误摘要，先观测后收紧。
- [trace 落盘后调试不便] → 提供“按需加载详情”入口，默认简洁、需要时可深挖。
- [引入 `orjson/filelock` 增加依赖复杂度] → 设为可选或分层依赖，保留标准库回退路径。

## 迁移计划

1. 在写入层引入原子写入工具函数，并替换 chunks/papers 等关键产物落盘入口。
2. 在写入与索引入口加文件锁，补充并发触发测试。
3. 重构校验：加入类型/约束检查、`max_errors` fail-fast、错误摘要输出。
4. 改造 QA 运行入口：显式传入 `run_id/run_dir`，移除目录猜测逻辑。
5. 改造 UI：
   - 回答流式渲染；
   - 导入流程进度回调；
   - 统一异常捕获与恢复提示；
   - session state 瘦身与 trace 懒加载。
6. 统一默认路径锚定与配置读取逻辑。
7. 回归测试覆盖：中断写入、并发写入、超量错误、网络波动超时恢复、非根目录启动。

回滚策略：
- 以模块开关方式保留旧路径（短期），若新路径触发阻断问题可按开关降级；
- 原子写入与锁机制可按配置逐项关闭以快速止血。

## 开放问题

- `orjson` 是否作为默认依赖，还是仅在可用时启用加速？
- 文件锁粒度应以“目标文件”还是“数据目录”划分，才能平衡安全与吞吐？
- fail-fast 默认阈值（如 50/100）应按文件规模动态调整还是固定配置？
- UI 流式输出是 token 级还是句段级，如何平衡刷新频率与性能？

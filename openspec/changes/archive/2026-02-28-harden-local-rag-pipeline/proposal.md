## 为什么

当前系统在中小规模数据下可用，但在百万级文档、长时本地运行和异常中断场景下存在明显风险：内存峰值不可控、写文件非原子导致半成品污染、校验仅做键级检查、以及 UI 交互在慢模型下反馈不足。现在需要把这条本地优先的 RAG 流水线提升到可持续运行的工程级基线。

## 变更内容

- 为核心产物写入引入原子落盘与失败清理，避免中途崩溃留下损坏文件。
- 在本地落盘与索引更新阶段引入进程互斥/文件锁，避免多标签页或重复触发导致并发写入冲突。
- 将大对象序列化与校验逻辑改为面向大数据量的流式/限额策略，避免 OOM。
- 将 chunk 校验从“字段存在”升级为“字段+类型+约束”校验，并支持 fail-fast 错误上限。
- 统一前后端运行目录生命周期，移除基于“最新目录猜测”的脆弱结果定位方式。
- 强化本地工作台体验：长耗时步骤提供流式输出与进度反馈，错误以可恢复提示呈现而非页面崩溃。
- 修复相对路径依赖，确保非仓库根目录启动和打包后运行仍可定位默认资源路径。
- 严格瘦身 `st.session_state`：仅保留对话渲染所需轻量字段，重型 trace 与检索中间物下沉本地磁盘并按需懒加载。

## 功能 (Capabilities)

### 新增功能
- `local-pipeline-durability-validation`: 定义本地流水线在大数据量场景下的原子写入、内存上界控制、强类型校验与失败恢复要求。

### 修改功能
- `paper-ingestion-pipeline`: 扩展入库产物写入与校验要求，增加流式写入、fail-fast 上限、类型约束和部分失败恢复语义。
- `research-assistant-workbench`: 增加本地长任务的进度反馈、可恢复错误提示与会话内存控制要求。
- `llm-answer-streaming-delivery`: 明确前端消费增量输出的行为契约，避免仅后端流式而 UI 阻塞。

## 影响

- 代码范围：`app/` 中 ingestion、serialization、validation、QA 运行编排与 Streamlit UI 调用链。
- 运行行为：新增临时文件与原子替换流程；新增校验失败阈值与降级路径。
- 配置与依赖：可能引入高性能 JSON 库（如 `orjson`）为可选依赖；新增与路径锚定、错误阈值、流式开关相关配置。
- 测试与验收：需要补充大文件写入中断、超量错误 fail-fast、类型校验失败、非根目录启动、长耗时流式显示等回归测试；并增加断网/断连恢复测试，验证外部模型调用超时或网络波动时应用可提示错误且保留历史会话连续性。

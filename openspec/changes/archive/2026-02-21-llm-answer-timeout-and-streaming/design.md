## 上下文

当前 `answer_use_llm=true` 时，回答链路使用单次非流式调用，受 `llm_timeout_ms` 统一约束。面对较大的 `evidence_grouped` 输入，回答阶段容易在首字前耗尽超时预算，触发 `llm_answer_timeout_fallback_to_template`。这会降低高质量证据化回答命中率，并使用户感知延迟显著。

本变更需要在不破坏现有证据门控、citation 子集校验和 fallback 语义的前提下，提升回答阶段稳定性与交互体验。

## 目标 / 非目标

**目标：**
- 支持回答阶段更高且可配置的超时预算，减少长上下文场景的误超时。
- 在回答阶段引入流式返回，使用户可持续获得增量内容。
- 保持现有降级路径与诊断对象一致，确保失败可审计、主流程不中断。
- 保持最终 QA 结构兼容（仍输出 `answer` 与 `answer_citations`）。

**非目标：**
- 不改变检索、重排、证据分配与 gate 判定算法。
- 不在本次引入多 Provider 动态路由。
- 不放宽 evidence 约束，不允许流式路径绕过 citation 校验。

## 决策

### 决策 1：回答阶段采用“阶段化超时配置”
- 方案：保留全局 `llm_timeout_ms` 兼容值，同时新增回答阶段专用超时配置（例如 `answer_llm_timeout_ms`），优先使用回答专用值。
- 原因：rewrite 与 answer 的时延分布差异大；统一超时会导致 answer 被过度压缩。
- 备选：仅调大全局 `llm_timeout_ms`。问题是会连带拉长 rewrite 超时，影响整体尾延迟。

### 决策 2：流式仅用于传输，不改变回答约束
- 方案：`answer` 路径支持 stream 调用，内部累积流片段并在结束后进入既有结构化校验与证据门控。
- 原因：可兼容现有 `answer/answer_citations` 输出契约，同时获得更低首字延迟。
- 备选：直接输出自由流文本并跳过结构化校验。风险是破坏 citation 可追溯性。

### 决策 3：流式失败统一回退到模板路径
- 方案：流式过程中若发生中断、超时、空流或非法载荷，统一映射到 answer fallback warning 并走模板路径。
- 原因：保证链路可用性与告警一致性，不引入新的不可控失败态。
- 备选：部分流内容“就地完成”输出。风险是输出结构不完整，难以通过证据校验。

### 决策 4：新增流式诊断观测字段
- 方案：增加流式启用、是否实际走流、首字延迟、流式回退原因等观测字段，并与 `answer_llm_diagnostics` 保持一致引用。
- 原因：仅有 `timeout` 维度不足以定位是“首字超时”“中途断流”还是“解析失败”。
- 备选：沿用现有诊断字段不扩展。风险是无法区分流式特有故障。

### 决策 5：CLI 显示与结构化校验解耦
- 方案：在 CLI 模式下将流式 token 增量实时打印到终端，同时继续在后台累积完整内容；流结束后仍执行既有 JSON/citation 校验与证据门控。
- 原因：满足“用户可见实时出字”体验，同时不破坏 `answer`/`answer_citations` 的结构化契约与降级语义。
- 备选：保持仅日志可观测而 CLI 不实时输出。问题是用户无法直接感知流式收益，违背交互目标。

## 风险 / 权衡

- [风险] 长连接与分片解析引入额外复杂度 -> [缓解] 统一在 `llm_client` 封装流式协议与错误归一化。
- [风险] 首字快但最终结构化失败 -> [缓解] 仅将流式作为传输层，完成后仍执行严格 JSON/citation 校验。
- [风险] 超时调大带来资源占用上升 -> [缓解] 回答阶段专用超时 + 保留重试上限 + 诊断追踪。
- [风险] 新字段影响历史兼容 -> [缓解] 新字段可选、默认值安全，保留旧字段语义。

## Migration Plan

1. 新增配置：回答阶段专用超时与流式开关，默认关闭流式以保证平滑迁移。
2. 扩展 `llm_client`：增加 stream 调用与错误标准化，保持非流式路径可用。
3. 更新 `qa` 回答路径：在证据充分且开关开启时尝试流式；失败统一 fallback。
4. 扩展 runlog/schema 校验与测试用例：覆盖超时上调、流式成功、流式中断、回退一致性。
5. 在 CLI 路径接入增量显示（含首包前提示、流中断收尾信息），并补充交互测试。
6. 逐步启用：先在测试/预发布开启，再扩展到默认环境。

回滚策略：关闭流式开关并恢复原超时配置即可，无数据迁移要求。
